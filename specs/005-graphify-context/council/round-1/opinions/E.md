## Suggestions

- [sequencing] Arm 2's failure-path control flow is named but not specified. `plan.md` L75 says a stale product "hard-warns and routes to regeneration," and L76 says a survivor-guard failure triggers "prune-or-rebuild" — but neither `plan.md` nor the deck's §3 states which concrete operation "regeneration" invokes (the new incremental `refresh.sh`, or the old ~753k-token full-corpus ritual this arm exists to avoid), nor what decides prune vs. rebuild. SC-004's "materially lower cost" claim (§7c) only holds if the common-case stale path actually runs cheap `refresh.sh` rather than the full ritual, so this ambiguity sits directly on the arm's core cost claim. It compounds with §7e's own admission that the survivors>0 branch has zero fixture coverage: an unspecified branch and an untested branch are the same branch here. Recommend the plan add one explicit state-transition line (stale → `refresh.sh` attempt → survivors==0 ships / survivors>0 → {prune | full rebuild, pick one}) before `tasks.md` is generated. (confidence: high)
- [risk] Live-reproduced grounding-tool gotcha: `graphify explain "member-prompt.md"` — the exact short-filename form the deck's own worked examples use throughout §3/§4 — silently resolves to the wrong node (a degree-1 concept, "Thin member interface (member-prompt.md {{lens}} slot)," sourced from spec 001's `plan.md`) instead of the intended degree-16 arm-4 home file, with no ambiguity warning. Contrast `graphify path`, which prints an explicit "warning: source match was ambiguous" when it hits a comparable collision on `verify-gate.sh`. The deck's degree-16 claim for member-prompt.md is itself accurate — confirmed it, but only by escalating to `graphify query` for discovery and re-querying by the node's full label. As arm 1 lands and adds many more `.sh`/`.yml` nodes across `extensions/*/`, same-named plumbing files (multiple `install.sh`-shaped scripts, etc.) will make this exact silent-wrong-match failure mode more likely, not less. Worth either hardening `explain`'s ambiguity detection to match `path`'s, or documenting a qualified-path query convention for the deck-prep/council consumers who rely on `explain "<short-name>"` for grounding. (confidence: high)
- [sequencing] No intra-cycle execution order is stated across arms 1→2→3. Arm 3's generator "Reads: the current (augmented, fresh) graph" (§3) — language that presumes `augment.sh` (arm 1) and `freshness.sh`/`refresh.sh` (arm 2) have already run in the current cycle — but grepping `plan.md` for wiring/ordering language ("runs after/before," hook-order, etc.) turns up nothing; the ordering is implicit at best. This isn't hypothetical: consumer fixture 4 (§7b, arm 4's own fixture) requires the member to read "its receipts diet" — i.e. arm 3's product must already exist for arm 4's fixture to be exercisable at all. Severability (which arm can ship without the others) and build/test order (what must exist before what can be validated) are being treated as the same question in this deck; they aren't. Recommend the plan (or `tasks.md`, if deliberately deferred) state the augment → freshness/refresh → generate order explicitly. (confidence: med)
- [sequencing] The query ceiling is described as tier-aware (§3, arm 4: "resolved from `council-config.yml`'s `member.query_ceiling` (tier-aware)"), but this round — whose per-member counts become "arm 4's own calibration baseline" (§3, R3 mitigation) — runs standard tier only (lazy, deck-only-required loading). A standard-tier round structurally produces a different query pattern than a full-tier round would (eager 3-doc reads vs. on-demand-only). Nothing in the deck states how the full-tier ceiling gets calibrated given that this baseline round can only measure the standard-tier pattern. Worth naming explicitly whether a full-tier baseline round is planned separately, or whether that ceiling will be derived by extrapolation from this one. (confidence: med)
- [sequencing] The freshness design (arm 2) addresses the single-threaded staleness gap it was modeled on (R1-S03's six-minute mtime lag) but the risk register (R1–R5) doesn't address a related, distinct hazard: a check-then-use race under parallel-wave execution. The plan's own shared/mutable collision list (§4) already establishes that concurrent writes to the same files are a live concern in this architecture ("serialize any tasks touching these; never co-schedule two in one parallel wave") — which means a `freshness.sh` check that passes for one wave-task's `before_implement` hook could be invalidated by a sibling task's concurrent edit landing before the first task actually reads `graphify-context.md`. Same failure family as R1-S03, just concurrent rather than sequential, and covered by neither the arm-2 fixture pair (§7a, both single-threaded: mutate-then-check) nor the risk register. Worth either an explicit argument for why parallel-wave hooks can't hit this window, or a fixture that tries. (confidence: med)
