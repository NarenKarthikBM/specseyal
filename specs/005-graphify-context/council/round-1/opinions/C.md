This is a well-argued deck — the four problems in §1 are each pinned to a specific, falsifiable measurement (verify-gate.sh's degree 4 at `004`, the 86 stale duplicate nodes at M3, the 753k-token ritual, the 25–38-call/5.25M-token spend at `002`), and §7d/§7e do something most decks don't: tally their own testability gaps against themselves instead of leaving a reviewer to find them. That honesty is a genuine strength and I've tried to match it rather than pile on where the deck has already done the work. Simplicity-lens verdict: the four-arm split by problem is the right shape, and each arm individually is about as small as it can be given what it fixes — but the packaging language overstates how "equal" the four arms are, and two of the deck's own residual risks (R2's drift mitigation, and the query tool I used to check this deck) deserve more than a mention in a risk-register cell.

## Suggestions

- [risk] `graphify explain` can silently resolve a natural, deck-consistent anchor to the *wrong* node with no ambiguity warning — a live footgun in the exact tool this plan's grounding methodology (D10) and arm 4's query-cost accounting both lean on. I ran `graphify explain "member-prompt.md"` to check the deck's §4 claim ("`extensions/council/extension/templates/member-prompt.md` — degree 16") and got back a *different*, degree-1 concept node from an unrelated spec (`Thin member interface (member-prompt.md {{lens}} slot)`, src=`specs/001-council-extension/plan.md`) — no warning that the match was ambiguous. (Contrast: the same session's `path "verify-gate.sh" "implement-parallel"` query *did* print an explicit "warning: source/target match was ambiguous" line.) Only after a follow-up `graphify query` did the actual document node turn up (`Council member base reviewer prompt`, degree 16, referencing `member_lenses`, `plan.md artifact`, and `reduced grounding note (FR-019)` — the deck's claim is correct once you find the right node). Net: the deck's fact holds up, but the path to confirming it has a silent failure mode that would let a rushed reviewer (or a future ceiling-constrained council member burning scarce arm-4 query budget) conclude "ungrounded" when the opposite is true, or waste 2–3 of a capped budget chasing a wrong resolution before noticing. Worth a fixture or at minimum a documented caveat: `explain` should warn on ambiguous matches the same way `path` already does. (confidence: high)
- [testability] §7e's own tally names three guard/complementary branches with zero fixture coverage, and the highest-priority one is arm 2's stale-survivor guard — the exact mechanism built to fix the M3 86-stale-node incident that motivates this whole arm. The committed fixture only exercises the 0-survivors pass-through case; nothing forces survivors > 0 to prove "prune-or-rebuild" actually recovers rather than merely being asserted prose with a guard-shaped name. I grepped `plan.md` for a stated disposition (accept-as-risk vs. defer-to-implementation-task) and found none beyond the same tally the deck already prints — this gap is disclosed but not yet *owned* by a task. Recommend it becomes an explicit committed task (force-inject a stale survivor, assert the guard fires and recovers) before arm 2 ships, not an open item that survives past the council a second time. (confidence: high)
- [simplicity] "Four independently-shippable, independently-reviewable arms" (§2) oversells the actual severability shape. By the plan's own detach-order text, only arm 1 has a working fallback and is genuinely detachable; arms 2, 3, and 4 are declared "core, never-detach" with *no* fallback at all — dropping any of them just leaves the named problem unmitigated. That's one severable arm plus three mandatory ones organized for independent review, not four peers with equal shippability. The underlying reasoning for *why* arm 1 alone is detachable is sound (fallback-driven, not importance-driven, and the deck says so) — the ask here is just that the headline in §2 lead with the 1-vs-3 asymmetry instead of "four independently-shippable," which reads as a stronger severability claim than the next paragraph actually delivers. (confidence: med)
- [risk] Arm 3's own drift mitigation (R2) is shaped like the exact "unenforced prose" failure the plan explicitly rejected an alternative for. §2 rejects "one sectioned file" because "read only your section" is unenforced prose (the D53 lesson) — but R2's mitigation for cross-product drift between the three separate diets is "one generator, one graph pass; a shared-provenance header; goldens per product." A shared-provenance header is only a structural guarantee if some fixture actually asserts all three products carry matching provenance (e.g. same graph hash/generation id) from one generator run; §7a's arm-3 fixture line describes only "a committed golden per product" — each product correct in isolation, not a cross-product coherence check. Recommend either naming that cross-product fixture explicitly or being honest that R2's mitigation is currently process discipline, the same category the sectioned-file alternative was rejected for. (confidence: med)
- [simplicity] Arm 4 splits one guarantee (ceiling-hit disclosure, SC-008 "never silent") across two write paths — mechanical orchestrator enforcement of the cap, plus model-authored disclosure prose from the member itself (§3). The D53 mechanical/model distinction motivating the split is coherent, but for a guarantee specifically billed as "never silent," having the actual disclosure *text* depend on the capped member correctly free-texting it (rather than the orchestrator mechanically appending a templated line the instant it enforces the cap) is a self-imposed way for the guarantee to fail exactly when it matters most — a member scrambling at its ceiling is the one moment its prompt-following is least reliable. Worth the plan stating explicitly why a mechanically-guaranteed disclosure line was passed over. (confidence: med)
- [correctness] Confirm `freshness.sh`'s report is recomputed at every consumption point, not cached/reused across multiple hook calls within one session. §3 is careful to say freshness "writes... no state file (freshness is a derived property, D32)," which reads as a deliberate answer to this exact question, but the deck doesn't say who calls `freshness.sh` when — if any consumer trusts a report computed earlier in the same session rather than re-invoking the check immediately before use, arm 2 would quietly recreate, one layer up, the same "stale artifact trusted without recheck" failure mode (the `002` mtime incident, §1) that this arm exists to close. Likely already true by design given the "derived property" framing — flagging only because the deck doesn't state the invocation contract explicitly enough to rule it out from the deck alone. (confidence: low)
