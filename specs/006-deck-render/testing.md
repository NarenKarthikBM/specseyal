## Verified by reading vs. would-execute in v2

Every row above is `report-claimed`: the confidence rests entirely on `completion-report.md`'s own prose, which was specific enough — naming exact task ids (T018–T037), exact mechanisms (sha256 stamp, `PYTHONPATH`-shadow `ImportError`, `os.replace`, `git ls-files`, spend-equality) and exact section labels matching each SC/FR — that no claim felt thin, generic, or convenient enough to warrant the lazy `implement.log.md` cross-check this session is permitted but not required to make. `implement.log.md` was not opened. The report itself states that T037's quickstart integration gate bound every SC (SC-001…SC-010) and every FR (FR-001…FR-016) to an executed check, which is why the coverage map above closes with zero `GAP` rows — a full 26-id bijection (16 FR + 10 SC), all `covered`.

Two ids carry an irreducibly manual half that the report itself flags rather than silently claiming green: SC-002 and SC-009 both require a human to actually open the rendered `.pptx` in a viewer — something no `run.sh` assertion can mechanize — while their stamp/content-faithfulness halves are machine-checked (T020, T036). A v2, test-*executing* tester would differ from this v1 mainly in degree, not kind: for the `run.sh`-cited rows (nearly all of them) it would actually invoke `sh extensions/deck-render/test/run.sh` and read its exit code and per-section output rather than trust the completion report's narration of a prior run; for the two manual-viewer rows it would still need a human (or a screenshot-diffing harness) in the loop, since "opens in a standard viewer" is not mechanically assertable from inside a shell script.

Two cross-feature ids were excluded from the coverage set as out of scope for this feature's bijection: `001-FR-001` and `001-SC-007`, both cited in `spec.md`'s prose as precedent for the non-technical overview deck, but defined under feature `001`'s own spec, not `006`'s. Additionally, "FR-019" appears twice in `spec.md`'s prose (as "the FR-019 reduced-grounding lineage") but is never defined under this spec's own `### Functional Requirements` heading — which runs only FR-001 through FR-016 — so it was treated as a cross-feature lineage citation, not one of this feature's own ids, and given no row.
