> **Reduced grounding** — no graphify-out/graph.json found; this review is deck-only.

The deck's central simplicity move — a self-contained, zero-hook extension that cannot reach into `council`/`graphify`, deterministic and model-free by construction (§2, §3) — holds up. The rejected-alternatives table (§2) is not a strawman list: hand-rolled OOXML, pandoc, and LibreOffice are each rejected on a stated, checkable ground (boilerplate/silent-failure risk, loss-of-control, absent-on-host), not waved away. The `sha.sh`-vs-`hash-object`-vs-`sha256` reasoning for the stamp (§2, §3) is genuinely load-bearing and correctly resolved. The decision to keep pptx as the output format is a settled owner ruling (D73(3)), not something this plan invented, so I did not re-litigate format choice.

Two gaps I did find sit exactly at the seam between the deck's documentation (`data-model.md`, `profile-schema.md`) and the not-yet-written code, which is precisely where a plan-defense council should catch them before they're locked in.

**Resolution-ladder conflation (data-model.md §1, line 30).** V2 is explicit and absolute: "An out-of-enum value is a hard failure... never falls back to `none`." But the resolution ladder's step 3 — "file absent, unreadable, or key absent ⇒ `none`" — silently folds a *malformed/unreadable* profile.yaml into the same safe default as a genuinely absent one. A profile.yaml that fails to parse (bad YAML from a merge conflict, stray tab, etc.) never reaches the enum check at all, so an operator who intended `deck_render: overview` and got a corrupted file is told nothing — the exact "silent degradation" class V2/SC-008 exist to prevent, just occurring one step upstream of the enum comparison the deck spent a whole rejected-alternative row (§2, "Follow the `council_tier` precedent") arguing against.

**Enum defined twice, with no shared source (§4 Project Structure, data-model.md §1).** The closed set `{none, technical, overview, both}` lives in `profile-schema.md`'s prose contract and will independently live in `profile_key.py`'s code — nothing in plan.md, data-model.md, or the Project Structure names a single canonical definition either side is checked against, and no fixture in the committed set (§4, §7c) asserts the two stay in lockstep. This is the same two-truths shape the deck itself calls out when explaining why `council_tier: standrad` degrading silently is a cautionary precedent (§2, §5) — the risk here is quieter (a doc/code mismatch, not a runtime silent-fallback) but structurally identical.

The deck is also commendably honest about two of its own remaining open branches rather than papering over them (§7c, §7b) — I'm elevating both from "flagged" to "should close before defense," since both are cheaply closeable with a fixture rather than a manual step.

## Suggestions
- [correctness] `data-model.md` §1's resolution ladder step 3 treats an unreadable/malformed `profile.yaml` the same as an absent one (silent `none`), while V2 mandates a hard failure for an out-of-enum value — an unparseable file can hide exactly the malformed value V2 exists to catch. Split the cases: absent file/key ⇒ `none` (V1), but a present-and-unparseable file should not silently resolve to `none`. (confidence: high)
- [simplicity] The `deck_render` closed enum is planned to live independently in `profile-schema.md` (prose) and `profile_key.py` (code) with no named single source of truth and no fixture asserting they agree — the same drift shape the deck cites as `council_tier`'s cautionary precedent. Name one canonical definition (e.g. code exports the enum, the contract doc's table is generated from or tested against it). (confidence: med)
- [testability] §7c flags, unresolved, that no committed fixture forces exactly one deck to fail under `deck_render: both` (the exit-2 partial-code path) — quickstart's "verify by hand once" is the only coverage named anywhere. Add a second, deliberately-invalid fixture deck (a construct outside the narrow surface plan.md line 22 censuses) so SC-004's partial-failure branch is asserted mechanically, not eyeballed once. (confidence: high)
- [testability] §7b names SC-007 (staleness-visible-from-stamp) as "the one genuinely ambiguous entry" — demonstrated in quickstart.md but not confirmed committed to `test/run.sh` in plan.md's Phase C list, despite being fully mechanizable (recompute sha256, assert mismatch — no human judgment needed, unlike SC-002). Commit it rather than leaving it a walkthrough step. (confidence: med)
- [simplicity] §6's Cost/Complexity Estimate says "~2 scripts" but §4's Project Structure lists three (`render.py`, `deck_md.py`, `profile_key.py`). Minor, but §6 is the section explicitly asking the simplicity lens to weigh scope, so its own count should match the tree above it. (confidence: low)
