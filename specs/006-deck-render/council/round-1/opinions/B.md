> **Reduced grounding** — no graphify-out/graph.json found; this review is deck-only.

The deck's central architectural bet — model-free, hook-free, gitignored-output — is sound and each boundary property genuinely does fall out of that shape rather than needing discipline to hold. The rejected-alternatives table (§2) is not a strawman list: hand-rolled OOXML, pandoc, and LibreOffice are each rejected for reasons that would actually bite (silent viewer-rejection risk, loss of control over rendering choices, dependency weight), and the two most tempting shortcuts — folding into `council`, or building a general profile validator — are declined with an honest complexity trade rather than a hand-wave. Where the deck is weaker is in a handful of places where a "guarantee claimed" is mechanically true on its own terms but doesn't hold up against the actual failure mode it's meant to rule out, or where the deck itself flags a gap (§7c, §7b) and I think the flag under-states the risk. Six items below, ordered roughly by how directly the deck's own text supports each one.

The sha256-stamp mechanism (§3.4b, §5 row 3) is the right technical choice over the two rejected alternatives — but "staleness visible from the stamp alone" (SC-007) is a mechanically true claim that doesn't survive contact with the persona the whole feature exists for. §1 names that persona explicitly: someone reviewing "from a phone," or "projecting the argument in a meeting." Neither can eyeball-compare two 64-character hex strings. The deck already flags SC-007's *test-suite* status as ambiguous (§7b) — I'm flagging something adjacent and more basic: even where the check is *performed*, it isn't usable by the audience it was built for. That's worth the council's attention independent of whether SC-007 ends up in `test/run.sh`.

I also want to push on the fidelity check's actual power. §3's "Fidelity verification's own architecture" describes both directions of SC-003 as substring/containment checks: source text is a substring of extracted text, and vice versa. That's a real, independent-reader design (good) — but containment alone cannot detect a defect where content is *present but reordered*, e.g. two H2 sections swapped. T10 explicitly promises paragraphs/bullets/tables "render in source order, never rearranged" — but nothing in the described SC-003 check would catch a violation of that specific promise, since order isn't part of what's being asserted in either substring direction.

Finally, on the two gaps the deck itself already surfaces (§7b, §7c) — I'd escalate rather than just note them: the missing `deck_render: both` partial-failure fixture (§7c, last bullet) leaves the exit-2 path resting on "verify by hand once," which is exactly the shape of gap the deck calls out as risky when discussing round-1's own history. Given SC-008's enum guard *does* get a committed adversarial fixture (`invalid.yaml`), there's no principled reason the partial-render branch shouldn't get the equivalent treatment before implementation, not after.

## Suggestions
- [risk] Step 4c (source markdown → target removed, then written) never states the target write itself is atomic (temp file + `os.replace`) — only `install.sh`'s manifest merge is described that way. If the process dies between removal and a completed write (OOM, kill, disk full), the human loses a previously-good render with no "failed" disclosure ever emitted, since the process never returned to report it. Confirm `render.py` writes via temp-file-then-rename, not directly to `renders/<deck>.pptx`. (confidence: high)
- [risk] §3 step 2's own resolution rule folds "profile.yaml unreadable" into the same `none` outcome as "key absent" — silent, exit 0. But an out-of-enum *value* (a milder problem: the file parses fine) hard-fails at exit 3 (SC-008). A profile.yaml with a YAML syntax error is arguably the worse failure and gets the quieter treatment. Worth deciding deliberately rather than by default. (confidence: high)
- [testability] SC-003's bidirectional fidelity check (§3) is containment-only in both directions and cannot detect a reordering defect, even though T10 promises source order is "never rearranged." Recommend the fidelity check assert block-sequence order, not just presence, or explicitly document that ordering is unverified and rests on code review alone. (confidence: high)
- [risk] SC-007's stamp-based staleness signal is mechanically sound but not usable by the persona §1 names as the feature's reason for existing (phone reviewer, meeting projection) — comparing two sha256 hex strings by eye isn't a realistic check for that audience, independent of whether SC-007 lands in the automated suite. (confidence: med)
- [testability] The `deck_render: both` partial-failure branch (one deck renders, one fails ⇒ exit 2) has no committed fixture anywhere in plan.md/data-model.md/Project Structure (§7c, last bullet, already flagged as ambiguous by the deck itself) — recommend a deliberately-broken second fixture deck before implementation, matching the treatment `invalid.yaml` already gives the enum-guard branch. (confidence: high)
- [risk] The `.specify/extensions.yml` `installed:` list merge point is named by the deck's own §4 as "an *un-instrumented* collision risk, not a verified-safe one," concurrent with `005`'s own registration there — yet no test in Phase C's list targets this specific shared-mutation point. Recommend a minimal reinstall/co-install test against a realistic `005`-plus-`006` `extensions.yml` state, not just SC-010's self/council/graphify reinstall matrix. (confidence: med)
