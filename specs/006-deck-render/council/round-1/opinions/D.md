> **Reduced grounding** — no graphify-out/graph.json found; this review is deck-only.

The overall shape holds up well under an adversarial read: zero hooks, gitignored output, a lazily-imported optional dependency, and a bidirectional independent-extractor fidelity check (§3, §7a) are a genuinely strong foundation, and the rejected-alternatives table (§2) fairly earns its conclusions rather than strawmanning pandoc/LibreOffice/hand-rolled OOXML. The deck's own §7c self-audit of guard falsifiability is unusually honest — most decks don't volunteer their own gaps. But that same self-audit is where the real testability problems live, and it doesn't go far enough.

**O5's "never a partial file" guarantee doesn't match its own mechanism.** §3 step 4c and data-model.md's O5 both claim a failed render "leaves no file, never an old one," and contracts/commands.md line 49 says "never a partial or 'fixed-up' file." But the described mechanism is: delete the *existing* target before the attempt, then write the *new* file directly to the target path (step 4e). That only protects against a stale-previous-run file being mistaken for fresh output — it says nothing about a crash, SIGKILL, or disk-full event partway through the write in step 4e itself, which (given python-pptx's `Presentation.save()` writes a zip archive straight to the destination) would leave a truncated, corrupt file sitting at exactly the path a human or the fidelity check would next read. The only forced-failure test named anywhere (research.md R6, the `PYTHONPATH` ImportError shadow) trips at step 4d — *before* any write starts — so the one failure mode that would actually violate O5 has no fixture exercising it.

**Two guard branches the deck itself never puts a fixture behind.** §7c is candid that the exit-2 partial-failure branch (`deck_render: both`, one deck rendering, one failing) has no committed fixture — Project Structure lists only one symmetric `{technical.md, overview.md}` pair, and quickstart Scenario 2 resorts to "verify by hand once by making one deck unreadable." That's a manual, non-repeatable check standing in for a mechanical one on a guard central to O3/SC-004. Less visibly, the same gap exists for T7's overflow/`(cont.)` branch: the risk register (§5) rates table-cell overflow "High" likelihood and cites T7 as the mitigation, but nothing in Project Structure, data-model.md, or quickstart confirms the frozen fixture (seeded from `005`'s deck) actually exceeds the line budget and forces a continuation slide. Unlike SC-008's `invalid.yaml` or SC-004's ImportError shadow — both explicitly branch-forcing — T7's failure/continuation path is asserted but not demonstrably reachable from a committed fixture.

**SC-007's committed-suite status is left as an open question rather than resolved.** §7b calls this out itself as "the one genuinely ambiguous entry," admitting staleness detection — load-bearing for FR-003/FR-008 — isn't named in plan.md's Phase C list. A council review is the right place to close that ambiguity before it becomes an implementation-time surprise.

**The footer's "short SHA" is unspecified.** data-model.md line 89 defines an "abbreviated stamp (declaration + short SHA)" for every slide's footer but never states a length. research.md R3's entire argument for choosing 64-hex sha256 over a commit SHA is that it's "self-evidently not" a 40-hex git SHA — if an implementer truncates the footer to a git-abbreviation-like length (7-8 hex) with no spec constraint against it, every slide but the title slide reintroduces exactly the confusion R3 was written to prevent, and no scenario in quickstart exercises staleness detection from the footer's truncated form (Scenario 6 only checks the title slide's full 64-hex stamp).

## Suggestions
- [correctness] Make the write in step 4e atomic (write to a temp file in the target directory, `os.replace()` into place only on full success) and add a fixture that forces a failure *during* the transform/write (not just at the pre-write ImportError check) to prove O5's "never a partial file" claim actually holds. (confidence: high)
- [testability] Commit a fixture that deliberately pairs one valid deck with one broken deck under `deck_render: both` to mechanically exercise the exit-2 partial-failure branch — §7c already flags this as ambiguous and quickstart Scenario 2 currently relies on a one-time manual check for a guard central to O3/SC-004. (confidence: high)
- [testability] Confirm (or add) a fixture that actually exceeds T7's line budget and forces a `(cont.)` continuation slide — the risk register rates overflow "High" likelihood, but unlike SC-008's and SC-004's guards, no committed fixture is named as branch-forcing for T7. (confidence: med)
- [testability] Resolve SC-007's committed-suite status before the plan is finalized rather than carrying the deck's own §7b-flagged ambiguity into implementation as an open question. (confidence: med)
- [correctness] Specify the footer's "short SHA" length explicitly and confirm SC-007's staleness check also works from the footer's truncated form — as written it's unbounded and could collide in appearance with a git short-SHA, undermining research.md R3's stated rationale on every slide but the title slide. (confidence: med)
