> **Reduced grounding** — no graphify-out/graph.json found; this review is deck-only.

The chosen shape (mechanical script, zero hooks, gitignored output) is sound and the rejected-alternatives table is honestly argued, not a strawman — hand-rolled OOXML and pandoc/LibreOffice are rejected for real, stated reasons rather than being set up to lose. The sha256-content-stamp-over-commit-SHA decision (§2) is well-reasoned and internally consistent with the git log's own framing of the design change. My concerns are concentrated in three places: the exit-code table has real unaddressed combinations, the "skip vs. malformed input" boundary in selection-resolution is under-specified relative to what FR-006/SC-008 promise, and two sections of the deck give contradictory characterizations of the same shared-file collision risk.

**Exit-code table (§3, "Exit codes").** The table enumerates 0 / 2 / 3 / 4 but only names four scenarios: all-rendered-or-nothing-selected-or-skipped-absent (0), partial rendered+failed under `both` (2), invalid input (3), all-failed (4). Under `deck_render: both` there are six possible per-deck-pair outcomes (rendered+rendered, rendered+failed, rendered+skipped, failed+failed, failed+skipped, skipped+skipped), and the table's prose only unambiguously covers three of them. A rendered+skipped pair and a failed+skipped pair are not named anywhere — is a `failed+skipped` pair "all selected renders failed" (exit 4, since the only real render attempt failed) or something else? Is `rendered+skipped` folded into exit 0's "or a deck skipped because its markdown is absent" clause even though one deck did render? This is discoverable purely from the deck's own text, not a grounding question — the mapping should be made exhaustive before implementation, not left to whatever `render.py` happens to do.

**Selection-resolution's "unreadable" catch-all (§3 step 2).** The deck states the resolution order defaults to `none` when "the file/key is absent or unreadable," while an out-of-enum value is a hard `exit 3` failure that "never silently degrades to `none`" (FR-006/SC-008's stated guarantee). But "unreadable" is not further defined: does it cover only I/O-level failures (permission denied, file vanished mid-read), or does it also swallow a YAML *parse* error (malformed `profile.yaml` syntax)? If the latter, a profile author who breaks their YAML gets silent no-op behavior — exactly the silent-degrade failure mode FR-006/SC-008 exists to prevent, just approached from a different malformed-input shape than "enum violation." The committed fixture set (`{none,overview,both,invalid,absent-key}.yaml`, §4 Project Structure) has no fixture for unparseable YAML — only key-absence and enum-violation are exercised, so this boundary is untested either way.

**§4 vs. §5 contradict each other on the `.specify/extensions.yml` collision.** §4's own Dependency/Graph Impact section says the installer's merge into `extensions.yml`'s `installed:` list "should [be] treat[ed]... as an *un-instrumented* collision risk, not a verified-safe one" specifically because there's no graph to check it against. But §5's risk register, addressing the same touchpoint from the sequencing angle (`005` merging around when `006`'s council convenes), asserts flatly "No file-level dependency exists in either direction." Both `005` and `006` write to the same `installed:` list via their respective install/uninstall lifecycles (per §4's own Project Structure) — that is a file-level dependency by definition, even if it's a benign append-only one. The two sections should be reconciled: either the risk register's "no dependency" claim is wrong, or §4's caution is overstated, but they can't both stand as written.

## Suggestions
- [correctness] The exit-code table (§3) only unambiguously covers 3 of 6 possible per-deck-pair outcomes under `deck_render: both`; `rendered+skipped` and `failed+skipped` combinations are not mapped to an exit code anywhere in the deck. (confidence: high)
- [correctness] §3 step 2's "absent or unreadable ⇒ silent `none`" branch is not disambiguated from a YAML parse error on a present `profile.yaml`; if malformed YAML is folded into "unreadable," that silently swallows a broken config exactly the way FR-006/SC-008 says an out-of-enum value must never be swallowed. No committed fixture tests malformed/unparseable YAML. (confidence: med)
- [correctness] §4's "un-instrumented collision risk" characterization of the shared `.specify/extensions.yml` `installed:` merge point directly contradicts §5's risk-register claim that "no file-level dependency exists in either direction" between `005` and `006` — both extensions write to the same list. (confidence: med)
- [correctness] O5's guarantee ("a failed render leaves no file, never an old one") relies on step 4c deleting the prior render before attempting, but nothing asserts the write in step 4e is atomic (temp-file + rename). If `render.py` writes `python-pptx`'s output directly to the final path, a mid-write crash (OOM, disk-full, kill) would leave a truncated `.pptx` at the target, violating the stated guarantee. The deck cites this exact atomic-write pattern for install.sh's manifest merge but never asserts it for the render write. (confidence: med)
- [testability] §7c already self-flags that no committed fixture exercises the `both`-mode partial-failure branch (one deck renders, one fails ⇒ exit 2) — quickstart calls it "verify by hand once." Recommend the council require a committed asymmetric two-deck fixture pair before this ships, rather than leaving exit 2 as a manual, non-repeatable check. (confidence: high)
- [testability] SC-007 (staleness visibility) is the risk register's own "load-bearing" SC (FR-003/FR-008) yet §7b confirms it is the one genuinely ambiguous entry — demonstrated via quickstart commands but not named in plan.md's Phase C committed-suite list. This gap doesn't appear anywhere in the §5 risk register itself and should be resolved (commit it or explicitly justify the exclusion) rather than left implicit. (confidence: med)
