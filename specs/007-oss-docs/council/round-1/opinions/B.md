This is a well-argued deck — the two-arm split is genuinely independent, the FR-019 hook is correctly isolated as a cuttable task, and §7's own falsifiability audit (P1/P4 both-branch confirmed, P3-pass flagged as a gap) is more self-critical than most decks bother to be. My review presses on where the deck's confidence outruns what's actually verifiable, and on the operational risk of flipping FR-019 on for real.

**The deck's central urgency claim does not hold up against the repo's actual history.** §1 states, unhedged: "This is not hypothetical: a real committed profile currently carries `council_tier: standrad` (a typo) and degrades silently rather than failing." I checked this directly rather than taking it on faith, since it's the single concrete instance grounding the whole "not hypothetical" framing (and by extension, part of D82's owner-scope rationale for folding I-27 into this feature). `git log --all -p -- '**/profile.yaml'` across the entire repository history returns zero matches for "standrad" — no committed `profile.yaml`, at any point, ever contained it. None of the 7 profile.yaml files present today (000, 001, 003–007) contain it either. Every occurrence of "standrad" in the repo is prose (docs/90's D79(2)/I-27 narrative, research.md, this feature's own spec/plan/quickstart/contracts as a *synthetic test fixture value*). The general defect class I-27 documents (no mechanical enforcement, `reopen_tier` dead, prose-only validation) is real and independently well-sourced — I'm not disputing that — but the specific "a real committed profile currently carries [this typo]" sentence appears to be a rhetorical carry-over from the D79(2) illustrative narrative, restated in this deck as present-tense fact rather than the historical/hypothetical example it actually is. Compare: the deck is careful to mark the graph-staleness claim as "independently re-verified against graph.json this session" (§4) — this claim gets no such flag, and doesn't survive one.

**No plan-time step verifies the repo's existing committed profiles against the new validator before FR-019's fail-closed hook goes live.** §5's risk register rates the hook's impact as "Medium," reasoning that "only genuinely malformed profiles are blocked" and "an absent profile still passes." That reasoning covers the *known* fixture classes (§7's V1–V6) but not the operational question: has anyone run `validate-profile.py` against every profile.yaml actually committed in this repo right now, to confirm the strict rules (unknown-key-is-error in particular) don't retroactively hard-block an in-flight feature's `plan` phase the moment the hook is wired in? Nothing in the deck's testability table (§7) or Project Structure (§4) names this as a check. This is a one-time, cheap smoke test (`for f in specs/*/profile.yaml; do validate-profile.py "$f"; done`) that costs nothing and directly de-risks a "flag day" surprise — worth naming as a task if FR-019 is ratified.

**The interpreter-ladder's last resort risks a hang, not the promised loud failure.** §3 step 3 lists `uv run --with pyyaml python` as the final fallback when no PyYAML-capable interpreter is found in-process, and the "Guarantee" immediately after promises "a loud non-zero failure — never silently treated as absent." But `uv run --with pyyaml` can attempt to resolve/fetch the `pyyaml` package on first use; in an offline or network-restricted CI environment this is a stall, not a clean non-zero exit — and an unbounded hang is arguably worse than a silent wrong-answer, since it also breaks SC-010's "<2s" claim (§7) with no timeout mentioned anywhere in the deck's description of this step.

**The `deck_render` subsumption mechanism is left as an open choice with an unequal-risk pair of options.** §3 step 5 and §5's risk register both present "runtime import, or a module constant a committed equivalence test pins equal" as interchangeable, deferred to implementation. They aren't equally risky: a runtime import of `profile_key` from `extensions/workforce/.../validate-profile.py` into `extensions/deck-render/.../profile_key.py` is a cross-extension import with no package structure visible in §4's tree (flat script directories, not installed packages) — that's `sys.path`/`importlib` mechanics that can break on refactor in ways a pinned constant + equivalence test simply cannot. The deck should recommend the pinned-constant approach now rather than leaving this ambiguous for whoever implements it to pick the fragile option by default.

**§7's own flagged P3-pass gap is higher-stakes than an ordinary untested branch, and deserves elevation rather than a footnote.** The deck is right to flag that no fixture is named for `full_auto: true` + both gates `auto` passing cleanly. From a risk lens: this is precisely the profile shape a genuine full-auto run depends on to proceed *unsupervised*. An untested pass branch elsewhere just means weaker coverage; an untested pass branch here means the one path where no human is in the loop by design has zero regression evidence that it actually validates cleanly. This should be a named, required fixture before FR-019 (or even Arm B alone) ships, not left as an open question for task-writing time to maybe resolve.

**The schema-doc-vs-validator-code pairing gets an equivalence test for exactly one field, not the schema as a whole.** §5's risk register names a committed equivalence test guarding `deck_render` against drifting from `profile_key.DECK_RENDER_ENUM`. No equivalent mechanism is proposed for the rest of the schema: `validate-profile.py`'s hardcoded rules (required keys, `council_tier` enum, the P1–P5 handshake logic) and `docs/contracts/profile-schema.md`'s prose are two independent representations of the same contract, and D-e explicitly anticipates `profile-schema.md` will be amended later (to address `reopen_tier`/`max_rounds`). Nothing here says the validator's hardcoded rules get updated in lockstep, or that any test would catch it if they didn't — which reintroduces, one level up, the exact prose-vs-enforcement gap this feature exists to close for `profile.yaml` itself.

## Suggestions
- [correctness] The deck's §1 claim "a real committed profile currently carries `council_tier: standrad`... this is not hypothetical" is not supported by `git log --all -p -- '**/profile.yaml'` (zero matches, ever) or by any of the 7 currently-committed profile.yaml files — it should be corrected to cite this as I-27/D79(2)'s illustrative example, not a present-tense live defect, before the council relies on it. (confidence: high)
- [risk] Add a plan-time task to run the new validator against every currently-committed `specs/*/profile.yaml` as a one-time smoke test before FR-019's fail-closed hook is wired live, to catch any retroactive hard-block of an in-flight feature on adoption day. (confidence: high)
- [risk] The interpreter ladder's `uv run --with pyyaml python` fallback (§3 step 3) can attempt a network fetch and hang in offline/restricted environments rather than failing loud — add an explicit timeout or drop this rung, since an indefinite stall contradicts both the "loud non-zero failure" guarantee and SC-010's <2s claim. (confidence: med)
- [risk] Commit now to the pinned-constant + equivalence-test mechanism for `DECK_RENDER_ENUM` consumption (§3 step 5, §5) rather than leaving "runtime import or pinned constant" open — a cross-extension runtime import between two flat script directories is measurably more fragile with no offsetting benefit. (confidence: med)
- [testability] Elevate §7's own flagged P3-pass gap (no fixture confirms `full_auto: true` + both gates `auto` validates cleanly) from a noted observation to a required committed fixture before ship — it's the one path a genuine full-auto run depends on passing unsupervised, making it higher-stakes than an ordinary untested branch. (confidence: med)
- [risk] No equivalence-test-style guard exists between `profile-schema.md`'s prose and `validate-profile.py`'s hardcoded rules for anything other than the single `deck_render` field — since D-e already anticipates a future schema amendment, note this as a known follow-on drift risk (schema-doc vs. validator-code) rather than leaving it unaddressed. (confidence: low)
