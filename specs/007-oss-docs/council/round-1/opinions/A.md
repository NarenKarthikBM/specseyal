This plan is unusually well-defended for a "docs feature" — Arm B's rule table, risk register, and testability claim are internally cross-checked in a way most decks don't bother with, and the deck author already self-flags one real gap (the P3-pass fixture question in §7) rather than burying it. My review focused on whether the *rule set itself* is complete and internally consistent, since that's the part no amount of good process hygiene around it can paper over.

I independently confirmed one grounding claim rather than taking it on faith: `graphify explain "profile_key.py"` returns "No node matching 'profile_key.py' found" — the deck's staleness claim holds.

I also opened `docs/contracts/profile-schema.md` directly (not just the deck's paraphrase of it) because two of the deck's claims about the schema's rule set were specific enough to be worth checking against the primary contract rather than trusting the rendering. That reading is the basis for suggestions 1 and 3 below.

**Strongest part of the deck**: §7's "Guard/branch falsifiability check" is genuinely good adversarial self-review — it's rare for a deck to name its own untested pass-branch rather than let a reviewer find it. My suggestion #2 just insists that self-flagged gap gets resolved as a concrete plan-time commitment rather than carried forward as an open question.

**Weakest part**: the rule-evaluation step in §3.4 is a flattened summary of `profile-schema.md`'s rule table, and flattening lost one distinction that matters — see #1 and #3.

## Suggestions

- [correctness] §3 step 4 says required keys are checked "present and typed" for `schema_version`, but never states whether the validator checks it against the literal value the contract's own schema block and both canonical examples always show (`"1.0"`). `profile-schema.md` §3 lists `schema_version` as "Required. Bump on any breaking change to this file" — implying an expected exact value, not an arbitrary string. If the validator only type-checks (any string passes), a typo'd or wrong `schema_version` (e.g. `"1.O"`, `"2.0"`) would silently pass — the exact silent-typo failure class this whole feature exists to close (the `council_tier: standrad` precedent cited in §1). This should be resolved explicitly before implementation (exact-match vs. type-only), and a fixture case added either way — none of the SC-007/SC-009 fixture classes in §7's table currently cover a `schema_version` mismatch. (confidence: high)

- [testability] §7 already names an unresolved gap: no fixture is confirmed to exercise the P3-pass branch (`full_auto: true` + both gates `auto` ⇒ VALID) — `conformant.yaml`'s `full_auto` value isn't stated in any plan-time artifact read for this deck, so if it isn't itself that profile, the strictest-rule pass branch has no committed exercise at all. The deck correctly flags this as "not yet something plan-time artifacts resolve either way," but leaves it there. It shouldn't ship to tasks.md unresolved: either confirm `conformant.yaml` covers it, or require a dedicated `full-auto-clean.yaml`-style fixture as part of V3/V6's committed set before SC-007 can be marked done. (confidence: high)

- [correctness] The deck (§1, §5 risk row 4) treats `reopen_tier` and `max_rounds` as an equivalent pair of "dead/duplicated keys" deferred to a future reconciliation (this framing traces back to `plan.md`'s D-e, so the deck faithfully transcribes the plan — the concern is with the underlying claim, not the transcription). But `profile-schema.md` §3 describes `reopen_tier` as driving real, distinct triage behavior *today*: "`auto` = triage proposes, human gate confirms. `delta`/`full` pin the tier and skip the proposal" — that's a live consumer, unlike `max_rounds`, which the same contract says is genuinely inert ("Validators reject `> 1` until the convergence rule is revised"). Bundling them under one "Low impact, just a documented dead key" risk rating understates that a conforming-but-wrong `reopen_tier` value has real behavioral stakes (whether a reopened plan sees a human) beyond "we'll still catch the typo." The validator's mechanical enum check catches a typo in either key regardless, so this doesn't change Arm B's code — but the risk register's "Low" impact characterization for this row deserves a second look, or at minimum shouldn't imply `reopen_tier` is as functionally dormant as `max_rounds`. (confidence: med)

- [risk] §4's "independently re-verified against graph.json" framing is honestly caveated (staleness, no import edges, `profile_key.py` absent from the graph — which I independently reproduced), but the practical consequence is under-stated: essentially none of *this feature's* actual new coupling (the new `validate-profile.py`, its consumption of `profile_key.DECK_RENDER_ENUM`, the `.specify/extensions.yml` hook if ratified) is graph-verifiable pre-implementation — the graph only confirms two *unrelated* sibling validators exist as a structural analogy. The "no collision" claims in the shared/mutable-file collision watch therefore rest entirely on stated plan intent, not on anything the graph actually confirms. Worth a housekeeping task to refresh `graphify-out/graph.json` after this feature lands, so FR-018's "no divergent copy" claim becomes independently checkable going forward rather than permanently resting on the equivalence test alone. (confidence: med)

- [correctness] §6's own component breakdown doesn't foot to its stated range: the named implementation items are 6 doc dispatches (README/CONTRIBUTING/CODE_OF_CONDUCT/SECURITY/issue-templates/PR-template) + "one or more" for validator+harness+fixtures + a conditional FR-019 hook dispatch — a minimum of 7, not the "~6-8" the section states, since even the cheapest reading (1 validator dispatch, 0 hook) is 7. That makes the final "15-18 sessions" ballpark (10 fixed + 6-8) more accurately "17-19+" by the deck's own arithmetic. Low-stakes, but a cost estimate meant to be load-bearing for council/human review should foot to its own listed components. (confidence: low)

- [risk] The dual tracked-copy requirement (`extensions/workforce/extension/scripts/validate-profile.py` + the `.specify/extensions/workforce/scripts/` installed mirror, §4 Project Structure) has no stated sync-verification mechanism, unlike FR-018's `deck_render` subsumption, which gets an explicit committed equivalence test. If the two copies are kept identical by an existing repo-wide install mechanism, this is a non-issue and the deck could simply say so; if not, a drifted install copy would be a real (if narrow) failure mode with no named fixture or test-coverage row for it in §7's table. (confidence: low)
