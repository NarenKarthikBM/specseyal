The deck is unusually disciplined about its own testability claims — §7's tally and the closing "guard/branch falsifiability check" already do half of an adversarial reviewer's job by naming a real gap against itself. That self-awareness is a genuine strength of the plan, but it also means the highest-signal thing I can do is push on exactly the seams the deck left open rather than relitigate settled ground (Arm A's manual-only tier is an explicit, reasoned scope call, not an oversight — I'm not re-opening it).

Arm B's testability story is otherwise strong: SC-007/SC-008/SC-009/FR-018 are each backed by a committed, both-branch fixture per the deck's own P1/P4 falsifiability check, and the "validator writes nothing" framing (Constitution I) keeps the surface small enough to actually fixture exhaustively. My concerns cluster around three places where the deck's confidence outruns what it actually shows me: the one gap it names but doesn't resolve, and two branches whose "both-branch coverage" claim I can't verify from the deck's own text.

On the P2/P3 pass-branch gap (§7, closing paragraph): the deck is right to flag it, and right not to silently paper over it. But it stops one step short of being actionable — it names the risk (`conformant.yaml`'s `full_auto` value is "not stated in any plan-time artifact read for this deck") without stating what should happen next. Since this is genuinely load-bearing for FR-017's "100% correct verdict across a fixture set covering both branches of every targeted class" claim, this needs to be closed at task-writing time, not carried forward as an open question into implementation — either confirm `conformant.yaml` already is the full_auto:true/both-auto profile, or add a dedicated `full-auto-satisfied.yaml` fixture.

On the interpreter-ladder guarantee (§3 step 3): the deck asserts "unreachable-interpreter cases (fail)" sit in the committed V3/V6 fixture set alongside the P1 absent-file case, but never explains the mechanism. "No PyYAML-capable interpreter reachable" is an environment property, not a file property — you can't fixture it the way you fixture a malformed YAML file; it requires either mocking the interpreter-ladder probe or manipulating `PATH`/available interpreters in the test harness itself. If the harness doesn't actually do one of those, this is a guarantee that reads as tested in the deck's tally but has no real failing-branch exercise behind it — the same false-confidence shape the deck itself worries about for P2/P3, just unnamed.

On message-content assertions vs. exit codes: step 6's guarantee is explicitly about content ("a human-readable stderr message naming the offending key... never an opaque parser traceback"), not just exit status. A regression that degrades a clean message into a raw traceback would still exit non-zero — so if `test_profile.sh`'s fixture assertions only check exit codes (as V2/V3's framing in §7 implies — "rejected, not silently degraded" is verified, but the message-quality guarantee isn't separately called out), the "never opaque traceback" claim is a documented but functionally untested promise.

## Suggestions
- [testability] Close the P2/P3 pass-branch fixture gap the deck itself names in §7: confirm at task-writing time whether `conformant.yaml` is a `full_auto: true`/both-gates-`auto` profile, and if not, add a dedicated fixture for it — otherwise FR-017's "100% correct verdict, both branches of every targeted class" claim has an unexercised branch in its own committed harness. (confidence: high)
- [testability] Verify how the "unreachable PyYAML-capable interpreter ⇒ loud non-zero failure" branch (step 3) is actually fixture-tested — this is an environment condition, not a file condition, and the deck doesn't describe a mocking/PATH-manipulation mechanism for it. If the committed harness can't deterministically force this branch, it isn't really both-branch-covered despite being listed alongside P1 in §7. (confidence: med)
- [testability] Confirm `test_profile.sh` asserts on stderr *message content* ("names the offending key," no opaque traceback) for at least one fixture, not only on exit code — the exit-code-only check can't distinguish a clean rejection from a regressed traceback that happens to also exit non-zero, which is exactly the guarantee step 6 promises. (confidence: med)
- [testability] SC-010's <2s claim is marked "Partial" in the deck's own tally — it's cheap to close: add a wall-clock assertion inside `test_profile.sh` (or a dedicated fixture) rather than leaving the timing bound as a quickstart-only, non-committed check. (confidence: med)
- [testability] The deck states "unknown key at any level ⇒ error" but only describes the fixture set as "1 conformant + 1-per-malformed-class" without confirming nesting depth is covered — e.g. an unknown key at top level (`profile.yaml`'s root) vs. one nested inside `gates.council`. A recursive-validation bug could pass one level and fail the other; confirm both are separately fixtured at task-writing time. (confidence: low)
- [testability] Arm A's FR-010/FR-011 grep checks (D2/D3) are explicitly "mechanical, but not CI-committed" (§7) — meaning even the automatable half of Arm A's testability story has zero re-run mechanism once this feature closes. Since the front door's paths/commands will inevitably drift as later features land, recommend committing the D2/D3 grep as a standalone re-runnable script now (no CI wiring required, matching the feature's scope boundary) so a future feature doesn't have to re-invent the check from scratch to re-verify SC-003/SC-004. (confidence: med)
