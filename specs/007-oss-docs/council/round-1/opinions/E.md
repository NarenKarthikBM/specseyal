This plan bundles two well-separated arms, and Arm A (the OSS front door) is exactly as inert as the deck claims — five independent docs with no runtime data flow and a mechanical grep gate. My review focuses on Arm B, specifically the ordering/timing questions around FR-019 enforcement, because that is where "when does this check actually run relative to when the thing it's guarding gets read" matters most.

**The central finding: `before_plan` is a single point-in-time check on a file that stays mutable for the rest of the feature's life, and the deck's central safety claim doesn't hold past that instant.**

Deck §3 and §5 both assert "no gate ever reasons about an invalid profile" once the `before_plan` hook is wired, because validation sits "at the pipeline's front, before council or tasks." I went and read `.specify/extensions.yml` directly to check this claim, since it's exactly the kind of assertion this council exists to pressure-test. What I found: the repo already has a working, precedented pattern for exactly this shape of enforcement — `speckit.git.verify-gate` is registered as a **mandatory, fail-closed** hook at *both* `before_tasks` (checks the council gate/plan.md freshness) *and* `before_implement` (checks the workforce gate/tasks.md+assignment.md freshness). That pattern exists precisely because the artifacts it guards can be edited after an earlier phase completes, and a single early check wouldn't catch a later edit. `profile.yaml` is exactly this kind of artifact — nothing in `plan.md` (I checked; grep for "hook" at lines 11, 91, 106, 114, 118) proposes SHA-pinning it into `gates.yml` the way `plan.md`/`tasks.md` already are. So: if a profile is edited after `before_plan` fires — to fix a typo, to flip `council_tier`, to change a gate mode — nothing re-checks it before `/speckit-council` actually reads `council_tier` to decide ceremony tier, nor before `after_council_approve`/`after_workforce_approve` record whatever gate state is in the file at that moment, nor before `before_implement`'s existing hook acts on `gates.workforce`. The deck's own two-option framing (`plan.md` line 114: "a fail-closed `before_plan` hook" vs. "standalone-only, no hook") never considers the third option that the repo's own hook history would suggest: mirror `verify-gate` and validate at `before_tasks`/`before_implement` too (or instead), i.e. at the actual moments the profile's fields get consumed rather than at the earliest possible moment before any of them are read. This isn't a hypothetical edge case — it's the same class of gap the existing `verify-gate` hooks were built to close for `plan.md`/`tasks.md`, just left open for `profile.yaml`.

**Second: a gap the deck itself surfaces but defers past the point where deferral is safe.** §7's falsifiability check flags that no fixture is named for the P3-pass branch (`full_auto: true` + both gates `auto`, validating cleanly) and calls this "not yet something plan-time artifacts resolve either way." But `tasks.md` is generated mechanically from `plan.md`'s Project Structure fixture list (§4 names only `conformant.yaml`, `full-auto-unsatisfied.yaml`, and the enum-equivalence pin explicitly). If the missing fixture isn't written into that list now, at plan-review time, there is no later mechanical step that reintroduces it — `/speckit-tasks` has no independent way to know a fixture is missing that the plan never named. SC-007's "100% correct verdict across a fixture set covering both branches of every targeted class" would then read as satisfied by `test_profile.sh` while actually leaving the handshake's strictest pass branch (P3) never exercised. A gap noticed at deck-review time needs to be written back into `plan.md` before tasks generation, not left as a note hoping a later phase picks it up.

**Third, a risk-register omission with a sequencing cause.** Today, `profile_key.py`'s interpreter-ladder PyYAML probe only gates deck-rendering — one optional feature, exercised late. If FR-019's `before_plan` hook is ratified, the *same* fragile probe (reused verbatim per D-b) becomes the mandatory front gate of `/speckit-plan` for every feature, everywhere. An environment where no PyYAML-capable interpreter is reachable turns from "deck-render degrades" into "planning is hard-blocked repo-wide," and that failure mode is indistinguishable at the exit-code level from a genuinely malformed profile (deck §3 step 3: both are "loud non-zero failures"). The Risk Register (§5) covers gate-coupling, enum drift, doc staleness, and leakage, but not this promotion-of-blast-radius effect from reuse at a new, mandatory, universal chokepoint. Worth a named row, or at minimum an explicit fallback discussion (e.g., does the hook distinguish "interpreter unreachable" from "profile invalid" in its message, even if both are non-zero?).

## Suggestions
- [sequencing] The `before_plan`-only enforcement point leaves `profile.yaml` unguarded after the check fires — nothing re-validates it before `/speckit-council` consumes `council_tier`, or before `after_council_approve`/`after_workforce_approve`/`before_implement` act on `gates.*`, since (unlike `plan.md`/`tasks.md`) it is never SHA-pinned into `gates.yml`. `plan.md` (line 114) presents only "before_plan hook" vs. "standalone-only" as the two options; it should also weigh mirroring the existing `verify-gate` pattern at `before_tasks`/`before_implement` (the actual profile-consumption points) before the council locks in a single early checkpoint. (confidence: high)
- [testability] §7's named P3-pass fixture gap (no committed case for `full_auto: true` + both gates `auto`) must be written into `plan.md`'s Project Structure fixture list now — before `/speckit-tasks` runs — or the gap has no mechanical path to ever become a task, and SC-007's "100% ... covering both branches" claim will silently overstate coverage. (confidence: high)
- [risk] Ratifying the `before_plan` hook promotes `profile_key.py`'s interpreter-ladder PyYAML probe from gating one optional feature (deck-render) to gating the mandatory front of every feature's plan phase; an environment lacking a reachable PyYAML-capable interpreter becomes indistinguishable, at the exit-code level, from a genuinely malformed profile, and repo-wide planning hard-blocks either way. This blast-radius promotion isn't named in the Risk Register (§5). (confidence: med)
- [sequencing] The FR-019 hook-wiring task and the validator-authoring/fixture tasks should carry an explicit dependency edge (hook wiring strictly after the validator + its full `test_profile.sh` suite passes), not just an informal "separable" framing — otherwise a parallel implementation wave could register the fail-closed hook before the script it depends on is hardened, hard-blocking `plan` on the validator's own bugs rather than on real profile defects. (confidence: med)
- [sequencing] The new mandatory `before_plan` hook's priority relative to the existing optional `graphify` `before_plan` hook (currently priority 5) is unspecified in the deck/plan; the two existing fail-closed `verify-gate` hooks both run at priority 1. Pin the new hook to priority 1 explicitly so its fail-closed check is guaranteed to run deterministically rather than depending on hook-list ordering. (confidence: low)
