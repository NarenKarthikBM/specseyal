This plan is unusually well-argued for itself — the deck pre-empts most of the obvious objections (dependency-free YAML acquisition, read-only enum consumption instead of an edit, a separable FR-019 hook, an explicit rejected-alternatives table). That makes it worth pressing on the places where the deck's own evidence, read carefully, cuts against its confidence rather than for it.

**On bundling.** The deck states plainly that Arm A (docs) and Arm B (validator) "share no code." The deck's own §7 SC-tally proves this isn't just a code-sharing gap but a *testability* gap: every Arm B claim (SC-007/008/009, FR-018) has a committed automated test; every Arm A claim (SC-001/002/003/004/005/006/011) is manual-only or grep-only. Two arms with disjoint code, disjoint risk profiles, and disjoint testability ceilings are being pushed through one plan, one council round, and (presumably) one tasks.md. D82's rationale — "last α-polish feature before the visibility flip, cheapest real win in the pile" — justifies why *both* land before the same deadline, but not why they need to be *one* artifact set rather than two features tracked toward the same deadline. If Arm B's review stalls (it's the one touching gate-correctness territory per D79(2)), Arm A's five independently-parallelizable, zero-blast-radius docs shouldn't be hostage to that stall, and vice versa.

**On the deck_render subsumption mechanism (FR-018).** The plan leaves the choice between "runtime import" and "pinned module constant + equivalence test" as an unresolved "implement-phase detail" (research.md D-d), and the Risk Register's #2 row exists *only* because that choice is still open — a pinned constant can drift; an import structurally cannot, because there is no second copy to drift from. That means the equivalence test in V4 is a compensating control for a risk the plan could eliminate outright by just deciding "import" now, unless there's an architectural reason (extension-to-extension import boundary?) the plan doesn't state for why import is off the table. As written, the plan pays for an extra committed test to guard against a risk it could have designed away.

**On the P2/P3 pass-branch fixture (verified against plan.md, not just the deck).** I checked `plan.md`'s Project Structure listing directly: the named fixtures are `conformant.yaml`, `out-of-enum.yaml`, `unknown-key.yaml`, `full-auto-unsatisfied.yaml`, and an unenumerated "..." (gate-scalar, feature-mismatch, max_rounds>1, bad-YAML, deck_render-out-of-enum). No fixture name in that list corresponds to the P2/P3 *pass* branch — `full_auto: true` **and** both gates `auto`, which the handshake requires to validate cleanly. Only the fail branch (`full-auto-unsatisfied.yaml`) is named. `conformant.yaml`'s shape is not specified anywhere plan-time, and this feature's own `profile.yaml` (§6 of the deck) is `council_tier: standard`, both gates `human` — i.e., not itself an instance of the strictest-handshake-pass case, so there's no existing example in this repo to point to as a template either. This is the deck's own self-flagged gap (§7), and my read of `plan.md` confirms it isn't resolved by anything named in the task-relevant artifacts yet — it should become an explicit fixture name (e.g. `full-auto-satisfied.yaml`) in the plan rather than left implicit inside "conformant."

**On schema_version.** The rule-evaluation list requires `schema_version` to be "present and typed," but nowhere does the deck state that its *value* is checked against the version the validator actually implements (v1.2). Under the stated "unknown key at any level ⇒ error" policy, a future schema bump would surface as an unrelated pile of missing/unknown-key errors rather than one clear "this validator implements v1.2, your profile declares v1.0/v1.3" message — a small addition now that meaningfully simplifies the failure mode at the next schema revision, which the plan already anticipates happening (D-e explicitly defers schema-content changes to later).

**On the FR-019 hard-block having no stated escape hatch.** If ratified, every future feature's `plan` phase is fail-closed behind this validator. The deck treats this purely as a gate-coupling proportionality question (D79(2), already well-argued in the Risk Register). The narrower angle not addressed: if the validator itself has a bug that false-positives against a profile shape it wasn't tested on (plan.md's stated validation subject set is only the 6 present + 2 absent `specs/00{0..7}` profiles), there's no stated bypass — a validator defect becomes "nothing can enter planning" rather than a contained failure. Worth a one-line statement of whether an override exists (or an explicit acceptance that there isn't one and why that's fine for a fail-closed hook this narrow).

## Suggestions
- [simplicity] Track Arm A and Arm B as separable task sequences (already true in code terms — "share no code") so a stall in one arm's review/implementation doesn't block the other; the SC-tally split in the deck's own §7 shows they already have disjoint testability profiles. (confidence: med)
- [simplicity] Resolve the FR-018 subsumption mechanism now (favor a runtime import over a pinned constant) rather than deferring to implement time — an import eliminates Risk Register row 2 (drift) structurally and makes the equivalence test unnecessary; state explicitly if there's an architectural reason imports across `extensions/*` trees are disallowed, since the plan doesn't currently say why the drift-prone option is still on the table. (confidence: high)
- [testability] Name the P2/P3 pass-branch fixture explicitly in plan.md's Project Structure (e.g. `full-auto-satisfied.yaml`) — verified directly against plan.md that no listed fixture name covers `full_auto: true` + both gates `auto`, and this feature's own profile.yaml isn't an instance of that case either, so nothing plan-time currently guarantees the strictest handshake rule's pass branch is ever exercised by a committed test. (confidence: high)
- [simplicity] Validate `schema_version`'s value against the version the validator implements (not just its type/presence), so a future schema revision produces one clear version-mismatch message instead of a pile of unrelated missing/unknown-key errors. (confidence: med)
- [risk] State whether the FR-019 `before_plan` hook has any override/bypass if the validator itself has a bug and false-positives on an untested-but-legitimate profile shape — today's stated validation subject set is only 8 fixture profiles (`specs/00{0..7}`), and a fail-closed hook with zero escape hatch turns a validator defect into a total planning outage rather than a contained one. (confidence: med)
