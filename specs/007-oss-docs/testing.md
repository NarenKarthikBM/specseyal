## Verified by reading vs. would-execute in v2

Every row above rests on `completion-report.md` prose alone (`report-claimed`) — none are `log-verified`. One lazy cross-check was made, per the R1-S05/D10 pattern: FR-013 and FR-014 (the general profile-contract-check requirement and its per-malformed-class failure requirement) are grounded more thinly than the other Arm B rows — the report names one concrete enum example (`council_tier`) and one concrete `full_auto`-handshake state (`P1`, "an absent path exits 0"), plus a generic "malformed classes" fixture-set description, but never itemizes an "unknown key" case by name. That thinness was enough doubt to open `implement.log.md`. It turned out to be a four-line wave-outcome ledger (timestamp, task ids, agent count, `outcome: success` per wave) with no per-fixture detail — it neither corroborates nor contradicts the specific claim, so per the honesty rule it earned no promotion to `log-verified`; both rows stayed `report-claimed` on the strength of the report's own (real, if not maximally itemized) prose. No row was downgraded to `GAP` — the underlying report claims are genuine even where less granular.

A future v2, test-*executing* tester would replace most of these reads with actual runs: for Arm A (FR-001–012, SC-001–006) it would execute `scripts/check-oss-docs.sh` itself and walk the quickstart's `D1–D5` steps directly, rather than trusting that the T017 gate ran them; for Arm B (FR-013–019, SC-007–010) it would run `test_profile.sh` and assert on each individual fixture — the enum case, the unknown-key case, and each `full_auto` P1–P5 handshake state — rather than accepting the report's grouped "malformed classes" description, and it would re-time the `<2s` wall-clock assertion itself rather than citing the committed `R1-S18` assertion as given. SC-011 is inherently a point-in-time-of-close judgment; a v2 tester would re-check it at actual merge/close time rather than relying on the gate's prospective claim.
