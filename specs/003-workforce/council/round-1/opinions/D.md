This plan's testability story is unusually disciplined for a plan-stage document — deck §6 renders four SCs as literal code assertions (D58-5), and names a single committed golden test (`extensions/agents/test/run.sh`) for the property the whole feature exists to guarantee (SC-005 byte-reproducibility). Pressure-testing the deck's own claims against `quickstart.md` (the artifact the deck's FR/SC table cites as verification for nearly every SC) turns up a real gap, though, between what is *claimed tested* and what the described tests can *actually reach*.

The sharpest finding is R1/SC-005 itself. The risk register attributes the determinism-boundary mitigation to "enforced by the golden test," but `quickstart.md`'s own description of that test says "Zero-AI, so CI-runnable without a model" — it feeds a frozen fixture straight into `assemble.py` and never invokes the skill-builder ("agent-creator" in `plan.md`'s Constitution Check table) Sonnet session at all. A test that never runs the LLM cannot be evidence about what the LLM does when it runs. What the golden test actually proves is that Python's `sorted(key=...)` is deterministic across two invocations on unchanged input — true, but never seriously in doubt. R1's actual fear (a real skill-builder session "helpfully" influencing ranking on a real gap) is untested by anything the deck names.

SC-006's golden test has the same shape of problem from the other direction: it can only prove the *passing* case. Every base in the v0 seed library declares `model: sonnet` (deck §2's seed table, confirmed again by quickstart S7's own wording), and the project's model policy retires Haiku for v1 entirely — so there is no base anywhere in the planned system that could make `assert selected_base.model == "sonnet"` fail. The D48 guard's `else: hard error` branch is dead code with respect to every fixture the plan describes; the golden test demonstrates the assertion holds, never that the guard actually *guards*.

Zooming out: of the plan's nine numbered SCs, only three (SC-004, SC-005, SC-006) are named as covered by the committed golden test. The remaining six — SC-001, SC-002, SC-003, SC-007, SC-008, SC-009 — plus S9 (additive-only rejection) are validated exclusively by `quickstart.md`'s manual "Expect:" runbook steps, with no committed script named for any of them. That's a defensible M3-exit dogfood story, but it means most of the SC surface has no regression protection once the dogfood run has passed once — nothing stops a later change from quietly breaking the 20% `general` cap or trace-roster matching.

Two of those manual-only SCs deserve a closer look because they're the ones the deck itself foregrounds. SC-002's claim is a conjunction — non-zero exit *and* no `categorization.md` written — but every cited verification (deck §6a, quickstart S2) checks only the script's exit code; the causal link from "script exits non-zero" to "the file was never written" is itself orchestration prose (the calling command has to correctly gate the write on that exit code), exactly the class of fragility R1 flags elsewhere. SC-008 fares similarly: quickstart S10's only check is "inspect traces.jsonl... matching its approved roster row" — inspection language, not a named script that cross-checks each trace against its roster row.

## Suggestions
- [testability] R1's mitigation is credited to "the golden test," but that test is explicitly zero-AI and never invokes the skill-builder session — it can only prove `assemble.py` is internally deterministic, not that the LLM session stays out of the ranking. Either reframe R1's mitigation to what the golden test actually covers, or add a real (non-zero-AI, but still committed) integration test that runs the actual skill-builder on a gap fixture and asserts the resulting roster reflects only `assemble.py`'s algorithm over the skill's declared tags/grants, not any other influence. (confidence: high)
- [testability] SC-006's D48 Sonnet-floor guard cannot fail under any fixture the plan describes: every v0 seed base is `model: sonnet` and Haiku is retired for v1, so no base exists to trip the `else: hard error` branch. The golden test proves the assertion passes, never that the guard fires. Add a synthetic, test-fixture-only non-Sonnet base to the frozen library snapshot so the failure branch is actually exercised at least once. (confidence: high)
- [testability] Only SC-004/005/006 are covered by the committed golden test in `extensions/agents/test/run.sh`; SC-001, SC-002, SC-003, SC-007, SC-008, SC-009, and S9 rely solely on `quickstart.md`'s manual "Expect:" steps, with no committed script named. Before `tasks.md` is generated, decide per-SC whether that's an accepted v0 scope call or whether some of these (SC-002's cap, SC-008's trace-match) should get a committed fixture too — right now the deck presents all nine as equally "testable" when only three carry regression protection. (confidence: high)
- [testability] SC-002's verification (deck §6a, quickstart S2) checks only `validate-categorization.py`'s exit code; the SC itself also claims "no `categorization.md` written," which depends on the calling command correctly gating the write on that exit code — untested orchestration prose. Assert file-absence (or an unchanged-directory diff) directly in the test rather than inferring it from the exit code alone. (confidence: med)
- [testability] SC-008's only named check (quickstart S10) is "inspect traces.jsonl... matching its approved roster row" — human inspection, not an automated cross-check, and the only SC in the FR/SC table phrased that way. Name a small script that mechanically diffs each dispatch trace's `skills[]`/`elevated_grants[]` against its roster row, rather than relying on eyeballing a roster that could have dozens of rows. (confidence: med)
- [testability] SC-007's "exactly one persisted SKILL.md" is tested (S8) with a single novel-tagged task, but the skill-builder is described elsewhere as running "per gap batch" — implying one session can face multiple gaps at once. No fixture covers two tasks sharing the same novel tag in one run, which is exactly where "exactly one" (dedup) vs. "one per task" (duplicate/collision) would be caught or missed. Add that variant to S8. (confidence: med)
