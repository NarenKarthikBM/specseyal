The deck's central determinism argument — LLM-categorize (non-reproducible) → code-assemble (deterministic) → LLM-skill-build (non-reproducible, gap-only) — is honestly scoped where it matters most. I cross-checked the deck's SC-005 framing against `spec.md`'s FR-015/SC-005/FR-022 text (the 2026-07-10 clarification): SC-005 is explicitly assembly-only, against a fixed library snapshot, and "gap-free" is made a checkable roster property (zero `built` marks, FR-022) rather than an assertion. That boundary is well-drawn; it is not where I'd spend council time.

Where I do have doubts is one level down, in whether `assemble.py`'s actual algorithm — as specified in `agent-library-schema.md` §3/§4/§6 and `skill-module.md` §6, which I read in full to check the deck's rendering — genuinely closes the loop the deck claims it does. The deck's §2/§6 pseudocode is a faithful summary of the base-lookup, Jaccard-rank, cap-3, and D48-guard steps. But three separate places where the plan leans on an LLM session (or a Python built-in) "just doing the right thing" have no code-level backstop, and one of the deck's own testability claims disagrees with an artifact plan.md itself produces. None of these break the headline SC-005 argument, but each is a plausible source of a silent, hard-to-diagnose failure exactly in the property (byte-reproducibility, gap-closure, enum-safety) the feature exists to guarantee.

## Suggestions
- [correctness] `agent-library-schema.md` §3 step 3 computes `grants = ⋃ {s.grants : s ∈ injected}` with no subsequent sort, and nothing later in the algorithm re-sorts it before it reaches the roster. CPython's `set` iteration order for string-keyed sets depends on the process's hash seed (`PYTHONHASHSEED`, randomized per-process by default since 3.3); two separate `assemble.py` invocations over the identical `categorization.md` + library can iterate an identical logical grant-set in a different order. If that order is written unsorted into `agents/assignment.md`'s Elevated-grants column, SC-005's "byte-identical rosters on gap-free runs" claim can fail intermittently — the same silent-non-determinism failure mode the deck explicitly rejects Bash/`yq`/`awk` over (§2 Rejected alternatives: "a locale- or flag-dependent sort would silently break determinism"), but for Python's own `set` type, which that rejected-alternatives rationale never mentions. Recommend the plan state explicitly that `grants` (and any other set-typed intermediate) is sorted before serialization, and that the SC-005 golden test asserts on grant *order*, not just membership. (confidence: high)
- [correctness] SC-006's golden test can only prove the D48 guard's affirmative path, not that it enforces anything. `agent-library-schema.md` §4 already validates, independently of `assemble.py`, that "a base whose `taxonomy.type` contains any of the 7 implementation types and whose `model ≠ sonnet` is invalid" — and the deck's own seed table (§2) states every one of the 7 seed bases is `model: sonnet`. Given that, `assemble.py`'s D48 assert (`if "prompt" in task.tags: assert selected_base.model == "sonnet"`) has no seed-library input that can ever fail it — the "`prompt`×`docs`×`ai-agents` lands on Sonnet" golden test (deck §6b, plan.md's D58-5b) proves the assignment is correct, not that the guard rejects a violation. The guard isn't redundant — it's the only protection against a hypothetical future `docs`-accepting-but-non-implementation-typed base slipping under §4's validator — but that's the failure mode that should be under test. Recommend a fixture with a deliberately non-Sonnet, `docs`-accepting base, asserting `assemble.py` hard-errors on it; as scoped now, a regression that silently drops the assert would pass every described test. (confidence: high)
- [correctness] Nothing enforces that a skill-builder's generated `SKILL.md` actually closes the gap that triggered it. The pipeline (deck §2 diagram) assumes one skill-build pass yields a "gap-free" re-run, but `skill-module.md` §6's 8 validation rules (frontmatter/id/version/origin, non-empty kebab-case `tags`, `grants⊄core`, `body_sha256`, S1-S3 body discipline) never check that the new skill's `taxonomy.tags` intersect the originating task's `tags`, and `spec.md`'s FR-006 only requires the new module to conform to `skill-module.md` — not to match the gap. If the builder session picks tags that don't overlap, `assemble.py`'s re-run is still gapped, and the deck's Risk Register (R1-R5) doesn't cover this path — it's the same class of "prose-enforced judgment with no code backstop" the deck flags as R1 for the assembler boundary, just one hop earlier, at categorize→skill-build. Recommend either a code check (in `assemble.py`'s re-run or `validate-skill.py`) that the generated skill's tags intersect the triggering task's tags, or an explicit defined behavior for a still-gapped task after one pass (retry / hard-fail / fall to `agt_generic` with a reported gap). (confidence: high)
- [correctness] The deck's own FR→verification table (§6) and `plan.md`'s D58-5 report disagree with `plan.md`'s file tree on whether SC-001/FR-002 (100% field coverage, closed-enum values) is code-enforced. The deck lists SC-001's method as "Quickstart S1 ... inspect `categorization.md`" (manual) — contrast SC-002's explicit "code-level, D58-5a" — and `plan.md`'s D58-5 section books exactly three code-verification items (FR-004/SC-002, FR-014/SC-006, FR-007/FR-011/SC-004); the fourth D58-5 item is the `web_search` decision, not a code item, so SC-001 is genuinely outside that set. Yet `plan.md`'s file tree annotates `validate-categorization.py` as "CODE cap-check **+ enum/coverage validation** (FR-004/SC-002)" — crediting the same script with enum/coverage validation, which is FR-002/SC-001's own claim, while labeling it only under the FR-004/SC-002 tag. If `validate-categorization.py` really does check enum-closure and field-coverage before the cap-check, SC-001 should get the same "code-level" credit SC-002 gets in both the deck and the D58-5 report; if it doesn't, `assemble.py`'s base lookup (`t.type ∈ b.taxonomy.type`) is implicitly trusting un-validated enum values out of `categorization.md`. Worth reconciling before triage either way. (confidence: med)
- [correctness] The seed-library table (deck §2) reviews only `specialization` per base; `agent-library-schema.md` §3's actual lookup key is `(type, specialization)`, and `taxonomy.type` is a *list* per base (§1.1's own example: `type: [service, endpoint]`). Rule 5's lane-uniqueness (§6.5) happens to hold trivially here since all 7 seed specializations are pairwise distinct — that part checks out — but the deck gives no per-base `type` list (only that `agt_ai_agents` "accepts `docs`" is mentioned in passing), so the council is asked to review this set (§2: "the plan's proposal; the council reviews this set") without the data needed to check whether the 7 bases' type-lists jointly cover the taxonomy's 8 types, or whether some type is left completely unclaimed and silently defaults every task of that type to `agt_generic`. Recommend the deck (or `plan.md`) publish each seed base's full `taxonomy.type` list alongside `specialization` for review. (confidence: med)
