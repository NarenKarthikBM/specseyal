The deck's spine — categorization is inference, assembly is code, skill-building is inference-on-gap-only — is the right cut, and I'm not contesting it. `assemble.py` being the sole matching authority is exactly what makes SC-005 possible, and the Python-over-Bash call in the Rejected Alternatives table (§2) is honestly argued: a locale- or flag-dependent shell sort silently breaking a total-order tie-break is a real, known Bash hazard, not a strawman. Where the plan strains under a simplicity read is in two structural choices that get less scrutiny than the machinery they authorize.

**Two extensions for one strictly-sequential pipeline.** Deck §5 names "two extensions delivered as one feature — a doubled install/uninstall/test/packaging surface" as complexity driver #1, and §3 adds a second cost on top: `categorize`'s `after_categorize` and `agents`' `after_agent-assign` both have to register hooks into `speckit.git.commit` (R7) because a cross-extension coupling can never be a source edit (D57). That's real, non-trivial machinery spent solely on keeping two things separate that are described everywhere else in the deck as one thing — "D40 makes them one composable system," a strict `tasks.md → categorization.md → assignment.md` pipeline with no independent use for `categorize` without `agent-assign` ever mentioned. I checked whether "council" actually supports the split, since the deck cites it twice (§3: "`extensions/council/` is additionally the multi-command pattern `agents` should mirror"; plan.md's Structure Decision: "the git/council sibling pattern"). It doesn't — `extensions/council/` is one extension directory holding three distinct command-skills (`speckit-council`, `speckit-council-approve`, `speckit-council-triage`; confirmed via `find extensions/council -maxdepth 2` and corroborated by a `graphify query` on the chairman/council nodes, all sourced under the single `.specify/extensions/council/` tree). That is precedent for *one* extension exposing multiple phase-commands, i.e. the opposite of what's being proposed for `categorize`+`agents`. And yet plan.md's own Complexity Tracking table (the constitution's designated place to defend exactly this kind of cost, "why needed / simpler alternative rejected because") contains a single row — Python vs. Bash — and never examines "why not one `workforce` extension with two commands." The largest complexity item the deck itself names never faces the table built to defend it.

**The seed library applies its own evidentiary bar unevenly.** Deck §2's Rejected Alternatives table refuses to seed the four unseeded specializations because doing so would "invent... 5 lanes with zero dogfood evidence" — a hard rule (taxonomy §5, "an unexercised enum value is a guess"). But `agt_backend_service` and `agt_data_persistence` are seeded on "the taxonomy worked-example's backend-service/data-persistence cluster" — plan.md line 120 draws its own distinction here, listing "the two real dogfood features" (001, 002) and then, separately, "*plus* the taxonomy worked-example's... cluster." A worked example in a contract doc is not a dogfood run; it's exactly the "zero dogfood evidence" category the plan uses to reject the other four lanes. R3 (seed-library overfit) only names `ai-agents`/`devtools-cli` as the SpecSeyal-overfit risk, which means two of the seven bases are carrying weaker evidence than even the risk register accounts for.

I did not open spec.md — nothing in the deck's spec-derived claims (US1-4, the FR-004 cap, D9/D41 gate framing) looked doubtful enough to warrant it under the lazy budget.

## Suggestions
- [simplicity] Fold `speckit-ext-categorize` and `speckit-ext-agents` into one `workforce` extension with two commands (`categorize`, `agent-assign`), mirroring `extensions/council/`'s own multi-skill-single-extension shape (confirmed: `speckit-council`/`speckit-council-approve`/`speckit-council-triage` all live under one `extensions/council/` tree). This removes the doubled install/uninstall/test/packaging surface *and* the `after_categorize`/`after_agent-assign` cross-extension hook coupling into `speckit.git.commit` (R7) — an internal two-step command sequence needs no hook-mediated coupling at all. If there's a reason the two must stay independently installable, that reason belongs in the Complexity Tracking table, which currently doesn't mention this choice at all. (confidence: high)
- [simplicity] `agt_backend_service` and `agt_data_persistence` fail the plan's own seeding bar ("zero dogfood evidence" — the exact reason the other four lanes stay unseeded): their sole justification is "the taxonomy worked-example," which plan.md itself lists separately from "the two real dogfood features." Either drop both to `agt_generic` like the other unproven lanes (5 bases + fallback, closer to docs/05's "5–6 specialists"), or extend R3's overfit-risk language to name them explicitly so the council is approving a seed library with two evidentiary tiers, not one. (confidence: high)
- [simplicity] Deck §5's complexity item "(2) Python introduced as a second scripting runtime alongside Bash" overstates the cost against plan.md's own Technical Context ("graphify already establishes Python 3 as the repo's scripting runtime"; "No new runtime dependency beyond what graphify installs"). The Python-over-Bash decision is sound on its own merits (see above) — it doesn't need a "new runtime" framing to justify it, and double-counting an already-installed dependency inflates the cost bar the whole feature is measured against. (confidence: med)
- [simplicity] The `web_search` open question (§2/R4) should be resolved as a firm default, not left as a preference: Option B is strictly more machinery than Option A on every axis the plan itself lists (a new grant site, a mandatory trace flag, a mandatory roster note) for a benefit — richer first-draft skill bodies — that's speculative until a concrete authoring-quality gap shows up. Recommend locking Option A now and revisiting only if practice demonstrates the gap plan.md concedes ("reversibility... trivially add later"). (confidence: med)
- [simplicity] Confirm the hand-rolled frontmatter parser (PyYAML rejected in favor of "a minimal stdlib parser over the `specseyal:` block," plan.md Technical Context) is one shared, independently unit-tested module consumed by both `assemble.py` and `validate-skill.py`, not two separately hand-rolled copies. A second bespoke parser is exactly the kind of silent-divergence risk (a quoting or whitespace edge case parsed differently in two places) that motivated rejecting Bash for `assemble.py` in the first place — it would be inconsistent to accept that risk in duplicated Python instead. (confidence: low)
