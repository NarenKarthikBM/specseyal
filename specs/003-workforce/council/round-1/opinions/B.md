The deck is unusually candid about its own risk surface — R1-R5 name real, non-strawman hazards, and the recommendation on the `web_search` branch-point (§2) is the correct risk call: the first elevated grant in the system staying skill-declared and gate-visible is worth more than richer first-draft skill bodies, and the asymmetry the deck itself notes (trivial to add later, hard to walk back once skills assume live lookup) is the right framing. But two of the deck's own load-bearing claims don't survive a check against the artifacts they cite, and three more risks — one in the seed's own reinstall story, one in the determinism boundary's enforcement mechanism, and one in the grant-audit chain's test coverage — are either understated or newly surfaced here.

**§3's "degree 15" claim for `.specify/extensions.yml` is not what the cited graph shows.** `graphify explain ".specify/extensions.yml"` returns a node with **Degree: 1** and exactly one inbound edge (`Installer hook-merge into .specify/extensions.yml [references]`), not 15. A repo-wide grep for the literal string turns up 66 hits — including all 15 currently-installed `.claude/skills/*/SKILL.md` files plus 6 install/uninstall scripts, already more than the deck's own number before counting anything else. This matters specifically for the risk lens because §3 uses "degree 15... the single highest-collision file in the repo" as the *evidentiary basis* for the "serialize, never parallelize" collision-watch policy on the exact file both new extensions (`categorize`'s and `agent`'s installers) must hook-merge into. The true blast radius is larger than either number on offer, and the policy that's supposed to protect it ("installer merge, never a source overwrite") is, on inspection of `extensions/git/install.sh`'s actual merge script, a plain read-modify-write with no lock file and no atomic rename — safe under the repo's current install-one-extension-at-a-time-by-hand discipline, but 003 is the first feature to ship two new, hook-coupled extensions in one round, and nothing in the deck states or enforces an install *order* between them.

**R5's stated hazard ("installers `rm -rf` this tree") is narrower than the deck thinks, but the real hazard it's reaching for is sharper.** Reading `extensions/git/install.sh` directly: the `rm -rf` is scoped per named skill directory (`rm -rf "${SKILLS_DEST}/$name"` inside a loop over the installer's *own* `skills/*/` payload) — a foreign extension's reinstall can only ever touch directory names it ships itself, so it can never blow away an `agents`-flywheel skill by accident. Quickstart S11 as described ("a prior `built` skill survives a foreign-extension reinstall") is testing a property the existing installer convention already guarantees repo-wide — it's not 003-specific insurance. The hazard that convention does *not* rule out is a **name collision**: `agents/install.sh`'s own seed payload ships `skl_refactor_discipline`, `skl_orchestration`, `skl_shell_scripting`, `skl_yaml_hooks`, `skl_installer_hygiene` by name, so a *self*-reinstall of `agents` (e.g. after the taxonomy §8 v0→v1 seed revision the deck itself flags as planned) will `rm -rf` and overwrite any of those five names unconditionally — including a flywheel-generated skill that happens to have landed on the same name, or a future 6th seed name that collides with something the builder already generated. Neither the deck's mitigation nor S11 as scoped seems to target self-reinstall-vs-generated-namesake.

**R1's mitigation is validated at the wrong layer.** The deck's own text says "the session **only** dispatches skill-building and **writes the artifact**" — i.e. it is the *session*, not `assemble.py`, that performs the write in pipeline step 4 (step 3 is `assemble.py`'s re-run producing "final roster"; step 4, "write `agents/assignment.md`," is listed as a separate action). That means SC-005 byte-reproducibility depends on two stacked assumptions: the session never re-ranks (R1's named concern), *and* the session transcribes `assemble.py`'s output into the artifact without alteration (an unstated second assumption). A golden test built the natural way — calling `assemble.py` as a pure function twice and byte-diffing — validates the first assumption's mechanism but not the second, and not R1's actual failure mode (judgment leaking into a *live* session on a *non-frozen* run). The deck already flags this exact tension for the council ("Explicitly flagged... to pressure-test whether command prose can truly hold this line") — my answer is: it can't, reliably, and the fix is cheaper than it sounds — make `assemble.py` itself the writer of `agents/assignment.md` (or at least of the Workforce Gate table), invoked as a tool call, so the session is structurally unable to hand-transcribe. That converts a prose promise into a tool-permission fact, which is exactly the D53 lesson the deck itself cites.

Two smaller points. R2's "mechanical phase, no trace on gap-free runs" is a reasonable classification, but it leaves the roster — a security-consequential artifact per D9/D41 — with no machine-checkable record of *which* library snapshot produced it beyond git history; a cheap fix is available since `assemble.py` already holds everything needed to stamp one. And §6's testability table has an explicit golden test for every assembly property *except* one: nothing asserts that the roster's Elevated-grants column is a complete, correct union when multiple injected skills each contribute grants — given D41's stakes, a silent drop or bad union here is precisely the failure mode the audit invariant exists to prevent, and it's the one place in §6 that gets asserted coverage rather than demonstrated coverage.

On the `web_search` question itself (§2): Option A is right on audit grounds, but the deck frames the choice purely as gate-visibility vs. richness. There's a third, unaddressed axis — when the builder *doesn't* stub-and-grant, it authors "from tags... and Claude's own knowledge," and nothing distinguishes a skill body written from potentially-stale trained knowledge about a fast-moving dependency from one that's actually current. S1–S3 validation is structural (negation, model/dispatch keys, relaxed obligation), not factual, so a confidently-wrong technical claim in a generated `SKILL.md` passes validation silently. This doesn't change the Option A recommendation, but the plan should name the builder's own decision rule for stub-vs-assert, because right now that boundary is exactly as prose-enforced as R1's.

## Suggestions
- [risk] `.specify/extensions.yml`'s "degree 15 / single highest-collision file" claim (deck §3) is not supported by `graphify explain` (returns degree 1, one inbound edge) or by a repo grep (66 hits, already >15 from `.claude/skills/*/SKILL.md` + install/uninstall scripts alone). Re-derive the number from the actual graph or drop the specific figure, and — since 003 is the first feature to install two new, hook-coupled extensions into this file in one round — state an explicit install order or add a lock/atomic-rename to the existing read-modify-write YAML merge rather than relying on "serialize, never parallelize" as an unenforced policy. (confidence: high)
- [risk] R5's mitigation and quickstart S11 target a hazard (foreign-extension reinstall clobbering the flywheel tree) that `extensions/git/install.sh`'s confirmed per-name `rm -rf` pattern already makes structurally impossible. The real gap is a name collision between a flywheel-generated skill and (a) `agents/install.sh`'s own seed names on self-reinstall, or (b) a future seed addition (taxonomy §8 v0→v1) that happens to match an already-generated skill's name. Require the skill-builder to check the live `.claude/skills/` listing before naming a new skill, and make reinstall skip-if-`origin:generated` rather than always-overwrite-by-name. (confidence: high)
- [risk] R1's golden test, built the natural way (calling `assemble.py` directly, twice), validates function-level determinism but not the failure mode R1 names — judgment leaking into the live session that the deck's own text says "writes the artifact" (pipeline step 4, separate from `assemble.py`'s step-3 re-run). Make `assemble.py` the actual writer of `agents/assignment.md`'s Workforce Gate table (session invokes it as a tool call, never hand-transcribes) so the boundary is a tool-permission fact, not a prose instruction — the D53 lesson the deck itself cites. (confidence: high)
- [risk] The `web_search` Option A recommendation is right on audit grounds but doesn't name the builder's decision rule for when to emit a stub-with-grant vs. author confidently from "Claude's own knowledge." S1–S3 skill validation is structural, not factual, so a stale/wrong technical claim in a generated `SKILL.md` about a fast-moving dependency passes silently. Name the heuristic, or mark builder-authored (non-stub) bodies with a provenance flag distinct from a stub's. (confidence: med)
- [risk] R2's "no trace on gap-free runs" classification is reasonable but leaves the roster with no machine-checkable record of which library snapshot produced it beyond git history. Have `assemble.py` stamp a content-hash of the consulted base+skill set into `agents/assignment.md` itself — cheap, no new trace writer, doesn't reopen R8's rejected gates.yml-duplication path. (confidence: med)
- [risk] §6's testability table has a named golden test for every assembly property except grant-union correctness: SC-003 only asserts the Elevated-grants column is never *omitted*, not that it's a complete/correct union when ≥2 injected skills each contribute grants. Given D41's stakes, add a golden test with a multi-grant-skill fixture asserting no grant is silently dropped or mis-merged. (confidence: med)
