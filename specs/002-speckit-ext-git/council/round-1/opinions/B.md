The plan's risk register (R1-R5) is honest about the seams it names but is calibrated against a mental model of the codebase that direct inspection does not support. I checked every load-bearing claim against the actual files in this repo (not just `graphify-context.md`, which itself turns out to be one of the stale sources) and found the plan's own "blast radius" and "no lost work" claims fail on contact with the files that already exist.

## 1. The Constitution Check's blast-radius claim is false, and it's false in the direction that matters most

`plan.md`'s "Dependency / graph impact" section states: *"The **only** existing file touched is `.specify/extensions.yml`... No source file is mutated."* The Constitution Check repeats this under principle I: *"Ownership (artifact-layout §6) is unchanged."*

This is contradicted by the plan's own §D/§F and by `contracts/commands.md`'s "Gate-command integration points (the R1 seam)" section, which requires:
- `/speckit-council-approve` to gain a call recording `plan.md @ $(speckit.git.sha plan.md)` into the `## Human Gate` section it writes.
- `/speckit-implement-parallel` to gain a call to `speckit.git.commit impl "wave K/N …"` after each wave (D-R7 in `research.md`).

I read both actual files. `.claude/skills/speckit-council-approve/SKILL.md` is byte-identical (`diff` confirms) to `extensions/council/skills/speckit-council-approve/SKILL.md` — it is **council's** payload, not git-ext's or stock spec-kit's, and contains zero mention of SHA today. `.claude/skills/speckit-implement-parallel/SKILL.md` is likewise byte-identical to `extensions/graphify/skills/speckit-implement-parallel/SKILL.md` — it is **graphify's** payload. Its "Execute waves" loop (steps 1-5) has no commit step at all, hand or scripted — `research.md`'s D-R7 claims "`implement-parallel` **already** commits per wave by hand (the M1 trail: `impl(001) wave K/8`)" as justification, but that behavior lives nowhere in the skill file; it was the M1 orchestrating session's own ad hoc discipline, not an instruction the skill encodes. So D-R7's empirical premise is itself unverified by the artifact it cites.

Both owning installers (`extensions/council/install.sh`, `extensions/graphify/install.sh`) do `rm -rf "${SKILLS_DEST:?}/$name"; cp -R "$skill_dir" "$SKILLS_DEST/$name"` on every (re)install — confirmed by reading both scripts. So whatever edit git-ext's build makes to satisfy FR-005/FR-008 either:
(a) lives only in the *installed* copies at `.claude/skills/`, and is silently discarded the next time council or graphify is reinstalled (a fresh clone + reinstall-both-extensions, or a version bump) — with no error, no warning, just a quietly reverted integration point; or
(b) requires git-ext's own build/install process to reach into and edit **another extension's source tree** (`extensions/council/skills/...`, `extensions/graphify/skills/...`) — an ownership coupling the risk register never names, and one that is *exactly* the pattern the plan explicitly rejected elsewhere: "Editing the stock spec-kit skills directly to insert commits... Rejected: hooks plus a commit primitive keep a Spec Kit re-init non-clobbering — the same compatibility guarantee graphify and council already rely on." That guarantee is precisely what breaks here — just against extension-owned skills instead of spec-kit's own.

R1 as written frames this purely as "will the calling command remember to call the primitive." The actual failure mode is worse and structural: the call site has to be hand-written into a file this plan doesn't list as touched, owned by a different extension whose own lifecycle can silently erase it. This deserves to be R1's headline risk, not a footnote.

## 2. Two-feature race turns R3's own mitigation into a silent merge, not a re-run

The dispatch asked directly what happens if two features race. I traced it: `create-new-feature.sh`'s `get_highest_from_specs()` does a plain directory scan of `specs/` to pick the next `NNN` (classic TOCTOU), and the existence guard right after (`[ -d "$FEATURE_DIR" ] && [ "$ALLOW_EXISTING" != true ]`) is followed by `mkdir -p`, which does not fail if the directory now exists. Two concurrent `/speckit-specify` invocations in the same checkout (two engineers, or two concurrent agent sessions — plausible in this pipeline's own multi-agent orchestration model) can both compute the same `NNN`.

R3 frames re-entrancy only as "if `specify` is re-run" for **the same feature**, and its mitigation — "`branch.sh` is create-if-absent-then-checkout (FR-012)" — is exactly wrong for the race case: applied to two genuinely *different* features that collided on `NNN`, "create if absent, else switch to it" makes the second racer silently check out onto the first racer's branch and directory, with **no error at all**. A hard git failure ("branch already exists") would at least surface the collision loudly; the idempotency mitigation specifically built to make same-feature re-runs painless converts a two-feature collision into a silent merge of two unrelated specs onto one branch. R3's likelihood ("Low... the mitigation makes this effectively unreachable") is answering a different, narrower question than the one the register's own title ("Branch-at-`after_specify` when the base already advanced / re-run") implies it covers.

## 3. R1's "no lost work, only coarser checkpoints" claim has a real ordering hole during interruption itself

The plan's mitigation for a missed per-wave call is: "the phase-boundary hooks backstop any missed per-wave/gate call... it does not lose work." That's true for interruption *between* phases. It is not obviously true for interruption *during* `implement`, which is the one scenario US2/SC-006 actually cares about — and by definition, if you're mid-multi-wave-run, `after_implement` (the claimed backstop) has not fired yet.

Walking the actual (shipped) `speckit-implement-parallel/SKILL.md` loop: step 3 marks a task `[X]` in `tasks.md` **per task, before** step 5 (Log); the proposed per-wave commit call, per D-R7, lands after logging — i.e., after all of a wave's tasks are already marked done in `tasks.md`. An interruption landing between "last task of wave N marked `[X]`" and "wave N's commit lands" leaves `tasks.md` asserting wave N is complete while git has not committed it — and the *same* interruption can simultaneously leave wave N+1's subagents' partial file writes sitting uncommitted in the identical working tree. Nothing in the design distinguishes "wave N, done, just missing its commit" from "wave N+1, partial, should be discarded" at that point — both are simply dirty files in one tree. `SC-006`'s "resumes... with nothing duplicated or lost" is not obviously satisfied by "the phase hook backstops it," because the phase hook cannot backstop a run that never reached its own end.

## 4. R2's "narrow window" framing is least true for exactly the phase where it matters most

`contracts/commands.md`'s hook table fires `before_implement` only at "start of `/speckit-implement*`" — once, for the whole run. `implement` is explicitly the long, multi-wave, hours-scale, most-interruption-prone phase (the plan's own words, §Summary). So a `tasks.md`/`assignment.md` edit landing an hour into a running multi-wave implementation rides unchecked for every remaining wave of that same run — the entry-only check's blind spot is structurally widest on the one phase with the highest token/time cost per undetected staleness window, not narrow as R2's Med/Med rating and "narrow but real" language suggests for phases like `tasks` (short, single-shot, where "narrow" is fair).

## 5. Two stale documents already assert the *rejected* `before_specify` design — including the deck's own cited source

`.claude/skills/speckit-specify/SKILL.md` (the actual, live, currently-installed skill) states as unconditional fact, in its final NOTE: **"Branch creation is handled by the `before_specify` hook (git extension)"** — and its Outline step 2 describes reading `BRANCH_NAME`/`FEATURE_NUM` JSON output from a `before_specify` hook. This is precisely the alternative plan.md rejects: *"`before_specify` branch creation (stock spec-kit's own model)... Rejected: fires before the spec ID exists."* `docs/contracts/artifact-layout.md` §2 **was** correctly amended for D51 (lines 76/91 confirm: "co-incident with `specify`... before `spec.md` is committed") — so the contract is right and the live skill file is stale relative to it.

Worse: this feature's own `graphify-context.md` — the deck's cited grounding source — repeats the same stale claim: *"002 needs this variant (it registers `before_specify` + commit hooks)."* The deck itself correctly describes `after_specify` throughout, but its own primary grounding document disagrees with it. Per this council's own grounding rule, a graph/context source disagreeing with the deck is exactly the kind of thing that should surface as a suggestion — and here it's not a graph query result, it's the literal grounding file the deck was built from. Neither stale document is listed anywhere in `plan.md`'s Project Structure as needing a fix.

## 6. Smaller, but real

- **Worktree naming collision (R5 / FR-015, tag: risk+simplicity):** `speckit-implement-parallel/SKILL.md`'s existing Guardrails section *already* tells the orchestrator to "consider giving each story its own git worktree (`Agent` `isolation: "worktree"`)" for wide waves — a pre-existing, harness-level, per-*story* mechanism, wholly unrelated to git-ext's proposed per-*wave*, shell-scripted Phase-4 spike. Both are "worktree," both live in the same file/phase. R5's "firewalled by construction" claim doesn't address this scope/naming overlap; the spike's write-up should explicitly distinguish the two so `implement.log.md`'s recorded outcome isn't later misread as covering (or being confounded by) the pre-existing mechanism.
- **FR-002's regex has no timestamp-mode carve-out (tag: correctness):** `speckit-specify`'s own Pre-Execution logic supports `feature_numbering: "timestamp"` (`YYYYMMDD-HHMMSS-slug`). FR-002's branch-name regex (`^[0-9]{3}-[a-z0-9]+...`) does not match that shape. This repo runs `sequential` (`init-options.json` confirms) so it's dormant here, but the extension is explicitly packaged for reuse ("exactly like `extensions/graphify/`") — the gap will bite on first install into a timestamp-numbered repo.
- **Partial-uninstall leaves dangling hook references (tag: risk):** the pattern R4 says git-ext's `uninstall.sh` mirrors (`extensions/graphify/uninstall.sh`) removes the skill directories **unconditionally** before attempting YAML hook-deregistration, and that deregistration step degrades to a non-fatal "warn, fix by hand" when no PyYAML-capable Python is found — leaving `extensions.yml` pointing at commands whose backing skill is already gone. That's a different, unaddressed failure shape from R4's "malformed edit" framing, and would be worse than a warning if git-ext ever registers an `optional: false` hook (a dangling mandatory hook reference would hard-block every phase command, not just this extension's).
- **No lock on the `extensions.yml` merge (tag: risk, low):** safety against two installers racing on the same file currently depends entirely on "the orchestrator promises to serialize" (graphify-context.md's own note). A cheap `flock`/lockfile in the merge step would be defense-in-depth independent of that discipline.

## Suggestions

- [risk] FR-005 and FR-008's call sites live inside `extensions/graphify/skills/speckit-implement-parallel/SKILL.md` and `extensions/council/skills/speckit-council-approve/SKILL.md` respectively — files owned by other extensions whose installers unconditionally overwrite their skills directory on reinstall. Either commit to editing those extensions' source trees explicitly (and document the cross-extension coupling in the risk register) or find a mechanism that survives council/graphify reinstalls; the current plan silently assumes neither problem exists. (confidence: high)
- [correctness] Plan.md's "Dependency / graph impact" claim that "the only existing file touched is `.specify/extensions.yml`... No source file is mutated" is false given the plan's own §D/§F integration points; fix the claim or fix the design so it's true. (confidence: high)
- [risk] R3 only analyzes same-feature re-entrancy. Two concurrent `/speckit-specify` runs can race `create-new-feature.sh`'s directory-scan NNN allocation to the same number; branch.sh's create-if-absent-then-checkout mitigation then makes the second racer silently switch onto the first racer's branch, merging two unrelated specs with no error. Add a genuine collision guard (e.g., `git checkout -b` failing loudly when the branch exists but the spec ID doesn't match the current session's, or an atomic directory-creation retry) distinct from the same-feature idempotency path. (confidence: high)
- [risk] R1's "coarser hook backstops a missed per-wave call, no work is lost" claim doesn't hold for interruption occurring inside an in-progress `implement` run, since `after_implement` by construction hasn't fired yet. Because `tasks.md`'s `[X]` marking (per task) precedes the proposed per-wave commit call, an interruption in that window leaves `tasks.md` and git state disagreeing about wave completion, with no mechanical way for a resuming session to tell "wave done, uncommitted" from "wave partial, discard." Consider committing before marking `[X]`, or making the per-wave commit atomic with the marker write. (confidence: med)
- [risk] R2's Med/Med "narrow window" rating undersells the exposure on `implement` specifically: `before_implement` checks freshness once at the start of a run that the plan itself describes as the longest, most multi-wave, most interruption-prone phase, so a mid-run edit to `tasks.md`/`assignment.md` rides unchecked for every remaining wave. Consider re-verifying gate freshness at each wave boundary for this phase only, given the cost asymmetry (a stale multi-hour parallel run vs. one cheap SHA check per wave). (confidence: med)
- [risk] `.claude/skills/speckit-specify/SKILL.md` currently states as fact that "branch creation is handled by the `before_specify` hook," directly contradicting the plan's ratified `after_specify` design (D51); this feature's own `graphify-context.md` repeats the same stale `before_specify` claim in its "Patterns to follow" section, even though `docs/contracts/artifact-layout.md` §2 was correctly amended. Add "correct the stale `before_specify` references in `speckit-specify`'s SKILL.md and regenerate `graphify-context.md`" to this feature's scope, or a future implementer will find two authoritative-looking documents pointing at the rejected design. (confidence: high)
- [simplicity] The Phase-4 worktree spike (FR-015) and the pre-existing, harness-level "give each story its own `Agent` `isolation: worktree`" guardrail already documented in `speckit-implement-parallel/SKILL.md`'s Guardrails section are two different mechanisms sharing one name in one file. Explicitly distinguish them in the spike's write-up so `implement.log.md`'s recorded outcome isn't later misattributed to, or confused with, the pre-existing mechanism. (confidence: med)
- [correctness] FR-002's branch-name regex (`^[0-9]{3}-[a-z0-9]+(-[a-z0-9]+)*$`) has no carve-out for the already-supported `feature_numbering: "timestamp"` mode (`YYYYMMDD-HHMMSS-slug`, live in `speckit-specify`'s own logic). Dormant in this repo (`sequential` mode) but will break on first install into a timestamp-numbered repo, undermining the "install like graphify anywhere" packaging goal. (confidence: low)
- [risk] The uninstall pattern R4 says git-ext mirrors (`extensions/graphify/uninstall.sh`) removes skill directories unconditionally before attempting YAML hook-deregistration, and silently warns-and-continues if no PyYAML-capable Python is found — leaving `extensions.yml` referencing now-deleted commands. Order uninstall so hook de-registration happens (or is verified) before payload removal, or fail hard instead of warning when the YAML step can't run. (confidence: low)
- [risk] The `extensions.yml` merge has no file lock; safety against two installers running concurrently depends entirely on orchestrator discipline (per `graphify-context.md`'s own serialization note). A `flock`/lockfile in the merge step would add defense-in-depth cheaply. (confidence: low)
