I checked this plan's ordering claims against two things: the actual, already-shipped hook-processing procedure embedded verbatim in every `.claude/skills/speckit-*/SKILL.md` file (not the deck's paraphrase of it), and the actual bodies of the two skills the plan's own Phase-1 contract admits it must edit (`speckit-council-approve`, `speckit-implement-parallel`). `graphify-out/graph.json` exists (19,556 lines) but is low-signal for this specific feature: `graphify explain` on `.specify/extensions.yml`, `speckit-council-approve`, and `speckit-implement-parallel` each return only a single degree-1 generic "contains" edge from a docs-summary node — consistent with the deck's own admission (§3) that "`extensions/git/` does not exist in the graph." For a coupling this deep in markdown-encoded skill *procedure* rather than source-code imports, direct file citation turned out to be the stronger signal, so that's what the findings below rest on.

**The hard-block hooks are specced `optional: true`, which the actual mechanism never runs automatically.** `contracts/commands.md`'s hook table header states hooks are "registered in `.specify/extensions.yml`, `optional: true`" — this covers every hook in the table, including `before_tasks`/`before_implement`'s `verify-gate.sh`, which the same table marks "non-zero ⇒ hard-block" (FR-009). `plan.md` §F repeats this: "Hooks are `optional: true, priority: N` (graphify's schema)." But I read the literal Pre-Execution/Post-Execution hook-processing prose baked into `speckit-tasks/SKILL.md` (lines 33–55) and `speckit-specify/SKILL.md` (lines 251+, identical pattern repeated verbatim across every stock skill) — for a hook with `optional: true`, the skill only *announces* it ("Prompt: {prompt} / To execute: `/{command}`") and proceeds without running it; only `optional: false` hooks are actually invoked and waited on ("you MUST actually invoke the hook and wait for it to finish before continuing"). Under `optional: true`, a stale council/workforce gate would surface as a skippable suggestion, not a block — precisely the "warn-and-override" pattern `spec.md`'s own 2026-07-09 clarification rejected as "defeating the binding." The same gap threatens SC-001's "zero manual git": if `after_specify`/`after_*` commit hooks stay `optional: true`, the pipeline's default behavior is unchanged from M0/M1 unless the engineer manually runs each suggested hook command every time.

**The "priority" ordering the deck/plan lean on is inert in this codebase.** `contracts/commands.md` states "Hooks compose with graphify's `before_*` by `priority`; the git commit hook runs after graphify's context hook at a shared boundary (FR-013)." I grepped `priority` across every stock skill's hook-processing section, `.specify/scripts/bash/common.sh`, and `extensions/graphify/install.sh`: nothing reads `priority` to order hook execution. It is written as a static `5` by graphify's own installer (`extensions/graphify/install.sh` lines ~103/151/160/169) and never consumed — the only other "priority" concept in the repo is the unrelated template-preset resolution stack in `common.sh`. The stock skills process a hook key's YAML list in as-written order, with no sort step. Two consequences: (a) the cited scenario is also slightly mislabeled — there is no shared key where a git-ext *commit* hook meets a graphify hook (commit hooks are all `after_*`, graphify's are all `before_*`); the real shared keys are `before_tasks`/`before_implement`, where git-ext's *verify-gate* meets graphify's context-gen. (b) Since the installer merges "append-only, never overwrite" (deck §2/§3) and graphify's entries are already first, git-ext's verify-gate entries land *after* graphify's context-gen at both shared keys — so a stale/blocking gate is discovered only after graphify's context-regeneration session has already run, not before. Not a correctness break, but a real, currently-unbudgeted cost/UX regression the plan's own "priority" citation doesn't actually prevent.

**The plan's own Phase-1 contract requires editing two already-shipped M1 skills, and that scope is missing from `plan.md`'s Project Structure and blast-radius sections.** `contracts/commands.md`'s "Gate-command integration points (the R1 seam)" is explicit: "`/speckit-council-approve` records `plan.md @ $(speckit.git.sha plan.md)`… `/speckit-implement-parallel` calls `speckit.git.commit impl "wave K/N …"` after each wave." I read both files in full. `speckit-council-approve/SKILL.md`'s "Record the Decision" step 3 composes the `## Human Gate` table with exactly `reviewer | decision | reviewed` — no SHA field exists yet, so this is a template edit, not just an added call. `speckit-implement-parallel/SKILL.md`'s "Execute waves (the loop)" has five steps; step 5 only logs one line to `implement.log.md` — there is **no commit step anywhere in this file**. The deck (§2, §5) and `research.md` (D-R7) both describe this as "`implement-parallel`... already commits per wave by hand — the M1 trail," which is true of session practice (the actual git log shows `impl(001) wave 8/8: …`) but not of the skill's own procedure — the wave-commit step must be newly authored, not merely rewired. Despite this, `plan.md`'s "Project Structure" (lines 97–125) lists only new files under `extensions/git/`, and its "Dependency / graph impact" section states "The only existing file touched is `.specify/extensions.yml`… No source file is mutated" — directly at odds with the Phase-1 contract it itself produced. Since `tasks.md` doesn't exist yet for `002`, and task generation keys off the plan's structure/file manifest, there's a real risk these two edits (which implement R1 — the deck's own top-flagged risk) get silently dropped from `tasks.md`, or get marked `[P]`-parallel with unrelated new-file work without the collision check their shared/mutable status deserves.

**`graphify-context.md` is already stale relative to the plan it's meant to ground, and nothing regenerates it.** File mtimes confirm the ordering: `graphify-context.md` (16:54) predates `contracts/` (17:00, containing `commands.md`) by six minutes — it was written by the `before_plan` hook before Phase 1 decided the R1 skill edits existed. Its shared/mutable-files list (per deck §3) names only `.specify/extensions.yml` and `.specify/feature.json`; it doesn't and couldn't yet know about `speckit-council-approve/SKILL.md` / `speckit-implement-parallel/SKILL.md`. If `/speckit-tasks-graph` or `/speckit-implement-parallel` trust this file for collision/parallelism decisions without a refresh, they inherit the gap above.

**The `before_specify` rejected-alternative reads as a strawman against a slot that already exists in this repo.** The deck (§2) and `plan.md` reject `before_specify` branch creation because it would fire "before the spec ID exists, forcing the hook to re-derive the NNN+slug logic itself and risking a mismatch with the directory." But `speckit-specify/SKILL.md` step 2 ("Branch creation (optional, via hook)") already documents a live `before_specify` hook contract in this exact repo: "If a `before_specify` hook ran successfully… it will have created/switched to a git branch and output… `BRANCH_NAME` and `FEATURE_NUM`. Note these values for reference, but the branch name does **not** dictate the spec directory name." That contract is explicitly advisory — it already tolerates the divergence the rejection cites as disqualifying. This doesn't undercut `after_specify` as the right choice (reading the final ID beats guessing it either way), but the plan never engages with the fact that a purpose-built, already-wired `before_specify` slot exists and will now sit permanently unused — worth one explicit line so a future reader who finds that dormant slot doesn't wonder if it was missed.

**Minor citation-accuracy issue:** deck §3 attributes spec-ID assignment and `feature.json` writing to `.specify/scripts/bash/create-new-feature.sh` ("computes the NNN+slug and writes `feature.json`"). I grepped `speckit-specify/SKILL.md` for any invocation of that script: zero hits. The Claude-Code-native skill performs the equivalent numbering/`mkdir`/`feature.json`-write **inline**, as markdown-prose instructions (Outline step 3), independent of the shell script. End state is equivalent (feature.json is populated before any hook fires either way), so FR-001/002 aren't broken by this, but the deck grounds itself against a script that isn't actually in the invocation path, which risks drift if someone "fixes" `create-new-feature.sh` expecting it to govern real behavior.

## Suggestions

- [correctness] `verify-gate.sh`'s `before_tasks`/`before_implement` hooks must not be registered `optional: true` — per the actual hook-processing mechanism (confirmed in `speckit-tasks/SKILL.md` and `speckit-specify/SKILL.md`), an optional hook is only announced, never auto-invoked, which silently downgrades FR-009's mandatory hard-block into the warn-and-override pattern the spec explicitly rejected. Needs `optional: false` (or an equivalent non-skippable path) for the two gate-verify hooks at minimum. (confidence: high)
- [sequencing] Pin explicit, tested priority/ordering behavior for the two genuinely shared hook keys (`before_tasks`, `before_implement`), where git-ext's verify-gate meets graphify's context-gen hook. The `priority` field the plan cites (FR-013, `contracts/commands.md`) is not read anywhere in the current hook-processing code — actual order is YAML list/append order — so an append-only merge puts verify-gate *after* graphify's context regeneration, discovering a stale gate only after paying for context-gen. Either make the installer insert git-ext's verify-gate entries ahead of graphify's at those two keys, or implement real priority-sorting if that was the intended mechanism. (confidence: high)
- [sequencing] `plan.md`'s "Project Structure" and "Dependency / graph impact" sections omit the two existing-skill edits that `contracts/commands.md`'s own "Gate-command integration points (the R1 seam)" requires (`speckit-council-approve/SKILL.md` gains a SHA-recording field/call; `speckit-implement-parallel/SKILL.md` gains a wave-commit step that doesn't exist today — confirmed by reading its full "Execute waves" loop). Fix the plan's structure/blast-radius sections to list both files as touched + shared/mutable before `/speckit-tasks` runs, or the R1 seam (already flagged as the design's top risk) is likely to ship unwired with no task catching it. (confidence: high)
- [sequencing] Regenerate `graphify-context.md` (or hand-amend its shared/mutable-files list) before `/speckit-tasks-graph`/`/speckit-implement-parallel` run for this feature — it was written by the `before_plan` hook six minutes before `contracts/commands.md` (confirmed by file mtimes: 16:54 vs 17:00) and so predates, and doesn't reflect, the two skill-file edits above. (confidence: med)
- [sequencing] The deck/plan's rejection of `before_specify` branch creation argues against a generic strawman rather than the concrete `before_specify` hook contract that already exists (and sits unused) in `speckit-specify/SKILL.md` step 2, whose own design already tolerates the exact "branch name might not match the eventual directory" risk cited as the rejection reason. `after_specify` still looks like the better choice on its own merits, but add one line acknowledging the dormant slot is deliberately unused, not missed. (confidence: med)
- [correctness] Deck §3 cites `.specify/scripts/bash/create-new-feature.sh` as the mechanism that assigns the spec ID / writes `feature.json` and that the `after_specify` hook must not duplicate. In this repo's actual invocation path, `speckit-specify/SKILL.md` never calls that script — it performs equivalent numbering/`feature.json`-write inline via markdown-prose instructions (Outline step 3). Low practical impact (end state is the same either way) but the grounding citation is wrong and should point at the skill's own Outline step 3, not the shell script, to avoid future drift between two independent numbering implementations. (confidence: med)
- [risk] `speckit.git.verify-gate`'s failure mode when it can't parse the `<artifact> @ <sha>` line out of `decision-record.md`'s `## Human Gate` / `assignment.md`'s `## Workforce Gate` section is unspecified in `contracts/commands.md` — no stated fail-open vs fail-closed behavior, and no format-version guard if a future edit to the gate-writing skills changes the table shape `verify-gate.sh` depends on. Given `verify-gate` is meant to hard-block, an unparseable-but-present section should almost certainly fail closed (treat as stale) rather than silently pass, and that should be stated as a requirement, not left to whoever writes `verify-gate.sh`. (confidence: med)
- [sequencing] Minor bootstrap self-reference, worth one line for consistency with the M1 ff-merge grandfather clause: `002`'s own council-gate approval will be recorded by `speckit-council-approve` *before* this feature's implementation adds the SHA-recording call to that same skill, so `002`'s own `## Human Gate` section will have no `plan.md @ <sha>` binding. Not a defect, but undocumented — the plan grandfathers M1's ff-merge explicitly and should do the same here for its own gate record. (confidence: low)
