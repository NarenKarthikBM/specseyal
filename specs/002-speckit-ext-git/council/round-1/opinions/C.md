## Overall verdict

The core spine — "the branch and commit graph *are* the state, no state file, ever" (plan.md, D32) — is genuinely the simple choice, and I'd defend it: it correctly rejects a sidecar state file, and FR-004's "no forced empty commit" rule avoids commit-noise. But the surface built on top of that spine is bigger than the deck's cost accounting admits. Counting from the plan's own Project Structure and contracts/commands.md: 7 hook registrations (5 brand-new keys — `after_specify/clarify/plan/tasks/implement` — plus append-only entries into 2 existing keys, `before_tasks`/`before_implement`), 3 primitives, 1 human command, 5 shell scripts, plus differentiated re-approval routing by gate type (D-R5). That's not disqualifying on its own — some of it is well-justified (see below) — but two specific pieces read as more elaborate than a solo/linear v1 needs, and one piece is undercosted outright.

**The installer is the biggest miss.** Deck §3 calls `install.sh`'s pattern "cosmetic (copy the shape)" and plan.md's Project Structure describes it in one line: "idempotent; graphify variant (merges hooks into extensions.yml)." I read both `extensions/council/install.sh` (55 lines, no hook-merge) and `extensions/graphify/install.sh` (193 lines) directly, since `graphify explain "extensions/graphify/install.sh"` and `graphify explain "extensions/council/install.sh"` both return "No node matching" — the AST extraction doesn't cover `.sh` files, so the deck's "grounded in graphify-context.md" framing for the installer pattern is actually unverified-by-graph prose, not a graph fact. What I found by reading it directly: graphify's hook-merge is an embedded Python heredoc invoked through a 5-way interpreter fallback chain (graphify's shebang python → specify's shebang python → `python3` → `python` → `uv run --with pyyaml`), and it has a documented failure mode — `extensions/graphify/docs/troubleshooting.md` has a standing entry for "Installer printed a 'Merge this into .specify/extensions.yml by hand' block" and another for "Comments disappeared from `extensions.yml`" (PyYAML doesn't round-trip comments). That's the complexity cost of merging **3 uniform** hook entries (same extension, same command, same shape) into 3 keys. Git's installer has to merge **7 entries of at least 3 behaviorally distinct shapes** (branch+commit chain, plain commit, gate hard-block) into 5 new keys plus 2 append targets — a harder merge, a harder manual-paste fallback for a human to get right by hand, and it isn't mentioned anywhere in the Cost/Complexity Estimate's "~5 short POSIX sh scripts" framing. This is very likely to be the largest, most failure-prone file in the deliverable, and it's currently invisible in the plan's own accounting.

**Grounding gap, narrower point:** deck §3 also cites `.specify/extensions.yml` as "depended on by *every* `/speckit-*` phase command" as a graphify-context.md-grounded blast-radius claim. `graphify explain ".specify/extensions.yml"` returns exactly one node — a README heading mention, degree 1 — not an edge set showing "every phase command reads it." The claim is plausible and I don't dispute it, but the graph doesn't substantiate it; extraction appears scoped to markdown, not `.sh`/`.yml`, so two of the deck's graph-grounded claims about this exact file/pattern are better described as engineer assertion dressed in graph language.

**Gate-SHA enforcement may be gold-plating for v1.** FR-008/009/010 plus R2 plus D-R3/D-R5's differentiated re-approval routing (council → D14 reopen tiers, workforce → re-run) is one of the deck's three named "hard parts" (§5) and 3 of 15 FRs. For a solo/linear pipeline, the trigger scenario — human approves `plan.md`, then edits it by hand outside the normal triage path, then runs `tasks` expecting the stale approval to still count — is a narrow, self-inflicted edge case, and R2 already concedes the hard-block only fires at gate *entry*, not continuously, so even this machinery ships with an acknowledged hole. US3's own framing ("provably about a specific, unchanged artifact") would be satisfied by recording the SHA and surfacing a mismatch without wiring an enforced hard-block + differentiated re-approval tree. That's a real simplification available for v1 if this council thinks the enforcement half is premature.

**D-R3's "supply vs. record" split isn't costed against the risk it creates.** Research.md D-R3 rejects "the git ext writes the SHA into the gate section itself" purely on Constitution principle I grounds (no second writer). But that rejection is what manufactures Risk R1 (med likelihood, "the design's load-bearing seam" per both deck §4 and plan.md's risk register) in the first place: every gate-writing command (`council-approve`, the M3 workforce-gate writer) now has to remember to call `speckit.git.sha` and format `<artifact> @ <sha>` correctly. Note R1 actually bundles two independent couplings with different justifications: the per-wave commit call from `implement-parallel` is earned — FR-005's own rejected-alternative reasoning shows phase-boundary-only commits would force re-running a whole multi-wave implementation (a real P1-story cost). The gate-SHA-record call is a discretionary purity choice, not a forced one; a narrower reading of principle I (an explicit exception for a purely mechanical, non-interpretive derived value like a SHA) could avoid manufacturing that half of R1 at all. Bundling both into one risk-register row obscures that they don't have the same cost-benefit case.

**`--no-ff`-always is closed off at the config layer for a consumer that doesn't exist yet.** data-model.md states `merge.policy` is "deliberately not engineered to accept `ff`." The stated reason is M5 needing a completion anchor — but `platform/` is empty until M5 (CLAUDE.md). The rejected-alternatives table (deck §2, plan.md) never considers a lighter anchor, e.g. a `complete/<spec-id>` tag at the completion commit, which would serve M5 equally well without forcing merge topology and would be trivially revisable if M5's actual needs turn out to differ. To be fair, ff-vs-no-ff was already ratified at spec review (D51), so this is really a challenge to D51 surfacing through the plan rather than a plan-level invention — the plan reproduces the position uncritically rather than re-examining it, which is a smaller complaint than if the plan had invented the rigidity itself.

**Two spots where "mechanical git" hides a real ambiguity, not just complexity:** (1) contracts/commands.md describes `commit.sh` as staging "the feature's changed paths" — but wave commits necessarily stage changes across the whole repo (M1's own trail, e.g. `impl(001) wave 1/8` touching `extensions/council/install.sh`, confirms this), while phase-boundary commits presumably should stay scoped to `specs/NNN/`. Neither research.md, data-model.md, nor quickstart.md disambiguates "stage everything currently dirty" from "stage only this feature's known paths," and these have very different safety properties — the former can sweep an unrelated stray edit into a phase-tagged commit, undermining the SC-003 claim that each commit's diff is trustworthy. (2) plan.md §F states "the git commit hook runs after graphify's context hook at a shared boundary" as the composition story for FR-013 — but per the plan's own hook table (§B) and contracts/commands.md, every git *commit* hook is registered at an `after_*` point and graphify only registers `before_*` hooks; the only two keys where a git hook and a graphify hook actually share a boundary are `before_tasks` and `before_implement`, where git's hook is `verify-gate` (a hard-block check), never a commit. The composition sentence describes a scenario that can't occur given the design's own table, and correspondingly no priority value is ever assigned to resolve the real question: should a cheap stale-gate check run before or after graphify's heavier context-generation at those two shared boundaries? Right now a stale gate could let graphify do a full context-generation pass before the hard-block fires.

**Rejected alternatives, overall:** five of seven are fairly argued and not strawmen (`before_specify`, state file, squash/rebase, automatic cleanup, editing stock skills all have real, specific costs named). The `ff`-when-linear entry is fair but inherited from D51 rather than examined here. The "git ext writes the SHA itself" entry is the one that reads thin — it's rejected on principle-I grounds without acknowledging that the alternative it chooses instead is the direct cause of the plan's own top-flagged risk.

## Suggestions

- [simplicity] The installer's hook-merge complexity is significantly undercosted: graphify's own install.sh needed a 193-line embedded-Python YAML merge with a 5-way interpreter fallback and a documented degrade-to-manual-paste failure mode (see `extensions/graphify/docs/troubleshooting.md`) just to merge 3 uniform hook entries; git's installer must merge 7 entries across 3 distinct shapes into 5 new keys + 2 append targets. Cost/Complexity Estimate should name this explicitly rather than filing it under "~5 short POSIX sh scripts," and Phase 1 should budget it as its own task, likely the largest single file in the build. (confidence: high)
- [correctness] Deck §3's two graph-grounded claims about the installer pattern and extensions.yml's "every phase command" blast radius aren't actually backed by graph edges — `graphify explain` returns "No node matching" for both install.sh paths and only a single degree-1 README-mention node for `.specify/extensions.yml`, consistent with the AST extraction not covering `.sh`/`.yml` files. The claims may well be true, but they should be labeled as engineer assertion, not graph-grounded, or re-derived by reading the files directly (as this review did). (confidence: med)
- [simplicity] Consider shipping gate-SHA binding as record-and-disclose only in v1 (satisfies US3's "provably about a specific artifact" framing) and deferring the enforced hard-block + differentiated re-approval routing (FR-009/FR-010, D-R5) to a later version, since the trigger scenario (manual post-approval edit outside the normal triage path) is narrow for a solo/linear pipeline and R2 already concedes the enforcement has an accepted gap at gate entry. This would remove one of the deck's three named "hard parts" entirely. (confidence: med)
- [simplicity] D-R3's rejection of "the git ext writes the SHA itself" is argued solely on Constitution principle I grounds but never acknowledges that this choice is what manufactures Risk R1's gate-recording half (the per-wave-commit half of R1 is separately and adequately justified by FR-005's own rejected-alternative reasoning — the two shouldn't be treated as one risk with one cost-benefit story). Either name this trade-off explicitly in the deck/plan, or consider a narrow, explicit exception to principle I for purely mechanical, non-interpretive derived values (a commit SHA) to avoid manufacturing that half of R1 at all. (confidence: med)
- [correctness] `commit.sh`'s "stages the feature's changed paths" (contracts/commands.md) is ambiguous between "everything currently git-dirty" and "only this feature's known artifact paths" — wave commits clearly need the former (M1's own trail touches source files outside `specs/`), but phase-boundary commits should probably be the latter to avoid sweeping an unrelated stray edit into a phase-tagged commit and undermining SC-003's "exactly one commit per boundary, trustworthy diff" claim. Resolve explicitly in Phase 1, not left implicit in a one-line contract description. (confidence: med-high)
- [sequencing] plan.md §F's claim that "the git commit hook runs after graphify's context hook at a shared boundary" doesn't match the plan's own hook table: every git commit hook is `after_*`, graphify is exclusively `before_*`, so the only real shared boundaries (`before_tasks`, `before_implement`) host git's `verify-gate` hard-block hook, not a commit hook — and no priority value is assigned there in any Phase-1 artifact. Fix the composition story to address the actual case (does a cheap stale-gate hard-block run before or after graphify's heavier context-generation pass at those two boundaries?) and assign explicit priorities so a stale gate fails fast instead of possibly paying for a full context-generation pass first. (confidence: med)
- [simplicity] `merge.policy` is hard-locked to `no-ff` in `git-config.yml` to serve M5's not-yet-built manager as a binding anchor. A lightweight tag at the completion commit (e.g. `complete/<spec-id>`) was not evaluated in the rejected-alternatives table and would serve the same anchor purpose without forcing merge topology now, while staying cheaply revisable once M5's actual requirements are known. This mainly challenges D51 (already ratified at spec review) rather than a plan-level choice, so treat as lower-priority than the others. (confidence: low)
