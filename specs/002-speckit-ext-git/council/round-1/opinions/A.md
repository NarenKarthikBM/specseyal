Grounding note before the review: `graphify-out/graph.json` exists (1013 nodes / 917 edges), so this is not a reduced-grounding review in the formal FR-019 sense — but for nearly every claim that mattered here, the graph itself turned out to be useless. `graphify explain ".specify/extensions.yml"`, `graphify explain "speckit-specify"`, `graphify explain "extensions/council/install.sh"`, and `graphify path "extensions/council/install.sh" "extensions/graphify/install.sh"` all return either "No node matching" or "No path found" — the extraction is heading-level markdown + shell-function `defines` edges only, with no edge type for "SKILL.md X reads config key Y" or "script A execs script B." So the deck's §3 framing ("Grounded in `graphify-context.md`") is true only for the two facts a heading-scan can see (the file exists, the shared file is `extensions.yml`); every substantive finding below came from reading the actual `.claude/skills/*/SKILL.md` and `docs/contracts/*.md` files directly, not from a graph query. That gap is itself worth naming: this plan's central claims live in *prose contracts* (skill markdown, hook dispatch instructions) that graphify's current extractor doesn't model at all, so "grounded in the graph" is doing less work here than the deck implies.

## Finding 1 — the after_specify/before_specify choice contradicts the feature's own grounding artifact and the already-installed skill

Plan §C and the deck §2 reject `before_specify` with one argument: it "fires before the spec ID exists, forcing the hook to re-derive the NNN+slug logic itself and risking a mismatch with the directory the specify skill computes." Two facts undercut this, and neither is acknowledged anywhere in `research.md` (D-R1), `plan.md`, or the deck:

1. **`specs/002-speckit-ext-git/graphify-context.md` line 13** — the plan's own Phase-0 context input (`plan | context in: spec.md, graphify-context.md` per `artifact-layout.md` §2) — says the opposite of what the plan chose: *"002 needs this variant (**it registers `before_specify` + commit hooks**), where council's installer did not."* Nothing in `plan.md`, `research.md`, or the deck's Rejected Alternatives table (which cites the deck-prep session's own sources as `plan.md`, `spec.md`, `graphify-context.md`) mentions this reversal. Either graphify-context.md was wrong and nobody said so, or the plan silently overrode its own cited grounding.
2. **`.claude/skills/speckit-specify/SKILL.md`** (already installed, unmodified by this feature) already has real, working `before_specify` hook-dispatch, with a documented output contract: step 2 reads *"If a `before_specify` hook ran successfully… it will have created/switched to a git branch and output JSON containing `BRANCH_NAME` and `FEATURE_NUM`"*, and the file's closing note reads *"Branch creation is handled by the `before_specify` hook (git extension)."* This is not a hypothetical stock-spec-kit model the plan can wave away — it is live infrastructure in this repo, sized exactly for a git extension, that the plan never examines. (`common.sh` even carries a forward-looking comment — *"e.g. the git extension's create-new-feature-branch"* — anticipating this.) Whether the plan's "slug mismatch" worry is real depends on whether `before_specify`'s branch-naming can be made to agree with the directory the skill computes in Step 3; the plan asserts the risk without checking whether the already-built hook contract already handles it (the skill explicitly tolerates a branch/dir name difference: *"The spec directory name and the git branch name are independent"* — which is itself in tension with FR-002's *"MUST equal the spec ID exactly"*, a second discrepancy the plan doesn't surface).

The `after_specify` choice the plan makes instead has its own undisclosed cost: per `speckit-specify/SKILL.md`, `spec.md` is drafted, checklist-validated (up to 3 iterations), and (if `[NEEDS CLARIFICATION]` markers remain) taken through a multi-turn user Q&A — all of Steps 1–7 — *before* "Mandatory Post-Execution Hooks" ever runs. For that entire window the new `specs/NNN-slug/` tree sits **uncommitted on the base branch's working tree**, no feature branch yet exists, and nothing in FR-012 or risk R3 describes what happens if the session is interrupted in that window (R3 only covers "branch already exists" re-entrancy, never "branch does not exist yet but spec content does"). Worse: `after_specify` is also the *only* hook in the whole design (per `contracts/commands.md`'s hook table) that runs `branch.sh`; every other `after_*` hook just calls `commit.sh <phase> "…"` with no branch-existence check. R1's mitigation — "the coarser phase hook… still backstops it, no lost work" — does not apply here: there is no hook that backstops a missed `after_specify`. If it fails to fire for any reason (the post-execution section is phrased as a prompt-level "You MUST," not a code-enforced gate — see Finding 2), every later phase commit for that feature silently lands on the **base branch**, which is exactly the SC-002/FR-002 failure the whole design exists to prevent, with no detection anywhere in the plan.

## Finding 2 — the "hard-block" in FR-009 has no enforcement mechanism in the hooks as currently specified

FR-009 requires the extension to **hard-block** `tasks`/`implement` on a stale gate; `contracts/commands.md`'s hook table states `before_tasks`/`before_implement` "non-zero ⇒ hard-block" as if this were self-evidently enforced. I checked how hook results are actually consumed today by grepping the "Pre-Execution Checks" block across all 16 `.claude/skills/*/SKILL.md` files (`speckit-specify`, `-clarify`, `-plan`, `-tasks`, `-implement`, `-analyze`, etc. all share near-identical text). Every one of them defines exactly three cases: hook not registered (skip silently), hook `optional: true` (present a prompt, don't auto-run), and hook `optional: false` (*"you MUST actually invoke the hook and wait for it to finish before continuing"*). **None of them defines a fourth case: what to do when the invoked hook signals failure.** The instruction is "wait for the result… before proceeding to the Outline" — not "if the result is a failure, stop and do not proceed." I confirmed this isn't just an omission I'm misreading by grepping all 16 skill files for abort/STOP/"do not proceed" language: every hit is a precondition local to that skill's own Outline (missing `plan.md`, incomplete checklists, JSON-parse failure) — none of them is wired to the generic hook-dispatch block, and none of the stock skills has ever needed hook-failure semantics before, because graphify's hooks (the only hooks that exist today) are informational context-generators, never blockers.

So as written, `verify-gate.sh council` exiting non-zero produces a message the agent running `/speckit-tasks` *sees*, but nothing in the shared machinery instructs it to stop. Whether the hard-block actually blocks anything depends on unenforced, undocumented agent judgment at the moment — a much weaker guarantee than FR-009's "MUST hard-block" or SC-004's "100% of injected-edit tests" implies. To make this real, the plan needs to add explicit stop-on-failure prose to `speckit-tasks/SKILL.md`'s and `speckit-implement/SKILL.md`'s Pre-Execution Checks (at minimum) — but that is an edit to **existing skill files**, and both `plan.md`'s "Dependency / graph impact" section and the deck's §3 state flatly that *"the only existing file touched is `.specify/extensions.yml`."* That claim is false if hard-blocking is to work mechanically rather than by convention, and the Risk Register doesn't carry this as a risk at all — it's a level below R2 (R2 assumes the entry-time check fires and blocks; this finding is about whether the block ever actually stops anything).

## Finding 3 — the SHA-freshness check is blind to uncommitted working-tree drift, in both directions

`contracts/commands.md` specifies `speckit.git.sha <artifact>` as **"prints the SHA of the HEAD commit that last touched that path… Read-only… No side effect."** That's a purely committed-history read. Two consequences neither `spec.md`'s Edge Cases nor the Risk Register (R1/R2 cover different gaps) address:

- **At record time**: if `council-approve` calls `sha plan.md` while `plan.md` has *uncommitted* edits sitting in the working tree (e.g. triage's revision hasn't been committed yet, or a human hand-tweaked the file moments before approving), the recorded SHA reflects the *prior* commit, not the content actually being approved. The binding is wrong from the moment it's written, and nothing in `research.md` D-R3 or the contracts pins down a "commit before you read the SHA" ordering — `speckit.git.sha` is explicitly specified not to do that itself.
- **At verify time**: symmetrically, if a human edits `plan.md` directly after approval and does *not* commit before running `/speckit-tasks`, `verify-gate.sh`'s SHA compare is blind to that edit — "recorded" and "current" both resolve to the same last-committed SHA, so the hard-block **false-passes** in exactly the plainest version of the threat FR-009 exists to catch (a human editing the approved artifact). `spec.md`'s own Edge Cases entry — *"Approved artifact edited after approval — the stale approval is detected via SHA mismatch"* — implicitly assumes the edit gets committed; `quickstart.md` Scenario 2 bakes that assumption directly into its test script ("Edit `plan.md`; commit (`<S2>`)"), so the uncommitted case is never even exercised, let alone caught.

## Finding 4 — FR-008's "extends an existing field" claim doesn't hold for the section that actually needs it

`data-model.md`'s `GateSHABinding` entity states the council gate binding target is `decision-record.md` `## Human Gate`, and that *"the section already cites artifacts `@ <sha>`."* I read `docs/contracts/decision-record.md` directly (§2 Format, §3 Field values, §5 Sections): the **current `## Human Gate` format has no SHA field at all** — its fields are `reviewer`, `decision`, `reviewed` (a plain filename list, no SHAs), per both the worked example and §5's section table. The `@ <sha>` citations that do exist today live in a *different* section, owned by a *different writer*, timestamped to a *different moment*: `## Round N`'s `**Plan reviewed:** plan.md @ <sha>` (written by `/speckit-council-triage`, capturing the plan's SHA when that round ran) and R4's `plan-delta @ <commit-sha>` (which commit applied an accepted suggestion). Neither is "the SHA of `plan.md` at the moment the human clicked approve," which is what FR-008 actually needs, and which can differ from the Round-N SHA if `plan.md` is touched again between triage and approval (plausible across a multi-day gap, or a same-round late edit). FR-001 got an explicit, named contract amendment (D51 → `artifact-layout.md` §2/§3 → 1.2); FR-008 needs an equivalent, explicit amendment to `## Human Gate`'s format in `decision-record.md`/`artifact-layout.md` §8, and doesn't have one — the plan currently reads as if the field already exists when it doesn't.

## Finding 5 — FR-003's phase-boundary list isn't matched by the plan's actual hook coverage; `analyze` is a live, buildable-today gap

FR-003 requires a phase-tagged commit at "specify, clarify, plan, council/triage, tasks, categorize, analyze, agent-assign, implement, complete." `categorize`/`agent-assign`/`complete` have no skill yet (M3/M4, confirmed — `.claude/skills/` has no `speckit-categorize`, `speckit-agent-assign`, or `speckit-complete`), so omitting them from the plan's hook table is defensible, though the plan should say so explicitly rather than silently drop three of ten FR-003 boundaries. `analyze` is different: `speckit-analyze` **already exists and already has working `before_analyze`/`after_analyze` hooks** (identical mechanism to `clarify`/`plan`/`tasks`, confirmed by grep), is already exercised in this repo's own history (`b4aa716 tasks(001-council-extension): tasks+waves, analyze, categorize`), and per `artifact-layout.md` §2 can revise `tasks.md`. Neither `plan.md`'s Chosen Approach §B table nor `contracts/commands.md`'s hook table mentions `after_analyze` anywhere. This is a same-day, no-new-skill-required hook the plan simply missed, and it directly weakens SC-003's "every completed phase boundary that changed an artifact has exactly one phase-tagged commit."

## Smaller items

- **Validated, not a bug**: I stress-tested the deck's completion-anchor claim (`--no-ff`, `git log --first-parent` enumerates one node per feature) against the Phase-4 worktree spike's own per-wave merges. It holds: from base's perspective the entire feature branch — including any internal merge commits the spike creates — collapses to the second parent of exactly one completion merge, invisible to a first-parent walk. The mechanics check out. (One caution worth a line in `quickstart.md`: if SC-003's wave-commit count is ever verified via `--first-parent` *on the feature branch itself* rather than full history, that specific method would undercount under the spike — the plan's own `git log --oneline main..HEAD | wc -l` in Scenario 3 avoids this by not using `--first-parent`, so it's fine as written, just fragile to a future edit.)
- `commit.sh`'s scope — "stages the feature's changed paths" (`contracts/commands.md`) — is never pinned down precisely. Per-wave `implement` commits need to capture source code changes outside `specs/`, so the scope can't be `specs/<id>/`-only, but nothing says what it *is* instead; too broad risks scooping up unrelated dirty files in a shared working tree, too narrow risks an incomplete wave commit. Low severity today (single-engineer CLI use), but worth pinning down before `commit.sh` is actually written.
