The deck's Testability Claim (§6) opens with "Every FR/SC is designed to be falsifiable without a model call — pure git-state and artifact inspection," and the SC→verification-method table is admirably honest about which SCs "ride" another test rather than getting a dedicated one (SC-002 rides SC-001; SC-006 rides SC-003). That honesty is good practice. But checking the claim against `quickstart.md`, `contracts/commands.md`, and the actual installed hook-dispatch mechanism this plan depends on (not just the deck's description of it) surfaces one load-bearing mechanism gap and several real coverage gaps the deck doesn't surface.

**The headline problem: the hooks that must be mandatory are specified as `optional: true`.** `plan.md` §F says "Hooks are `optional: true`, priority: N (graphify's schema) so they compose," and `contracts/commands.md`'s hook table is headed "Hooks (registered in `.specify/extensions.yml`, `optional: true`)" — applied uniformly to every hook in the table, including `before_tasks`/`before_implement` (whose own Exit-contract column says "non-zero ⇒ hard-block") and `after_specify` (the branch-birth hook SC-001/SC-002 depend on firing every time, unprompted). I checked what `optional: true` actually does in this repo rather than taking the deck's word for it (D10): `extensions/graphify/extension/extension.yml` L21-33 is the literal exemplar being mirrored, and every stock skill's hook-dispatch prose (`.claude/skills/speckit-tasks/SKILL.md` L24-56, identically in `speckit-plan`, `speckit-implement`, `speckit-specify` — all present since the original D45 bootstrap commit `8065cee`, i.e. stock spec-kit behavior, not something an extension adds) draws a hard distinction: a `optional: false` ("Mandatory") hook is auto-invoked — "you MUST actually invoke the hook and wait for it to finish before continuing" — while an `optional: true` hook is merely *announced*, printed with a `Prompt:` field and a "To execute: `/{command}`" line, with no instruction anywhere that the calling session must run it. Graphify's hooks are correctly `optional: true` because skipping them only degrades grounding. Git-ext's are not that: SC-001 requires the branch/commit hooks to fire unprompted every time, and FR-009 requires the gate-verify hooks to be capable of an actual hard block. As specified (`optional: true` copied wholesale from graphify's schema, per §F's own "so they compose" rationale — a composition concern, not a correctness one), the dispatch mechanism this plan relies on would only *announce* these hooks and wait for a session to voluntarily run them — which is a materially different, non-deterministic guarantee than what SC-001/FR-009 claim. I also checked whether some other layer enforces this instead of the LLM-read prose: every skill file defers `condition` evaluation to something it calls "the HookExecutor implementation" (15+ occurrences, repo-wide); a full-repo grep for `HookExecutor` finds it named only in that boilerplate, never implemented anywhere (no class, script, or binary). There is no code-level enforcement layer here at all — hook dispatch, today, is 100% LLM-prose-mediated. That makes the `optional` flag the *only* lever this plan has for turning "advisory" into "guaranteed," and the plan sets it to the advisory value for hooks whose entire reason to exist is to be guaranteed.

**This mechanism gap makes the quickstart scenarios untrustworthy as SC proof, not just incomplete.** If `after_specify`/`before_tasks`/`before_implement` are registered `optional: true`, then Scenario 1's "Expect: a branch named `NNN-slug` exists" and Scenario 2's "Expect: `tasks` is hard-blocked" are not properties of the system — they're properties of whether the particular LLM session driving that quickstart run happened to act on an announced-but-not-mandated prompt. A quickstart pass under those conditions doesn't distinguish "the mechanism guarantees this" from "the assistant was cooperative this run," and a re-run isn't guaranteed to reproduce the result. `quickstart.md`'s Setup step (`grep -A2 'after_specify\|before_tasks' .specify/extensions.yml`) only confirms the hooks are *registered*, never checks their `optional` value — so this gap would sail through the plan's own validation undetected.

**SC-006 (P1, resumability) has zero coverage in the one document that's supposed to make every SC runnable.** `quickstart.md`'s own opening line is "Runnable checks that prove each SC" — but I read the full file (5 scenarios, 37 lines) and there is no interrupt/resume step anywhere. Scenario 1 step 3 only runs an uninterrupted ≥3-wave implementation. `spec.md`'s US2 Independent Test (interrupt after wave 2, confirm resume starts at wave 3, nothing duplicated) is never operationalized into a quickstart step; the deck's own SC table (§6) admits SC-006 "rides" SC-003's mechanism rather than getting a dedicated test, but doesn't note that the *actually executable* document has nothing for it at all. This is the second P1 story (alongside US1) and arguably the harder one to get right — interruption timing, restart-point detection, and duplicate-work avoidance are exactly where resumability bugs hide, and it's the one piece nobody has written a runnable check for.

**SC-007's "zero traces" check is close to vacuous given the current state of the tracing capability.** `spec.md`'s own Constraints & Assumptions (last bullet) and D46 both state that non-council build sessions (specify/clarify/plan/tasks/implement) currently write **no** `traces.jsonl` records at all — tracing is wired only to council sessions in this bootstrap period. `artifact-layout.md` §7 rule 4 nominally requires a trace record for every session-running phase, so this is a known, pre-existing, unresolved tension, not something 002 introduces — but 002's quickstart Scenario 4 (`grep -c '"role":"git' specs/NNN/traces.jsonl` → 0) inherits it directly: for most of a "full run," the file the check inspects has no records at all regardless of what the git extension does, because nothing in that part of the pipeline is writing to it yet. The grep would return 0 whether the extension is flawlessly mechanical or badly broken, because there is no code path today by which a hook script could plausibly acquire a session and append a `role:"git` record in the first place. This isn't quite unfalsifiable in the strict sense (a future implementation that mistakenly routed a git-ext primitive through a Task/Agent dispatch could trip it) but as written it is not a meaningful negative control — it proves the absence of something that had no opportunity to appear.

**Branch birth at `after_specify` (FR-001, and R3's idempotency mitigation) has no isolated test — every proof path requires a full, model-driven pipeline run.** This directly answers the question of whether it's testable in isolation: as specified, no. `research.md` (D-R1), `data-model.md`, and `contracts/commands.md` describe `branch.sh`'s create-if-absent-then-checkout behavior precisely enough to unit-test against a scratch repo with a fabricated `feature.json` (no LLM needed) — but no such harness appears anywhere in the Project Structure (`plan.md`'s `extensions/git/` tree lists five `scripts/*.sh` and no `test/` directory), and the only verification path offered is quickstart Scenario 1, which requires actually running `/speckit-specify` — a real, model-driven session — to reach the hook at all. That conflates two very different failure modes (a bug in `branch.sh`'s idempotency logic vs. a stalled/failed spec-kit session) behind one slow, expensive, non-deterministic check, and it's the opposite of what the deck's own "falsifiable without a model call" framing promises: the *operation* is model-free, but as written, the *test that exercises it* is not.

**SC-008's "removing it leaves Scenarios 1–4 green" is asserted, never exercised.** Scenario 5 checks that `implement.log.md` has a spike entry and that no `extensions/git/**` file references `worktree` outside "the spike task" — but that is a structural-coupling check (does anything reference the spike), not the claim SC-008 and FR-015 actually make (that *removing* the spike leaves the v1 loop unaffected). Nobody deletes the spike code and re-runs Scenarios 1–4 to confirm this. Separately, "the spike task" as an exclusion boundary doesn't correspond to any named file in `plan.md`'s own Project Structure — there's no `worktree.sh` or any spike-owned file listed under `extensions/git/`, only `branch.sh`/`commit.sh`/`sha.sh`/`verify-gate.sh`/`cleanup.sh`. As written, the grep's exclusion clause has no concrete referent to exclude.

**Secondary gaps**, lower severity but real: SC-002 and SC-004 claim "100%" language ("100% of features," "100% of injected-edit tests") that a single manual quickstart run (one throwaway feature; one injected edit) cannot actually establish — there's no fuzzed or repeated test matrix across spec-ID shapes, re-run/interrupted-specify cases, or multiple edit scenarios anywhere in the plan's Testing field or `quickstart.md`. SC-005's reachability check (Scenario 3 step 2, `git log main | grep -c '(NNN' unchanged`) is a commit-message text-match proxy computed from two differently-scoped `git log` invocations (`main..HEAD` before cleanup vs. `main` after), not a SHA-level `git merge-base --is-ancestor` check against the original commit list — a coincidental count match could mask a lost-and-replaced commit. And Scenario 2's injection step ("Edit `plan.md`; commit (`<S2>`)") is the one step in the whole document without a literal, copy-pasteable command, unlike every other line in `quickstart.md` — a minor but real reproducibility gap for the one scenario proving the hard-block actually works.

## Suggestions

- [correctness] `plan.md` §F and `contracts/commands.md`'s hook table register every git-ext hook as `optional: true`, copied from `extensions/graphify/extension/extension.yml`'s schema — but the stock hook-dispatch prose (`.claude/skills/speckit-tasks/SKILL.md` L33-44 et al.) only auto-invokes `optional: false` ("Mandatory") hooks; `optional: true` hooks are merely announced with a prompt. This breaks SC-001's "no manual step" guarantee and FR-009's hard-block for `before_tasks`/`before_implement`, since nothing forces the announced hook to actually run. Set the branch/commit/gate-verify hooks to `optional: false` (or otherwise specify an enforcement path), and update `contracts/commands.md`'s Exit-contract column to match what the chosen flag actually delivers. (confidence: high)
- [testability] No automated verification harness exists or is proposed anywhere in the plan — `quickstart.md` is entirely "Expect: ..." prose for a human to eyeball, `extensions/git/`'s Project Structure has no `test/` directory or `verify.sh`, and the repo-wide conformance checker (I-11) is explicitly still unbuilt (docs/90 I-11: "the checker build... remains open"). For a "mechanical, zero-AI, deterministic" extension — exactly the kind of thing that's cheapest to unit-test — shipping zero scripted assertions is a missed opportunity and leaves every SC's pass/fail as a matter of human judgment call. Add a script (e.g. `extensions/git/test/run.sh`) that runs quickstart's checks and exits non-zero on failure, so quickstart.md becomes CI-able rather than a manual checklist. (confidence: high)
- [testability] SC-006 (P1, resumability — spec.md US2) has no corresponding scenario anywhere in `quickstart.md`, despite the file's own claim to be "runnable checks that prove each SC." Add an explicit interrupt/resume scenario (kill mid-implementation after wave 2, resume, verify restart at wave 3 with no duplicated/lost work) rather than letting it silently ride SC-003's mechanism by assertion only. (confidence: high)
- [testability] SC-007's verification (`quickstart.md` Scenario 4: grep `traces.jsonl` for zero `role:"git` records) is a weak negative control: per `spec.md`'s own Constraints & Assumptions and D46, non-council build-phase sessions currently write **no** trace records at all, so the grep returns 0 regardless of whether the extension behaves correctly — there's no code path today by which it could do otherwise. Either note explicitly that this check is provisional until build-phase tracing is wired more broadly, or add a check that's actually sensitive to a regression (e.g., confirm the git-ext's shell scripts never call anything that could acquire a session/trace_id). (confidence: high)
- [testability] Branch birth (FR-001) and its idempotency guarantee (R3, FR-012) have no isolated/unit-level test — every proof path in the plan requires a full, model-driven `/speckit-specify` run (quickstart Scenario 1), conflating hook-script bugs with spec-kit session behavior. `branch.sh`'s create-if-absent-then-checkout logic is unit-testable in isolation (a scratch repo + a fabricated `feature.json`, no LLM) and should be, both for speed and to keep hook-script correctness distinguishable from spec-kit/LLM flakiness. (confidence: high)
- [testability] SC-008's quickstart check (Scenario 5) verifies the *absence of a reference* to `worktree`, not the claim FR-015/SC-008 actually make (that removing the spike leaves Scenarios 1–4 green) — nobody deletes the spike and re-runs 1–4. Additionally, the exclusion clause "outside the spike task" doesn't name any file that actually exists in `plan.md`'s Project Structure (no `worktree.sh` or spike-owned file is listed), so the instruction isn't concretely executable as written. Either name the spike's actual file(s) once implementation defines them, or replace the grep with a literal remove-and-rerun step. (confidence: med)
- [testability] SC-002 ("100% of features") and SC-004 ("100% of injected-edit tests") are not earned by the described verification method — a single manual run on one throwaway feature (Scenario 1) and one injected edit (Scenario 2). Either soften the SC language to match an existence-proof methodology, or add a small repeated/fuzzed test matrix (varied spec-ID shapes, a re-run/interrupted-specify case, multiple edit-timing scenarios) before claiming 100%. (confidence: med)
- [testability] SC-005's reachability check (`quickstart.md` Scenario 3 step 2) compares commit *counts* from two differently-scoped `git log` invocations (`main..HEAD` pre-cleanup vs. `main` post-cleanup, matched by a message-text grep on `'(NNN'`) rather than checking each original commit SHA's ancestry directly. Prefer capturing the C commit SHAs before cleanup and asserting `git merge-base --is-ancestor <sha> main` for each afterward — a coincidental count match can't mask a swapped or lost commit the way it currently can. (confidence: med)
- [testability] Quickstart Scenario 2's injection step ("Edit `plan.md`; commit (`<S2>`)") is the only step in the document without a literal, copy-pasteable command — every other step gives exact `grep`/`git` invocations. For the one scenario that proves the hard-block (SC-004) actually fires, under-specifying how the stale edit is produced risks non-reproducible results between test runs. (confidence: low)
- [risk] The deck's Testability Claim (§6) states every FR/SC is "falsifiable without a model call," but the hook-dispatch layer that FR-001/FR-009 depend on is entirely interpreted by an LLM reading markdown instructions — every skill file defers `condition` evaluation to an undefined "HookExecutor implementation" that does not exist anywhere in the repo (grepped repo-wide: zero hits beyond the boilerplate that names it). This is a systemic property of the current spec-kit + extensions architecture, not unique to this plan, but it means even a correctly-specified `optional: false` hook (see first suggestion) is enforced by an LLM's willingness to follow "you MUST actually invoke" prose, not by code — worth naming explicitly rather than leaving implicit in "mechanical git" framing. (confidence: low)
