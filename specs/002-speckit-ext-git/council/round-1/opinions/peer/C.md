All four Stage-1 opinions are unusually strong: every one of them cites specific files and line ranges rather than paraphrasing the deck, and every one of them independently rediscovered (via direct file reads, not the graph — `graphify explain`/`path` return no useful edges for this feature, as A and D both note) that the deck's "the only existing file touched is `.specify/extensions.yml`" claim is false. I re-ran the load-bearing verifications myself before writing this — `diff` on both allegedly-identical skill-file pairs, `grep` for `rm -rf`/`cp -R` in both installers, `grep` for `priority` and `HookExecutor` repo-wide, and a direct read of `create-new-feature.sh`'s directory-creation guard — and every one of the specific factual claims below checked out exactly as the citing member stated it. I don't think this round needs a forced 1-4 ranking; three of the four opinions converge on the same central defect from different angles, which is real corroboration, not noise, and the differences between them are more about which remedy is right-sized than about who found the truest fact.

## The R1 cross-extension-coupling finding is the round's center of gravity, and B's version is the sharpest

A (Finding 4), B (Finding 1), and D (third finding) all land on the same underlying fact: FR-008's SHA-recording and FR-005's per-wave commit both require call sites inside `speckit-council-approve/SKILL.md` and `speckit-implement-parallel/SKILL.md` — files this plan's own "Dependency / graph impact" section doesn't list as touched. B's framing is the one with teeth: I confirmed `diff .claude/skills/speckit-council-approve/SKILL.md extensions/council/skills/speckit-council-approve/SKILL.md` and the `implement-parallel` equivalent are both byte-identical, and both `extensions/council/install.sh` and `extensions/graphify/install.sh` do `rm -rf "${SKILLS_DEST:?}/$name"; cp -R "$skill_dir" "$SKILLS_DEST/$name"` unconditionally on every install. That means whatever git-ext's build does to satisfy R1 is silently discarded the next time council or graphify reinstalls — not a theoretical risk, a mechanical certainty given the installers as written. A's Finding 4 (no SHA field exists in `## Human Gate` today) and D's Project-Structure-omission framing are both correct and both real, but they describe the *shape* of the gap; B's is the only one that shows the gap *doesn't stay fixed*. This deserves to be the chairman's top-flagged item.

Underneath this sits a design-level tension worth surfacing that none of the four raised: D32 ("no state file, ever") plus Constitution principle I ("the git ext must not co-write another phase's artifact") together are *why* the SHA has to be grafted into a borrowed artifact in the first place — `speckit.git.sha` supplies, but something has to *record*, and D32 forbids the git ext from owning any storage of its own to record it into. A git-ext-owned artifact (e.g. `specs/NNN/git/gate-bindings.md`, written only by `speckit.git.commit gate "…"` as a side effect of the commit call the gate command must make anyway) would let `verify-gate.sh` read from git-ext's own file instead of parsing `decision-record.md`'s table — dissolving A's Finding 4 without adding a SHA column to a foreign artifact's shape. This is arguably not "the state file D32 forbids" (D32 reads as aimed at a branch/commit-tracking ledger duplicating what git refs already encode, not a normal per-feature artifact every other phase already writes one of) but it's a Constitution-adjacent call, so I'm flagging it for the chairman/triage rather than asserting it as a fix.

## The `optional: true` / `HookExecutor` finding: D and E agree, and E's version is deeper than its own confidence tag admits

D and E independently traced the same defect — hooks that must hard-block (`before_tasks`/`before_implement`) are specified `optional: true`, and the shared Pre-Execution prose only auto-invokes `optional: false` hooks — via different files and line numbers, which is real corroboration. E goes one step further and traces every skill's `condition`-evaluation language to an undefined "HookExecutor implementation." I grepped this myself: 13 files name `HookExecutor`, all in the identical boilerplate clause, zero implement it. That means even a corrected `optional: false` hook is enforced by an LLM choosing to follow "you MUST actually invoke" prose, not by any code path — the entire hook-dispatch layer this plan's mandatory hard-blocks depend on is currently 100% prose-mediated. E filed this as its *lowest*-confidence suggestion ("low"), which undersells it badly — of everything in this round, this is the one finding that undercuts the deck's "falsifiable without a model call" claim at the mechanism level, not just for this plan's specific FRs. It should be weighted well above low.

## Three peers hit the same `before_specify`/`after_specify` fact, but disagree on how hard to push — I side with the lighter remedy

A, B, and D all independently found that `speckit-specify/SKILL.md` still documents a live `before_specify` hook contract, and that `graphify-context.md` line 13 states the plan "needs... `before_specify`" — directly contradicting the ratified `after_specify` design. All three facts check out (I read the same lines). But the three peers weight the *remedy* very differently: A's Finding 1 treats it as a high-confidence correctness defect and offers "either explain... or switch to `before_specify`"; D treats it as a strawman-framing issue worth one acknowledging line; B treats it as a stale-docs scope item. D's read is the best-calibrated one. Switching to `before_specify` would resurrect exactly the NNN+slug re-derivation duplication the plan correctly rejected — the dormant hook slot tolerating a branch/dir mismatch is evidence it made the *easier* tradeoff, not a better one. The simplicity-correct fix here is documentation-only (regenerate `graphify-context.md`, correct the stale NOTE in `speckit-specify/SKILL.md`), which is what B and D both actually propose. I'd ask the chairman not to let A's "or switch" framing invite a design reversal in triage.

## Two places where a peer's own suggestion adds more machinery than the finding needs

A's fix for the interrupted-`after_specify` risk ("a defensive branch-existence check inside every subsequent `after_*` hook") duplicates a guard N times where N = hook count. Since `branch.sh` is already specified as idempotent create-if-absent-then-checkout, the simpler fix is to fold that same call into `commit.sh` itself, so every commit is self-healing through one entry point instead of guarded independently at each call site. Separately, B's `extensions.yml` flock/lockfile suggestion (B's own lowest-confidence item) duplicates a mitigation the design already states and relies on for the identical install-time race — `graphify-context.md`'s own collision-watch note says the merge "must be serialized (orchestrator glue), never handed to a parallel subagent," i.e. discipline, not code, which is the same standard the entire hook-dispatch mechanism already runs on (see `HookExecutor` above). Singling out one call site for file-locking while every structurally identical race in this design leans on discipline is inconsistent — either discipline is the accepted standard here or it isn't.

## A smaller, genuinely new observation: council-approve is inconsistent with its own extension's convention

`speckit-council/SKILL.md`'s Pre-Execution step 1 already implements a generic `hooks.before_<command>` dispatch block ("the same rules every speckit command uses") for its own `before_council`. `speckit-council-approve/SKILL.md` has none — I grepped it for any hook-related line and found zero. If council-approve adopted the same block for an `after_council_approve` hook, FR-008's SHA-recording call could move from a bespoke hand-edit into a normal `.specify/extensions.yml` hook registration, which is the exact mechanism git-ext already safely owns. This only shrinks R1, it doesn't eliminate it — the block would still live inside a council-owned file subject to council's own reinstall (B's finding stands until council's maintainers adopt it), and it says nothing about `implement-parallel`'s per-wave case, where I confirmed no stock per-wave hook vocabulary exists anywhere in this codebase (matching `research.md` D-R2's own rejected-alternatives note). Worth naming as the more idiomatic target state even though it isn't a full fix.
