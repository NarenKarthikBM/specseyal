This review covers both D68 concerns per the dispatch brief. Overall the plan is unusually testability-forward — §7's FR/SC table is honest about exactly which of the 10 SCs have a committed test (4: SC-001/002/008/010) versus manual/dogfood-only (6), and the deck's own R1–R5 risk register already self-flags two of the weakest spots (SC-004's uncross-checked completeness, SC-005's appendix-variant ambiguity) rather than papering over them — a good sign for adversarial defensibility. My review focuses on where "committed test" claims are less concrete than they look on inspection, and where a few of the plan's most load-bearing guarantees currently rest on live-run observation with no durable artifact to falsify them after the fact.

Cross-checking the deck's SC-008/SC-010 rows (§7) against `plan.md` surfaced a real asymmetry. The Concern-2 (I-17) checkbox-delta regression is spelled out in full: bind a fixture `tasks.md`, flip a checkbox (→ pass), edit a task line (→ block), recheck after a git-ext reinstall AND a foreign-extension reinstall (`plan.md` L162). But the Concern-1 testing-seam regression that SC-008 actually names ("the `testing`-phase git seam survives a git-ext reinstall and a foreign-extension reinstall") is nowhere specified with comparable precision — `plan.md` L134 gives only "Reinstall-survival-tested (`extensions/git/test/run.sh` §3 model, FR-013)," an unexplained reference to a "§3 model" with no named assertions. Since SC-006 (both `complete` and `testing` leave a phase-tagged commit) explicitly rides on "the SC-008 seam test" with no standalone regression of its own (deck §7), this underspecification isn't cosmetic: two SCs' worth of "Yes, committed test" in the table currently point at a test whose actual assertions aren't written down anywhere I could find in the deck or the plan.

Two of the "proven live" claims in §7 (SC-009, SC-010) are integration-only by design — reasonable for SC-009 since its components (SC-001/002/006) are each separately tested, but SC-010's "zero hand assistance" phrasing is a claim about *process* (nobody intervened), not just *outcome* (wave 2+ completed), and a successful wave 2+ can't by itself distinguish "the mechanical fix passed it" from "a human worked around the gate and it passed anyway." Neither the deck nor `plan.md` describes a durable artifact that would let this be checked after the M4 dogfood run rather than taken on faith.

I also checked the deck's own explicitly-flagged open item on SC-005 (with/without-appendix, R5/§7) against `plan.md`'s Testing field (L24) — it confirms the deck's caution was warranted: `plan.md` says only "contract-validation of a golden `completion-report.md`" (singular), with no mention that the golden set includes both an appendix-bearing and an appendix-free sample. This is exactly the kind of thing that's cheap to settle now and expensive to discover missing later.

Did not query the graph: the deck's own I-13 caveat already establishes shell-script edges are graph-blind for both concerns' actual edit targets, and none of the findings below are blast-radius questions.

## Suggestions
- [testability] SC-008's reinstall-survival regression (the Concern-1 testing-seam test the deck cites for SC-008/SC-006/FR-013) is specified only as "`extensions/git/test/run.sh` §3 model" (plan.md L134) — no named fixture or assertions, unlike the I-17/SC-010 regression's fully-spelled-out bind→flip→pass/edit→block/reinstall×2 sequence (plan.md L162). Name the SC-008 regression's concrete assertions with the same precision (e.g., dispatch a fixture `/speckit-complete`+`/speckit-testing`, assert 2 phase-tagged commits exist, reinstall git-ext, repeat, reinstall a foreign extension, repeat) before this reads as "Yes, committed test" in the SC table. (confidence: high)
- [testability] SC-010's "zero hand assistance" is a claim about process, not just outcome, but is only provable by observing M4's own wave 2+ complete — which can't distinguish "the new checkbox-delta branch passed it" from "a human routed around the gate and it passed anyway" (e.g. by hand-rebinding `gates.yml` or re-running `on-gate-approve.sh`). Neither deck nor plan.md names a durable artifact for this. Add a log/trace line fired specifically when the new classification branch (not the pre-existing exact-match fast path) produces the PASS, so the dogfood run leaves independently-auditable evidence of *which* path let wave 2+ through. (confidence: high)
- [testability] SC-004's "100% of spec SCs+FRs mapped" (deck R1, already self-flagged) has no code-level completeness check — only structural presence is validated. Since `spec.md`'s `SC-\d+`/`FR-\d+` IDs are regular, add a mechanical membership check to `extensions/testing/test/run.sh` (or the `/speckit-testing` dispatch itself): extract every ID from `spec.md`, assert each appears at least once in `testing.md`'s coverage map. This can't judge whether a mapped verification approach is *good* (still tester judgment) but closes the "silently dropped item" failure mode R1 names, at low cost. (confidence: high)
- [testability] SC-005's with/without-appendix claim (deck's own flagged open question, §7/R5) checks out as genuinely unresolved: `plan.md`'s Testing field (L24) names only "a golden `completion-report.md`" (singular), no mention of an appendix-bearing/appendix-free pair. Resolve this now rather than leaving it an assumption — commit both fixture variants to the golden set, each independently contract-validated, so SC-005 has an actual two-branch test rather than one. (confidence: high)
- [testability] The wave-1/pre-wave sequencing guarantee for the I-17 fix (FR-016/SC-010 — deck §3's architecture diagram calls this out in caps as a "SEQUENCING GUARANTEE") has no check before the live dogfood run: if `tasks.md` were authored such that the fix's task landed in wave 2 instead of wave 1, nothing catches it until wave 2 itself fails live. Add a static assertion — e.g. a `tasks.md`/execution-waves lint, or a named check in `quickstart.md`'s SC-010 scenario — that the I-17 fix's task ID is actually in wave 1 before implement starts, rather than discovering a violation only via the failure it was meant to prevent. (confidence: med)
- [testability] "Validates against its contract" (SC-001/002) is runnable today (`extensions/testing/test/run.sh`, confirmed at plan.md L24), but neither deck nor plan.md states whether its golden-fixture assertions are derived from `docs/contracts/*.md`'s own section list or hand-maintained in parallel prose. If the latter, the contract doc and its validator can silently diverge — a real cost specifically because 004 is the milestone declaring these pipeline-wide (every future feature's `complete`/`testing` phase validates against them, per deck §4's Structure Decision). Worth confirming the linkage now, before drift has a chance to compound across features. (confidence: med)
