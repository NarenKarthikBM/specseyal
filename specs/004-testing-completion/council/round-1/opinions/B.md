Both D68 concerns hold up structurally, but the risk surface concentrates exactly where the deck itself flags it (R1–R4) plus several spots the deck under-states relative to `plan.md`. I doubted two of the deck's I-17 renderings enough to cross-check them against `plan.md` directly (lines 142–163) — both checks landed on the same pattern: `plan.md`'s own language is precise and closes the gap, but the deck's condensed rendering drops the qualifying language that makes it precise. That's itself a risk worth naming, given the standard-tier contract asks lazy-context members to trust the deck alone.

**Concern 2 (I-17).** Deck §3's classification rule ("`[ ]`→`[x]`/`[X]`, otherwise byte-identical") reads as direction-agnostic — nothing in that wording rules out a `[x]`→`[ ]` reversion also passing as "a pure checkbox flip." `plan.md` L148 is actually explicit ("advanced from unchecked to checked") and closes this correctly via the BLOCK catch-all, so the underlying design is sound — but a lazy-context reviewer working from the deck alone could not have ruled out the ambiguity without opening `plan.md`, which defeats the point of the D62-strengthened §3. Same pattern on reinstall-survival: deck §3's "SEQUENCING GUARANTEE" sentence and §7's SC-010 table row both name only "git-ext reinstall," while `plan.md` L162 explicitly commits the I-17 regression to "a git-ext reinstall **and** a foreign-extension reinstall," matching the SC-008 pattern exactly. Design is fine; the deck's wording is what's under-specified, on the single most security-critical mechanism in the plan.

Separately — and this one survives the plan.md check: the wave-1/pre-wave sequencing guarantee (FR-016/SC-010, `plan.md` L163) and the `commit.sh`/`verify-gate.sh` same-file collision watch (deck §4) are both stated as musts, but I grepped `plan.md` for "parallel," "`[P]`," "serialize," and "wave 1" and found only the prose requirement at L163 — no cited mechanism that forces the not-yet-generated tasks-graph to actually place the I-17 fix in wave 1 with no `[P]` marker relative to the testing-seam's `commit.sh` edit. Easy to state in a plan, easy to lose in an auto-generated DAG.

**Concern 1 (testing extension).** The doc-only `tester`'s honesty is entirely second-hand in a way R1 doesn't quite name. Per `plan.md` L108 its context-in is `completion-report.md` + `spec.md` only — never `implement.log.md`, never source. `completion-report.md` is itself already a lossy compression of `implement.log.md`, authored by the main-thread orchestrator with only structural/contract validation (SC-001), not fidelity validation. If the report over-claims (mistranscribes a caveat as a pass), `testing.md`'s coverage map inherits that silently — the tester has no independent source to catch it against, by design. That's a sharper failure mode than R1's "under/mis-mapped ID" framing: it's about the tester's entire evidentiary base being unverifiable at its own layer, not about coverage-map completeness.

R2's stated mitigation ("the one-trace-record structure makes a violation auditable after the fact") doesn't survive contact with the actual trace schema: `plan.md` L108 lists exactly `role`, model, `agent_id`, `skills`, `elevated_grants` — no context-in / files-read field. If the `tester` subagent violates its context-hygiene contract, nothing in that schema would surface it after the fact.

Also worth naming: the deck specs fail-closed behavior in detail for the I-17 branch (unparseable/unreachable diff ⇒ block) but is silent on failure handling for the two *new* commit hooks themselves — `after_complete`/`after_testing` firing `commit.sh` with a new phase tag. If that invocation errors (nothing staged, a conflict), is that phase-halting, symmetric with `complete`'s own "malformed/absent report leaves `complete` INCOMPLETE" guarantee — or can it silently no-op, leaving SC-006/SC-009's "both phases leave a commit" claim looking satisfied when it isn't until someone checks `git log` at dogfood time.
