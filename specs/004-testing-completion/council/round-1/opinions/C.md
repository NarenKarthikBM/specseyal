The deck bundles two genuinely separable concerns (already ruled in-scope together by D68), and on a simplicity read most of the plan's mechanism choices are already close to the minimal shape given the constraints — not gold-plating. Working through the four questions I was pointed at, plus one adjacent finding:

**`/speckit-complete` as a command, not orchestrator-inline.** The deck's own argument (needs a hook boundary for `after_complete`/FR-012) is sufficient on its own, but there's a second, independent reason it doesn't state: every other phase boundary in this pipeline — specify, clarify, plan, tasks, analyze, categorize, agent-assign, implement(-parallel), council, council-triage, git-cleanup — is already invoked through its own `/speckit-*` command. Inline-prose authorship for `complete` alone would be the *only* exception to a repo-wide one-command-per-phase convention, on top of losing the hook boundary the deck already flags. Ratify.

**Two contract files vs. one.** The deck frames the plan's choice as convention-matching (7 existing 1:1 `docs/contracts/` siblings) and explicitly invites the council to prefer a merge (§2, "Open branch point"). I'd affirm the plan's position rather than take the invitation: a merged file would be the *first* exception to that 1:1 convention, trading one small convenience (fewer files to open) for a standing irregularity every future contract-reader has to remember. The "pointer, not co-location" cross-reference (D46 rule 3) is already the established, lighter-weight way to relate two schemas — no need to invent a combined-file shape that doesn't exist anywhere else in the directory.

**Checkbox-delta classifier — simplest mechanism that blocks content edits?** This is the one place I don't think the plan is at the simplicity floor yet. As §3 describes it, the check is "git diff + iterate hunks + per-line regex classify," run fresh on every freshness failure. A masked/canonicalized hash does the same job with less new machinery: at bind time, hash `tasks.md` with every checkbox marker normalized to one placeholder; at check time, recompute the same canonicalized hash and compare to the recorded one — PASS iff equal. That collapses the whole classification branch into one more hash comparison, reusing the exact SHA-comparison idiom the gate already uses everywhere else, instead of adding a bespoke diff-parsing code path that only runs on failure. It also dissolves R3's edge-case worry for free, since hash comparison doesn't care about hunk alignment or line offsets the way a line-by-line classifier does.

**Is `extensions/testing/` over-built?** No. Packaging it as a 4th sibling extension matches both the other 3 extensions' shape and this repo's own documented layout (CLAUDE.md already lists `testing` alongside `graphify`/`council`/`git`/`categorize`/`agents` as a pipeline extension). Something lighter-weight would be the irregular choice here, not the plan's actual position — I'm not flagging this one.

**Adjacent finding (Concern 2).** The "MUST land in wave 1 (or pre-wave)" sequencing guarantee (FR-016/SC-010, stated emphatically in §3) is, per the Testability table, proven only by a regression test plus a live dogfood check — nothing in the deck ties it to a mechanical wave assignment in `tasks.md` itself. This repo already owns the tool for exactly that job (`/speckit-tasks-graph`'s Execution Waves DAG, which `/speckit-implement-parallel` consumes); using it here would make the guarantee structural rather than a narrative promise resting on the tasks.md author remembering.

## Suggestions
- [simplicity] Ratify the plan's two-`docs/contracts/`-file position over the deck's invitation to merge: a combined file would be the *first* exception to the existing 1:1 schema-per-file convention (7 siblings), trading a minor convenience for a standing irregularity. The pointer-based cross-reference (D46 rule 3) already gives cheap coupling without co-location. (confidence: high)
- [simplicity] Ratify `/speckit-complete` as a real command: beyond the deck's hook-boundary argument, every other phase in this pipeline (specify/clarify/plan/tasks/analyze/categorize/agent-assign/implement(-parallel)/council/council-triage/git-cleanup) is already its own `/speckit-*` command — orchestrator-inline authorship would be the sole exception to that pattern. (confidence: high)
- [simplicity] Replace the checkbox-delta classifier's "git diff + per-line regex" branch with a canonicalized/masked-checkbox hash comparison: normalize every checkbox marker to one placeholder, hash at bind time and at check time, PASS iff the hashes match. Same guarantee, less new code (reuses the existing SHA-comparison idiom instead of adding a diff-parsing path), and it structurally sidesteps R3's hunk-alignment edge cases rather than relying on the "every changed line" quantifier to handle them by construction. (confidence: med)
- [sequencing] The wave-1/pre-wave requirement for the I-17 fix (FR-016/SC-010) has no stated mechanical enforcement in the deck — only a regression test plus a live dogfood proof. Use `/speckit-tasks-graph`'s Execution Waves DAG (already consumed by `/speckit-implement-parallel`) to make "wave 1" a structural property of `tasks.md`'s own dependency graph rather than a narrative promise. (confidence: med)
- [simplicity] R1's "100% of spec SC/FR mapped" gap (SC-004) is currently prose/prompt-enforced only, per the deck's own admission. A mechanical complement is cheap: grep the ID sets (`SC-\d+`/`FR-\d+`) out of `spec.md` and out of `testing.md`'s coverage map, diff them, fail on any spec ID absent from the map. That's barely more code than the checkbox-regex work already planned for Concern 2, and it turns a silently-droppable requirement into a hard-fail instead of trusting the tester subagent's diligence. (confidence: med)
