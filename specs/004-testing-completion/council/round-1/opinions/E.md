This round bundles two concerns with very different sequencing shapes: Concern 1 (the testing extension) is a straightforward tail-of-pipeline addition; Concern 2 (the I-17 fix) carries a genuine self-referential constraint — the fix must be live before the very machinery it corrects is exercised against M4's own implement run. I focused on whether that self-reference is a real bootstrap paradox, whether the "wave 1 (or pre-wave)" guarantee (deck §3; plan.md §2.3, FR-016/SC-010) is actually enforced anywhere, and how the two concerns' shared git-ext `install.sh` interacts with wave placement.

**No true bootstrap paradox — but the reasoning that proves it is never stated.** Tracing the two firing points the deck itself names (§3: "`verify-gate.sh` workforce fires at `before_implement` (once) AND per-wave... repeatedly") — the *first* check fires against a pristine, just-bound `tasks.md`, before wave 1 has touched anything, so it passes under the pre-fix script regardless. The staleness defect is only *triggered* by wave 1's own commit, so the check that actually matters is the one gating entry to wave 2 — leaving the whole of wave 1 as a safe window to build, reinstall, and prove the fix before it is ever load-bearing. This is consistent with, and actually explains, the plan's separate "I-16 does not ride along" call (plan.md §2.3: "M4 does not build that machinery (003 did, live), so M4's `after_workforce_approve` binds `tasks.md`/`assignment.md` normally") — but neither the deck's §3 "SEQUENCING GUARANTEE" callout nor plan.md §2.3 ever connects these two facts into the argument that answers *why* the wave-1 requirement is achievable rather than circular.

**The guarantee has no described enforcement mechanism, and the graph confirms this isn't a paper cut.** `tasks.md` doesn't exist yet (deck §6: "doesn't exist until after this round clears triage"), so "MUST land in wave 1" is a plan-level intention, not yet a scheduled DAG. I checked live whether the graph-aware wave-DAG tooling has any signal that would force this ordering automatically: `graphify explain "verify-gate.sh"` → degree 4, connections limited to `contains`/`defines` edges (`die()`, `mark_stale()`, `usage()`); `graphify path "verify-gate.sh" "implement-parallel"` → **no path found**. This independently confirms the deck's own I-13 caveat in practice, not just as an assertion — the wave-DAG generator has zero graph-derived reason to place the I-17-fix task ahead of wave 2.

**Concern 1 and Concern 2 share one `install.sh`, but only one of them has a stated wave**, and the risk register's own framing doesn't agree with the structure section on what that sharing implies (staged vs. bundled) — detailed in the suggestions below.
