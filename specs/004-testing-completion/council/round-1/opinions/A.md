Both concerns hold up structurally, but each has a gap between what's *claimed* and what's *mechanically enforced* — the deck is candid about most of these (R1, R2, R3 are self-flagged), which is a good sign, but two of the self-flagged risks stop short of a concrete fix, and I found two more the deck doesn't name at all.

**Concern 1 (testing extension).** §2/§3's `/speckit-complete` design leans on tasks.md + implement.log.md being "already in main's context — no dispatch" rather than an explicit re-read. That's cheaper, but it's the one place this plan quietly trades away the project's own resumability principle (every phase idempotent/resumable from artifacts alone) for a context-retention assumption that a long multi-wave implement run, a compaction, or a resumed session can break. Separately, R1 (SC-004's "100% mapped" claim) is correctly identified as prose-enforced only, but the deck stops at flagging it — a mechanical fix is available and precedented (003's coverage-completeness check) and should be a build task, not a standing risk.

**Concern 2 (I-17 fix).** The checkbox-delta classifier is sound in design (diff-based, fail-closed), but the named regression coverage (plan.md §2.3: flip → pass, edit-a-line → block) doesn't match the classifier's own stated scope: §2.2's prose explicitly names "a task added/removed" as a blocking case, which is a structurally different diff shape (unpaired insertion/deletion) than either tested fixture, and it's untested. The flip pattern is also directionally asymmetric by construction ("advanced from unchecked to checked") with no described recovery path for a legitimate reverse edit, now that an I-16 bootstrap-escape was explicitly ruled out. Lastly, the wave-1/pre-wave sequencing guarantee (FR-016/SC-010) that the whole fix depends on has no static enforcement anywhere in the deck — only a unit regression on the classifier itself plus a live dogfood proof — so a bad wave assignment in the not-yet-generated tasks.md would only surface as a live wave-2 failure, the exact symptom being fixed.

I checked one deck claim I doubted against plan.md directly: §3's "idempotent on sha256" phrasing initially read to me like a claim that an LLM-authored report regenerates byte-identically, which would be false on its face — plan.md L104 clarifies it's actually about M5's future push using the file's sha256 as a dedupe key, not about `/speckit-complete` determinism. Not a defect; withdrawn after verification.

## Suggestions
- [correctness] `/speckit-complete` reads tasks.md + implement.log.md by assuming they're "already in main's context — no dispatch" (deck §2, §3) instead of an explicit disk read. This trades away the resumability guarantee (every phase resumable from artifacts alone) for a context-retention assumption that a long multi-wave implement run, a context compaction, or a resumed session can silently break. Have `/speckit-complete` explicitly re-read both files from disk on every invocation regardless of what's already in context. (confidence: high)
- [correctness] R1 (SC-004's "100% of SC+FR mapped" claim) is acknowledged as prose/prompt-enforced only, with no code-level cross-check that every spec.md SC/FR ID lands in testing.md's coverage map. A concrete fix is available and precedented: mechanically extract every SC-/FR- ID from spec.md and diff against the IDs cited in the coverage map, failing contract validation on any gap — the same completeness-checking pattern 003's categorization validator already established. Recommend committing to this as a build task rather than leaving R1 as an accepted-but-unaddressed risk. (confidence: high)
- [correctness] The checkbox-delta regression suite (plan.md §2.3) tests exactly two same-line-count fixtures (checkbox flip, edited task line), but plan.md §2.2's own prose separately names "a task added/removed" as a blocking case — a structurally different diff shape (unpaired line insertion/deletion vs. a 1:1 changed-line pair) that no named fixture exercises. This is a distinct gap from R3's mixed-line/unreachable-SHA examples and deserves its own fixture before the classifier is trusted fail-closed on this shape. (confidence: high)
- [sequencing] FR-016/SC-010's wave-1/pre-wave sequencing guarantee has no named static enforcement — per the deck's own §7 table its only backing is the classifier's unit regression plus "a required live dogfood proof." Since tasks.md/wave assignment doesn't exist yet, nothing stops the eventual generated tasks.md from placing the verify-gate.sh fix outside wave 1; the earliest that would surface is a live wave-2 failure mid-implement — the exact bug being fixed. Worth a cheap pre-implement static check (e.g., assert the fix-task's wave ≤ 1 in tasks.md) rather than relying solely on the dogfood run to catch a violation after the fact. (confidence: med)
- [correctness] The checkbox-flip pattern is directionally asymmetric by design (plan.md §2.2: marker must have "advanced from unchecked to checked") — a reverse flip ([x]→[ ], e.g. a human correcting a wrongly-marked-done task) is not a pure flip and BLOCKs like any content edit. Combined with the plan's explicit rejection of an I-16 bootstrap-escape, there's no stated recovery path for a legitimate un-check other than a full re-gate. Worth the plan confirming this is intentional and naming the recovery path. (confidence: med)
