The deck is unusually self-aware about its own gaps — it already surfaces the `dequote()` other-callers fixture hole, the SC-008/SC-009 manual-only status, and the undecided `--ref` transport as "open for council." That candor is good, but it also means the highest-value adversarial pass here is (a) the one load-bearing claim the deck does *not* flag as a gap, and (b) a couple of concrete, checkable spots the deck's own sourcing (`contracts/*.md`) exposes that the deck's prose glosses over.

**FR-015 is asserted, never verified.** §1 and §3 both call the gate-semantics-neutral invariant "load-bearing" and the deliberate contrast with `007`'s gate-touching profile validator — the entire justification for bundling six unrelated items into one feature rests on it holding. But cross-checking spec.md's own SC list: FR-015 maps only into SC-010 ("no change altered council/workforce gate semantics"), and §7's own coverage table marks SC-010 **Manual**, enforced by "`quickstart.md` close-out block: grep `docs/90` for all six I-codes." A grep for I-code strings in a decision log can confirm the close-out bookkeeping happened; it cannot possibly detect whether `extensions/council/` or `extensions/workforce/` gate logic was touched. So the one invariant the deck calls out twice as *the* thing that makes this bundle safe has, in the plan as written, zero actual verification mechanism — not even the same "manual quickstart walkthrough" tier the other soft SCs get. This deserves either a real check (e.g. a fixture/CI step that diffs or greps the gate-relevant files — `extensions/council/**`, `extensions/workforce/**`, `gates.yml` schema — between the feature branch base and tip and fails if anything outside the six named edit sites changed there) or, at minimum, an explicit human sign-off line item distinct from the I-code grep.

**`--ref` is unpinned by omission and unsanitized by omission.** `contracts/bootstrap-install-command.md` closes one gap I went looking for — `<extension-name>` is a closed enum (`git|graphify|council|workforce|testing|deck-render`, contract line 13) — but the deck's own §3/E1 never surfaces that constraint, which matters because it's the thing standing between "arbitrary string interpolated into `git sparse-checkout set extensions/<name>`" and "safe." Two things the contract does *not* pin down: (1) the actual default value of `--ref` — research R1, the contract, and quickstart.md all just say "the release ref documented in the quickstart" / `<pinned-ref>` placeholder; nothing anywhere commits to an actual tag vs. a moving branch. R1's whole security rationale is "reviewable, pinned posture" — an undecided default is not pinned, it's a TBD with pinned vocabulary. (2) no B-row in the contract commits to validating or escaping the `--ref` value itself before it's passed to `git clone`/`sparse-checkout`; a ref string beginning with `-` is a known git argument-injection vector (the class CVE-2017-1000117 lives in) and nothing in research.md, the contract, or the deck states this input is rejected/escaped. Both are cheap to close now, before there's a public curl-and-run script for anyone to point at an untrusted ref.

**The manual SC-001/002/003 walkthrough can't test the audience it claims to.** This is a sequencing risk the deck doesn't mention: `008` ships before the D73 visibility flip, so whoever runs the "clone-free install, true outsider" quickstart walkthrough for SC-001/002/003 will almost certainly be an authenticated maintainer hitting a still-private repo — `raw.githubusercontent.com`/unauthenticated `git clone` against a private repo behaves differently than against a public one post-flip. A pass on this manual check today doesn't actually prove the unauthenticated path works, because nobody without repo access can run it yet. Worth an explicit note that SC-001/002/003's manual sign-off is provisional and must be re-confirmed against the real public repo immediately after D73, not treated as closed by the pre-flip walkthrough.

**Same-shaped gap, three times, in the hardening items.** I-26 (deck §7: `dequote()`'s other two callers unexercised), I-29 (SC-008: E6 excluded from `data-model.md`'s fixture-covered entities), and I-31 (SC-009: E7 excluded, same list) are three independent instances of the identical pattern — the fix's primary claim gets a fixture, the invariant the fix is supposed to *preserve* doesn't. For a feature whose stated purpose is hardening latent defects before public eyes land on them, shipping 3 of 4 hardening items without proof of the thing each one is supposed to guarantee is a coherent, repeated shortfall, not three unrelated misses — worth a single triage decision (require a minimal regression fixture per claimed invariant, not just per primary fix) rather than three separate ones.

## Suggestions
- [risk] FR-015 ("no gate-semantics change") — the deck's own load-bearing invariant — has no verification mechanism; SC-010's only check (grep `docs/90` for I-codes) proves close-out bookkeeping, not gate-semantics non-interference. Add a real check (diff/grep over `extensions/council/**` + `extensions/workforce/**` + gate schema files against the six named edit sites) or an explicit separate sign-off line. (confidence: high)
- [risk] `bootstrap.sh`'s `--ref` default is never actually decided in any artifact (research.md/contract/quickstart all defer to each other with a placeholder) — commit to a concrete pinned tag before implementation, not a moving branch; an undecided default undermines R1's own "pinned posture" security rationale. (confidence: med)
- [risk] No contract row commits to validating/escaping `--ref` (or `<target-repo-dir>`) before interpolation into `git clone`/`sparse-checkout`; a ref beginning with `-` is a known git argument-injection vector. `<extension-name>` is at least a closed enum per `contracts/bootstrap-install-command.md` line 13, but that constraint isn't surfaced in the deck's own architecture section (§3/E1) at all. (confidence: med)
- [sequencing] SC-001/002/003's manual quickstart walkthrough will be run pre-D73 by an authenticated maintainer against a still-private repo, so it cannot actually exercise the unauthenticated/true-outsider path the SCs describe. Flag this sign-off as provisional and require re-confirmation against the real public repo immediately after the D73 flip. (confidence: med)
- [testability] I-26/I-29/I-31 each independently exhibit the same gap shape — fixture proves the fix, not the invariant the fix must preserve (dequote()'s other callers; E6/E7 excluded from data-model.md's fixture-covered entity list). Treat as one systemic triage item — require a minimal regression fixture per claimed invariant, not just per primary fix — rather than three isolated notes. (confidence: med)
- [risk] `check-conformance.py` shells out to the three existing validators and reads only exit code + stderr (FR-008, by design) — reasonable, but no contract pins the delegated validators to a stable stderr-format/exit-code interface; an unrelated future edit to one validator's error text could silently change what `check-conformance.py` reports as "the cause" with nothing catching the drift. Worth one line in the contract naming this as a frozen interface. (confidence: low)
