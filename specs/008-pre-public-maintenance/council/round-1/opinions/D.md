The deck's own §7 "Testability Claim & Plan-Time Verifications" is unusually candid — it self-reports that only 3 of 10 Success Criteria (SC-004, SC-006, SC-007) carry a genuinely committed, automated, both-branch fixture, with SC-005 partial and the remaining 6 (SC-001/002/003, SC-008/009/010) manual-only. That honesty is a point in the plan's favor, but an adversarial testability review has to ask what actually closes that gap, and the answer isn't fully there yet.

**The core gap: manual verification has no committed executor.** Six SCs rest entirely on a human running `quickstart.md`. I read `quickstart.md` directly and its steps are genuinely well-specified, runnable shell blocks (e.g. US2's PASS/FAIL/determinism/both-branch block, US3's per-item inspect commands) — this isn't hand-wavy. But neither `plan.md`'s Technical Context → Testing field (line 21, which names only `extensions/*/test/run.sh` + committed fixtures) nor the deck names any pipeline step that actually *executes* that walkthrough. This matters because `/speckit-testing`'s own contract is "executed: none always, doc-only, no test runs" — it maps each SC to a verification approach and marks evidence source (report-claimed/log-verified), it does not run anything. So the six "Manual" rows risk being recorded in `testing.md` as covered-by-quickstart without the quickstart ever actually running, right before the D73 public-visibility flip where these are exactly the guarantees that matter most (SC-001/002/003 are the entire adopter-experience promise; SC-008/009/010 are the three latent-defect fixes this feature exists to close).

**A specific fixture hole the deck flags but doesn't mandate closing.** `data-model.md`'s E5 invariant states "`dequote()` and its other callers unaffected" by I-26's inline-comment strip. I independently confirmed via `graphify explain "dequote"` that it has exactly 3 callers — `parse_shell()`, `resolve_target()`, `parse_hook_commands()` — matching the deck's degree-4 claim exactly. The plan's fixture plan (per `data-model.md`'s cross-entity list) only exercises the strip itself, not the other two callers post-edit. The deck correctly names this as a residual gap in §7 but frames it as an observation, not a requirement — it should be one, given the round-1 lesson this deck itself cites ("a guard that can only ever pass proves nothing" applies equally to "an invariant with no fixture that could trip it").

**A determinism claim that's demonstrably manual, not automated.** I read `quickstart.md` lines 37-38: the SC-005 determinism check ("same input twice → identical verdict") is a manual re-run-and-eyeball command, and it is *not* included in the "both-branch fixture suite" invocation on the next line (`sh extensions/workforce/test/run.sh`). So the deck's own "Partial" rating for SC-005 is accurate — but a same-output diff assertion is cheap to automate and there's no reason it should stay manual.

**Collision hazard extends past the fix site to the guard itself.** I confirmed via `graphify explain "print_manual_block"` that an unqualified lookup resolves to deck-render's copy (degree 2, community 262) — exactly the deck's §4 claim, and a real hazard. The deck discusses this as a risk to correctly *locating* the I-23 fix, but the same collision risk applies to the FR-011 divergence guard's own implementation: if `extensions/git/test/run.sh`'s "generalized grep over the full manifest" isn't scoped to `extensions/git/install.sh` specifically, it risks silently validating against, or being satisfied by, an unrelated extension's same-named function.

## Suggestions
- [testability] Name an explicit executor for the 6 manual-only SCs (SC-001/002/003, SC-008/009/010) — e.g. bind `quickstart.md` to the repo's own quickstart-integration-gate discipline before `/speckit-complete`, or otherwise state in `plan.md`'s Testing field who runs it and when — since `/speckit-testing` itself never executes anything (doc-only, "executed: none" by its own contract) and these are the exact guarantees load-bearing for the D73 public flip. (confidence: high)
- [testability] Turn the dequote()-other-callers gap from an observation into a requirement: add regression cases for `parse_shell()` and `resolve_target()` to `extensions/graphify/test/run.sh` proving I-26's edit doesn't regress them — graph-confirmed 3 callers (`graphify explain "dequote"`: degree 4, called by `parse_shell()`, `resolve_target()`, `parse_hook_commands()`), but the committed fixture plan only exercises the strip, not the other two call sites the invariant in `data-model.md` E5 claims are safe. (confidence: high)
- [testability] Automate the SC-005 determinism check instead of leaving it a manual quickstart step: `quickstart.md` lines 37-38 show a "run twice, compare by eye" command that is explicitly excluded from the `run.sh` both-branch suite invocation on the next line — fold a double-run byte-diff assertion into `extensions/workforce/test/run.sh` so SC-005 moves from partial/manual to code-verified alongside SC-004/006/007. (confidence: med)
- [testability] Make explicit that I-29/I-31's fixture absence (data-model.md excludes E6/E7) is architectural, not incidental: both edit SKILL.md prompt text interpreted by a live LLM-dispatched session, unlike I-23/I-26's pure bash/Python edits, so no run.sh-style fixture can exercise them the same way — the deck's "two exceptions surfaced in §7" phrasing understates this. Separately, since `check-conformance.py` already directly-checks the trace-schema contract (deck §3, E2 step 3), consider folding I-31's "no traces.jsonl record names a gitignored/untracked artifact path" rule into that checker against a hand-authored fixture (violating + clean record), moving SC-009 out of the permanently-manual bucket without new tooling. (confidence: med)
- [testability] Scope the FR-011 divergence guard's grep to the file path, not the bare function/hook name: `graphify explain "print_manual_block"` confirms an unqualified lookup silently resolves to deck-render's copy (degree 2, community 262) rather than git's — the same collision the deck flags as a fix-targeting hazard in §4 applies equally to the guard's own implementation in `extensions/git/test/run.sh`, which must not be satisfiable by matching an unrelated extension's same-named function. (confidence: med)
- [testability] Convert SC-005's "no duplication of existing validators" from an asserted design property into a one-line mechanical guard: a static check (e.g. `check-conformance.py`'s source contains no `import validate_profile`-style source import, only subprocess/shell-out calls) would catch a future accidental regression toward source-level duplication, which the deck itself says is currently untestable by design. (confidence: med)
