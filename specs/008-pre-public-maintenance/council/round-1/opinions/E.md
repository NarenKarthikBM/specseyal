The plan's structural bones are sound — six additive, gate-semantics-neutral items, delegation over duplication, and a §4 collision watch that already does real sequencing work by naming four shared/mutable files that must be serialized. That collision watch is the deck's strongest sequencing artifact. But several ordering questions the deck either glosses over or defers wholesale to `/speckit-tasks` are exactly the kind of thing that should be pinned down at plan-review time, before a tasks-generation session has to guess at implicit ordering constraints from prose.

**1. The I-23 guard-falsifiability claim has an ordering hole.** §7 states H1's contract "explicitly requires proving the guard fails on the current pre-fix state before it can be trusted post-fix" — good, that's the right falsifiability discipline (the round-1 lesson the deck itself cites: "a guard that only ever passes proves nothing"). But §4's collision watch says `extension.yml` and `install.sh` "must move in lock-step in one task, not split across a wave." Neither the deck nor (per my spot-check) `plan.md` states the *internal* ordering within that one task: generalize the guard first, run it and observe it fail against the current manifest/block, *then* apply the `after_complete`/`after_testing` rows, *then* re-run to observe it pass. If an implementer instead writes the guard already shaped around the post-fix manifest (or writes guard + fix together and only ever runs it once, post-fix), the "fails pre-fix" half of the falsifiability claim becomes unprovable after the fact — there's no git-observable moment where the pre-fix failure was actually witnessed. This is a real gap between what SC-006 claims and what the plan actually pins down as required sub-sequencing.

**2. Reinstall-mirror resync timing isn't sequenced against the SC-006 guard run.** The Risk Register accepts that I-23's edit may trigger a reinstall that leaves `.specify/extensions.yml` stale, mitigated by "re-sync as a separate `infra(git)` commit" — "if triggered." What's missing is *when*, relative to running `extensions/git/test/run.sh`'s guard, that resync must happen. If the guard test is run before a triggered reinstall is detected and resynced, it may validate against a stale installed mirror rather than the just-edited source, producing a false-confidence pass/fail that doesn't reflect the actual fix. The task-generation phase should be told explicitly: detect-and-resync-mirror is a precondition to the guard run, not a cleanup step that can trail it.

**3. FR-016's same-session close-out is a hard fan-in that sits awkwardly against the "independently shippable" framing.** `spec.md` states each user story is independently shippable (US1 alone "already delivers the highest adopter value on its own"), and the deck's own Risk Register flags the shared `docs/90` append as "High (by design)" likelihood. But the deck doesn't address what happens if the six items don't finish in the same implementation wave — five owner piles, several serialization constraints (docs/90 itself, the git pair, README) almost guarantee this won't be a single flat parallel wave. If the piles complete at different times, FR-016 forces the close-out step to become a blocking join-point that waits on the *slowest* pile, which is in tension with `speckit-implement-parallel`'s whole reason for existing (independent waves). The plan should say explicitly whether close-out is allowed to be itself a final serialized wave (my read of the intent) or whether it's meant to gate per-item completion in some other way — right now this is implicit.

**4. `bootstrap.sh`'s state machine has no named failure state for a mid-delegate failure.** §3's E1 states `absent → acquired (temp dir) → installed (target) → temp cleaned`, and separately says cleanup happens "on exit, including failure paths." That's fine for temp-dir cleanup, but the state diagram never names what happens to the *target* if step 3 (delegating to the extension's own `install.sh`) fails partway through — is the adopter left with a partial install in their real target repo, with only the temp fetch cleaned up? The deck reasons this is inherited, pre-existing risk from the already-idempotent `install.sh`, which is a fair point, but it's asserted rather than shown, and the ordering of "propagate install.sh's exit code" vs. "run temp cleanup" (trap-based EXIT handling, presumably) isn't specified enough for tasks.md to pin down without re-deriving it.

**5. The collision watch is prose, not a task-graph constraint, and the plan doesn't say who's responsible for translating it.** §6 confirms `tasks.md` doesn't exist yet — Phase 2, deferred to `/speckit-tasks`. That means the four serialization requirements in §4 currently live only as narrative in the deck/plan, with no stated mechanism ensuring they become actual dependency edges (rather than being silently lost or re-derived incorrectly) when tasks are generated — especially if `/speckit-tasks-graph`'s automated wave/DAG builder is used, which reasons over declared dependencies, not deck prose. The plan should require these four collision-watch items to be explicitly encoded as hard depends-on edges at tasks-generation time, not left as something a human has to remember to re-read from this deck.

## Suggestions
- [sequencing] I-23's task instructions must pin the sub-order within the single lock-step task: generalize the guard → run it and confirm it fails against the current pre-fix manifest/block → apply the extension.yml/install.sh fix → re-run and confirm it passes. Without this stated explicitly, H1's "prove it fails pre-fix" contract is easy to satisfy only in appearance. (confidence: high)
- [sequencing] Sequence the `.specify/extensions.yml` mirror resync (if a reinstall is triggered by the I-23 edit) strictly *before* running `extensions/git/test/run.sh`'s guard, not as an independent "separate commit" with no stated ordering relative to the SC-006 verification step — otherwise the guard may validate a stale installed mirror. (confidence: med)
- [sequencing] Clarify whether FR-016's same-session, all-six close-out to `docs/90` is meant to be the final serialized wave after all five owner piles land, or whether it's expected to gate earlier — given the plan's own §4 collision watch all but guarantees the six items won't land in one flat parallel wave, this join-point behavior should be stated, not implicit. (confidence: med)
- [risk] `bootstrap.sh`'s E1 state machine (§3) has no named state or exit-code-propagation rule for a step-3 (delegate to install.sh) failure that partially writes the target repo — cleanup is only specified for the temp dir. Worth an explicit sentence in tasks.md rather than leaving it inherited-by-assumption from install.sh's idempotency. (confidence: low)
- [sequencing] Require the four §4 collision-watch constraints (docs/90, git's extension.yml+install.sh pair, README, the mirror resync) to be encoded as explicit depends-on edges when `/speckit-tasks` (or `/speckit-tasks-graph`) generates tasks.md, rather than relying on the task-generation session to re-derive them correctly from this deck's prose. (confidence: med)
- [sequencing] I-29 (council-apparatus provenance) edits `speckit-council/SKILL.md` and `speckit-council-triage/SKILL.md` — the very files this 008 council round is currently running under. Neither the deck nor `plan.md` addresses whether 008's own `suggestions.md`/decision-record should retroactively carry the new provenance line once I-29 ships within this same feature, or whether it's acceptable that the feature which adds council-apparatus provenance is itself the one round left un-provenanced. Given this repo's own dogfooding rule ("the pipeline builds itself" from M1 onward), this self-referential edge case deserves an explicit answer rather than silence. (confidence: med)
