## Review

The deck for `008-pre-public-maintenance` is unusually self-critical for a plan-defense document — §7's own tally (6/10 SCs manual-only) and §4's collision-watch list are exactly the kind of honesty an adversarial council wants to see, and the Rejected Alternatives table (§2) shows real work was done to keep each of the six items in its simplest defensible shape (delegation over reimplementation for the checker, wrap-not-fork for the installer, surgical single-site edits for the four hardenings). Judged as a whole, the architecture is appropriately boring: zero new third-party deps, no new pipeline phase, no gate-semantics change (FR-015). That said, three places where the deck's simplicity claims either understate real complexity or leave an unguarded seam are worth flagging.

**Graph-verified finding.** §4 states the graph "disambiguates multiple same-named nodes, one per extension installer" for `print_manual_block()`, and that an unqualified `graphify explain "print_manual_block"` resolves to deck-render's copy (degree 2, community 262) rather than git's. I ran this myself: `graphify explain "print_manual_block"` returns exactly `deck_render_install_print_manual_block`, source `extensions/deck-render/install.sh L207`, degree 2, community 262 — the deck's claim checks out precisely. But I also scanned the full node list for every `*print_manual_block` id and found **five** copies, not the "one other" the deck's prose implies: `deck_render_install_print_manual_block`, `git_install_print_manual_block` (confirmed via `graphify explain`: `extensions/git/install.sh L159`, degree 2, community 64 — the correct I-23 target), plus `graphify_install_print_manual_block`, `testing_install_print_manual_block`, and `workforce_install_print_manual_block`. The deck under-states the size of the collision-prone-name pattern it's using as its own cautionary example — see Suggestion 1.

**Fetch-path complexity vs. its own test coverage.** §3 E1 step 2 specifies two distinct fetch mechanisms for `bootstrap.sh` — `git clone --depth 1 --filter=blob:none --sparse` plus a codeload-tarball fallback "if sparse-checkout is unavailable" — but §7's coverage table marks SC-001/SC-002/SC-003 (the three SCs that exercise this exact code) as **Manual**, with "no automated harness named for `bootstrap.sh` in `plan.md`'s Testing section." Two independent fetch code paths is the single largest branch-count increase anywhere in this six-item bundle, landing in the one item with zero automated coverage. See Suggestion 2.

**check-conformance.py's direct-check step is an informal second parser.** §3 E2 step 3 has the checker "read the rule text from `docs/contracts/*.md` itself" for six contracts and check presence/frontmatter/format "against the feature dir directly." This is architecturally sound versus the rejected alternative of duplicating the three real validators (§2, FR-008) — but for the six contracts with no dedicated validator, the checker is now the *only* code that has to stay in sync with prose in `docs/contracts/*.md`, with nothing enforcing that sync. That is the same divergence shape the deck itself names at §2 ("duplication invites the exact manifest-vs-block divergence class I-23 fights") one layer removed — docs-vs-checker instead of manifest-vs-block. See Suggestion 3.

Two smaller points: the deck's own §7 tally that 6 of 10 SCs are manual-only doesn't appear as its own line in the §5 Risk Register — it's arguably the largest concrete gap disclosed anywhere in the deck and deserves a row of its own rather than living only in the testability table (tagged testability, not simplicity, but raising it since it's real). And separately, the six-item bundle's serialization burden is already well-owned by §4/§6, but nothing in the plan currently pins the *order* of the shared `docs/90` close-out step relative to the other five items landing — worth making explicit at `/speckit-tasks` time.

## Suggestions

- [simplicity] The graph shows **five** same-named `print_manual_block()` nodes across extension installers (deck-render, git, graphify, testing, workforce — confirmed via `graphify explain "print_manual_block"` plus a full node-id scan), not the "one other" collision the deck's §4 prose implies. I-23 correctly scopes to git's copy only (`extensions/git/install.sh L159`, graph-confirmed), but the deck/risk-register should say explicitly that four other same-named, structurally-identical functions exist repo-wide and are equally susceptible to the same manifest/manual-block drift I-23 fixes — otherwise a future reader may assume I-23 closed the pattern everywhere. (confidence: high)
- [simplicity] `bootstrap.sh`'s fetch step carries two independent code paths (sparse partial-clone, tarball fallback), yet §7 marks all three SCs that exercise it (SC-001/002/003) as manual-only with no automated harness named. Either collapse to a single fetch mechanism (a tarball-only fetch is simpler and has fewer environment preconditions than sparse-checkout support-detection) or add a minimal automated smoke test that exercises both branches, since this is currently the least-tested and most-branchy item in the bundle. (confidence: med)
- [simplicity] `check-conformance.py`'s direct-check step re-derives rules by parsing `docs/contracts/*.md` prose for six contracts with no validator, creating an unguarded checker-vs-docs sync dependency — the same divergence shape the deck names at §2 for manifest-vs-block, one layer removed. Recommend a lightweight self-check (e.g., the checker's own `_self_test()` asserts the specific section headers/field names it parses still exist in each `docs/contracts/*.md` file) so a future contract-doc edit fails loudly instead of silently desyncing. (confidence: med)
- [testability] §7's own tally — 6 of 10 SCs are manual-verification-only — is not reflected as its own row in the §5 Risk Register, even though it's the largest concrete gap the deck discloses. Recommend promoting it from a testability-table footnote to an explicit risk-register entry so triage tracks it as a first-class item rather than something a reader has to reconstruct from a table footnote. (confidence: med)
- [simplicity] The six-item bundle's shared-file serialization burden is well-identified (§4, §6), but nothing pins the *order* of the `docs/90` close-out step relative to the other five items — SC-010's only check (§7) is a post-hoc grep for all six I-codes, which can't distinguish "closed in the right order" from "closed out of order but all present." Recommend `/speckit-tasks` make the close-out task explicitly last and gated on the other five tasks' completion, not just co-scheduled in the same session. (confidence: low)
